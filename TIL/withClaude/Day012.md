# Day 012. 왜 서버는 한 대로는 부족할까? — 로드 밸런싱과 Scale-Out

> 🗓️ 2026-03-13  
> 🏷️ Tags: `로드밸런서` `Scale-Out` `ALB` `NLB` `무중단배포` `Docker네트워크` `iptables`

---

## 🤔 질문

서버를 여러 대로 늘리면 요청을 어떻게 나눠서 보낼까?  
JWT가 stateless인 이유는 Scale-Out과 어떤 관련이 있을까?  
Docker는 포트 매핑과 컨테이너 네트워크를 내부에서 어떻게 구현할까?

---

## 📌 핵심 요약

로드 밸런서는 단순한 트래픽 분배기가 아니라, **알고리즘 기반 분배 + 헬스 체크 + 무중단 배포**를 담당하는 Scale-Out의 핵심이다.  
JWT의 stateless 설계는 **여러 서버에서 세션 공유 문제 없이 동작하기 위한** Scale-Out 필수 조건이며,  
Docker의 포트 매핑은 로드 밸런싱이 아니라 **iptables를 이용한 패킷 포워딩**이다.

---

## 🔍 Part 1. Scale-Up vs Scale-Out

### 1-1. 서버 한 대의 한계

```
사용자 100명   → 서버 1대 → 잘 동작
사용자 10만 명  → 서버 1대 → 응답 10초 → 사용자 이탈
사용자 100만 명 → 서버 1대 → 서버 다운!
```

### 1-2. 두 가지 선택지

```
Scale-Up: 서버 한 대를 더 좋은 걸로 교체
  → CPU 4코어 → 64코어, 메모리 16GB → 256GB
  → 한계: 물리적 한계 존재, 성능 2배에 가격 4배 (비선형 증가)

Scale-Out: 서버 여러 대로 나눠서 처리
  → 서버 1대 → 10대 → 100대
  → 이론상 무한 확장 가능
  → 핵심 문제: "요청을 어떻게 나눠서 보내지?"
```

---

## 🔍 Part 2. 로드 밸런서 — 트래픽을 분배하는 교통 경찰

### 2-1. 분배 알고리즘

**라운드 로빈:** 순서대로 1번 → 2번 → 3번 → 1번. 가장 단순하지만 서버 성능이 다르면 불균형이 생긴다.

**가중 라운드 로빈:** 서버 성능에 따라 비율을 다르게 배분. 고성능 서버에 더 많은 요청을 보낸다.

**최소 연결:** 현재 처리 중인 요청이 가장 적은 서버로 전송. 실시간 부하를 반영하므로 라운드 로빈보다 똑똑하다. 비유하면, 라운드 로빈은 마트 계산대에 순서대로 줄을 세우는 것이고, 최소 연결은 "줄이 가장 짧은 계산대로 가세요"다.

**IP 해시:** 사용자 IP를 해시하여 같은 사용자를 항상 같은 서버로 보낸다. 세션을 서버 메모리에 저장하는 경우 유용하다.

---

### 2-2. Scale-Out에서 세션 문제 — JWT와의 연결

```
세션 방식 + 로드 밸런서:
  서버 1 메모리: { session_abc: "홍길동, ADMIN" }
  서버 2 메모리: { session_def: "김철수, USER" }

  홍길동이 항상 서버 1로 가면 괜찮지만...
  서버 1이 죽으면 → 세션 소멸 → 강제 로그아웃!
```

해결 방법 3가지:

```
Sticky Session: 같은 사용자를 같은 서버로 고정
  → 서버 죽으면 세션 소멸, 부하 불균형

Session Clustering: 서버끼리 세션 공유 (Redis에 저장)
  → Redis 의존성 추가, 네트워크 오버헤드

JWT (Stateless): 서버에 세션을 안 저장
  → 어느 서버로 가든 JWT만 검증하면 됨
  → Scale-Out에 가장 적합!
```

JWT의 stateless는 "멋있어서"가 아니라 **Scale-Out 환경에서 세션 공유 문제를 없애기 위한 설계**다.

---

### 2-3. 헬스 체크

로드 밸런서는 각 서버가 살아있는지 주기적으로 확인한다.

```
로드 밸런서가 주기적으로:
  GET /health → 서버 1: 200 OK ✅
  GET /health → 서버 2: 200 OK ✅
  GET /health → 서버 3: 응답 없음 ❌ → 목록에서 자동 제외!

서버 3 복구 시 → 헬스 체크 통과 → 자동 복귀
```

---

### 2-4. 무중단 배포

**Rolling 배포:** 서버를 하나씩 빼고 업데이트 후 복귀. 항상 최소 N-1대가 요청을 처리하므로 사용자는 중단을 느끼지 못한다.

```
서버 1(v1), 서버 2(v1), 서버 3(v1)
→ 서버 1 제외 → v2 업데이트 → 복귀
→ 서버 2 제외 → v2 업데이트 → 복귀
→ 서버 3 제외 → v2 업데이트 → 복귀
→ 완료! 다운타임 없음
```

**Blue-Green 배포:** 현재 운영(Blue) 그룹과 새 버전(Green) 그룹을 준비하고, 로드 밸런서가 트래픽을 순간적으로 전환한다. 문제 발생 시 즉시 롤백 가능. 단점은 서버가 2배 필요하다는 것.

---

### 2-5. 로드 밸런서 이중화 — SPOF 방지

모든 트래픽이 로드 밸런서를 거치므로, 로드 밸런서 자체가 단일 장애 지점(SPOF)이 될 수 있다.

```
실무 구조:
  [LB Active] ↔ heartbeat ↔ [LB Standby]

  Active 장애 → Standby가 즉시 승격 (수 초 이내)
  AWS 환경 → ALB/NLB 자체가 이미 이중화되어 있음
```

## 🔍 Part 3. L4 vs L7 — ALB와 NLB

### 3-1. 핵심 차이

```
ALB (Application Load Balancer, L7):
  → HTTP 요청 내용(URL, 헤더, 쿠키)까지 보고 분배
  → "/api/*는 API 서버로, /static/*는 정적 서버로"
  → 똑똑하지만 패킷을 열어봐야 하므로 약간의 지연

NLB (Network Load Balancer, L4):
  → IP/포트만 보고 분배
  → 패킷 내용을 안 보니까 극한의 속도
  → 초당 수백만 연결 처리 가능
  → 대신 URL별 세밀한 라우팅 불가
```

비유하면, ALB는 **"어디 가세요?" 물어보고 안내하는 데스크 직원**이고, NLB는 **물어보지 않고 바로 통과시키는 자동 회전문**이다.

### 3-2. 실무 선택 기준

```
ALB (90%의 경우):
  → REST API, MSA 서비스별 라우팅, HTTPS 종단, 웹소켓

NLB (특수한 10%):
  → 게임 서버, 실시간 트레이딩, IoT 등 극한 성능/저지연 필요
  → 고정 IP 필요한 경우 (NLB만 지원)
```

## 🔍 Part 4. Docker의 네트워크 — 포트 매핑은 로드 밸런싱이 아니다

### 4-1. 포트 매핑 = 통로 열기

Docker의 `-p 8080:80`은 트래픽 분배가 아니라 **연결 통로를 여는 것**이다. 내부적으로 Linux 커널의 **iptables**를 사용한다.

```
docker run -p 8080:80 nginx 실행 시:

Docker가 iptables에 규칙 추가:
  "호스트 8080으로 들어온 패킷 → 컨테이너 172.17.0.2:80으로 전달"

패킷의 여정:
  외부 요청 → 호스트:8080
  → iptables: "8080? → 172.17.0.2:80으로!" (DNAT)
  → docker0 가상 브리지 → 컨테이너:80
```

비유하면, 포트 매핑은 **건물에 문을 만드는 것**이고, 로드 밸런서는 **정문에 안내 직원을 배치하는 것**이다.

---

### 4-2. Docker 내부 네트워크 구조

```
호스트 머신:
  호스트 네트워크 (eth0: 192.168.1.100)
       │
  ─────┴─────
  │ docker0 │ ← 가상 브리지 (172.17.0.1), 공유기 역할
  ─────┬─────
       │
  ┌────┴────┐  ┌─────────┐  ┌─────────┐
  │ veth001 │  │ veth002 │  │ veth003 │  ← 가상 랜선
  ├─────────┤  ├─────────┤  ├─────────┤
  │컨테이너1 │  │컨테이너2 │  │컨테이너3 │
  │172.17.0.2│  │172.17.0.3│  │172.17.0.4│
  └─────────┘  └─────────┘  └─────────┘
```

docker0는 Docker가 만든 **가상 스위치(공유기 역할)**, veth는 컨테이너와 브리지를 잇는 **가상 랜선**, 컨테이너 IP는 Docker가 자동 할당하는 **내부 IP**다.

---

### 4-3. 컨테이너끼리의 통신

같은 Docker 네트워크 안의 컨테이너끼리는 **컨테이너 이름으로 통신**할 수 있다.

```yaml
services:
  app:
    image: my-app
    # "db:3306"으로 접근 가능!
  db:
    image: mysql
    # 외부 포트를 안 열어도 app에서 접근 가능
```

```
app → Docker 내장 DNS: "db" → 172.17.0.4
    → docker0 브리지 → db 컨테이너:3306

→ iptables 포트 매핑 없이 내부 네트워크에서 직접 통신
→ DB 같은 건 외부에 안 열고 내부에서만 접근하는 게 안전
```

---

### 4-4. Docker에서 로드 밸런싱은 별도 도구가 필요

Docker 단독으로는 로드 밸런싱을 하지 않는다.

```
소규모: Docker Compose + Nginx
  → Nginx 컨테이너가 L7 로드 밸런서 역할
  → 앱 컨테이너 여러 개에 분배

중규모: Docker Swarm
  → docker service create --replicas 3
  → Docker가 IPVS(Linux 커널 내장 L4 LB)로 자동 분배

대규모: Kubernetes + AWS ALB/NLB
  → K8s가 컨테이너 관리 + 내부 로드 밸런싱
  → ALB/NLB가 외부 트래픽 처리
  → 오토스케일링, 자동 복구, 무중단 배포 전부 지원
```

---

### 4-5. Docker의 모든 것은 Linux 커널 기능의 조합

```
격리:        네임스페이스
자원 제한:   cgroups
파일 시스템:  UnionFS / overlay2
네트워킹:    가상 브리지 + veth + iptables
로드 밸런싱:  IPVS (Linux 커널 내장)

Docker가 직접 만든 저수준 기술: 없음!
전부 Linux 커널의 기존 기능을 편하게 포장한 것
```

---

## 💡 한 줄 정리

> 로드 밸런서는 **트래픽 분배 + 헬스 체크 + 무중단 배포**를 담당하는 Scale-Out의 핵심이며, JWT의 stateless는 이를 위한 필수 조건이다. Docker의 포트 매핑은 **iptables 패킷 포워딩**이지 로드 밸런싱이 아니며, 실제 분배는 **Nginx(소규모), Docker Swarm(중규모), Kubernetes + ALB/NLB(대규모)** 가 담당한다.
