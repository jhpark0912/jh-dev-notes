# Day 005. 왜 0.1 + 0.2는 0.3이 아닐까?

> 🗓️ 2026-03-01  
> 🏷️ Tags: `부동소수점` `IEEE754` `BigDecimal` `2진법` `소수` `정밀도`

---

## 🤔 질문

왜 거의 모든 프로그래밍 언어에서 `0.1 + 0.2 ≠ 0.3`일까?  
이 문제를 실무에서는 어떻게 해결하며, BigDecimal은 내부적으로 어떻게 오차를 없앨까?

---

## 📌 핵심 요약

10진법의 `0.1`은 2진법에서 **무한 순환소수**가 되어, 유한한 비트로 잘라 저장하면 오차가 생긴다.  
BigDecimal은 소수를 **"정수 + 소수점 위치"로 분해**하여 2진법 소수 변환을 원천 회피한다.  
입력(문자열) → 저장(정수) → 연산(정수) → 출력(문자열) 전 과정에서 2진법 소수가 등장하지 않는다.

---

## 🔍 Part 1. 왜 오차가 생기는가?

### 1-1. 모든 언어에서 발생하는 현상

```javascript
// JavaScript
0.1 + 0.2  // → 0.30000000000000004

// Python
0.1 + 0.2  // → 0.30000000000000004

// Java
System.out.println(0.1 + 0.2);  // → 0.30000000000000004
```

버그가 아니라, 컴퓨터가 소수를 저장하는 방식의 **구조적 한계**다.

---

### 1-2. 2진법 소수 변환의 원리

10진법에서 소수 각 자리가 10의 거듭제곱인 것처럼, 2진법에서는 2의 거듭제곱이다.

```
2진법 소수 자릿값:
0.1(2)   = 1/2   = 0.5
0.01(2)  = 1/4   = 0.25
0.001(2) = 1/8   = 0.125
```

2진법 소수로는 **1/2, 1/4, 1/8, 1/16...의 조합**만 깔끔하게 표현 가능하다.  
0.5(=1/2)나 0.25(=1/4)는 딱 떨어지지만, 0.1(=1/10)은 이 조합으로 맞출 수 없다.

---

### 1-3. 0.1을 2진법으로 변환해보기

변환 규칙은 단순하다. **"2를 곱해서 정수부(0 또는 1)를 순서대로 뽑아내는 것"** 이다.

```
0.1 × 2 = 0.2  → 1 미만 → "0"
0.2 × 2 = 0.4  → 1 미만 → "0"
0.4 × 2 = 0.8  → 1 미만 → "0"
0.8 × 2 = 1.6  → 1 이상 → "1", 1을 빼면 0.6
0.6 × 2 = 1.2  → 1 이상 → "1", 1을 빼면 0.2
0.2 × 2 = 0.4  → "0"  ← 0.2가 다시 나옴! 무한 반복 시작
...

결과: 0.0001100110011... (0011이 무한 반복)
```

한 번 나왔던 숫자(0.2)가 다시 나오는 순간, 이후는 동일한 패턴이 영원히 반복된다.  
10진법에서 `1/3 = 0.333...`이 끝나지 않는 것과 완전히 같은 원리다.

반면 0.25는 깔끔하게 끝난다.

```
0.25 × 2 = 0.5  → "0"
0.5  × 2 = 1.0  → "1", 1을 빼면 0.0 ← 끝!

결과: 0.01(2) → 정확!
```

## 🔍 Part 2. IEEE 754 — 부동소수점의 국제 표준

### 2-1. 왜 표준이 필요했나?

1970~80년대, 컴퓨터 제조사마다 소수점 처리 방식이 전부 달랐다.

```
IEEE 754 이전:
  IBM:  0.1 + 0.2 = 0.300000001
  DEC:  0.1 + 0.2 = 0.299999999
  Cray: 0.1 + 0.2 = 0.300000004
  → 같은 수식인데 컴퓨터마다 결과가 다름!

IEEE 754 이후:
  모든 컴퓨터: 0.1 + 0.2 = 0.30000000000000004
  → 오차는 여전하지만, 전 세계 어디서든 같은 결과
```

오차를 없앤 게 아니라, **"오차의 규칙을 통일한 것"** 이다.

---

### 2-2. 누가 만들었나?

**IEEE(전기전자공학자 협회)** 에서 제정했다. Wi-Fi 표준(IEEE 802.11)도 이 단체에서 나왔다.

UC 버클리의 수학과 교수 **윌리엄 카한(William Kahan)** 이 중심이 되어 설계했으며, 이 공로로 1989년 튜링상(컴퓨터 과학의 노벨상)을 수상했다. 1985년 첫 표준, 2008년 개정판(IEEE 754-2008)이 현재까지 사용된다.

---

### 2-3. 64비트 배정밀도(double)의 구조

```
64비트 구조:

[부호 1비트] [지수 11비트] [가수 52비트]
   ↓              ↓             ↓
 +인지 -인지   크기 스케일    실제 숫자 정밀도
```

2진법 과학적 표기법이라고 보면 된다.

```
10진법: -3.14 × 10²     → 부호(-), 가수(3.14), 지수(2)
2진법:  -1.0100 × 2⁵    → 부호(-), 가수(1.0100), 지수(5)
```

가수부가 52비트이므로 소수점 아래 52자리까지만 저장 가능하다. 그 뒤는 잘려나가며, 이 잘림이 오차의 원인이다.

---

### 2-4. 0.1 + 0.2의 실제 계산 과정

```
0.1 저장 시:
  무한소수를 52비트로 자름
  → 0.1000000000000000055511151231257827...

0.2 저장 시:
  무한소수를 52비트로 자름
  → 0.200000000000000011102230246251565...

더하면:
  0.1000...055 + 0.200...011 = 0.3000000000000000444...
  ≈ 0.30000000000000004
```

0.1도 미세하게 크고, 0.2도 미세하게 큰데, 이 오차가 합산되어 눈에 보이는 수준이 된 것이다.

참고: `0.1 + 0.1 = 0.2`가 정확해 보이는 건, 오차가 없는 게 아니라 **결과의 오차가 0.2 자체의 저장 오차와 우연히 일치**했기 때문이다.

## 🔍 Part 3. 실무에서의 해결법

### 3-1. 정수로 바꿔서 계산 (가장 간단)

소수점을 아예 없애고 정수 단위로 처리한다. 정수는 2진법에서 오차가 없다.

```javascript
// ❌ 부동소수점
0.1 + 0.2  // → 0.30000000000000004

// ✅ 센트 단위 정수
10 + 20 = 30  // → 정확! 표시할 때만 100으로 나눔
```

Stripe 같은 결제 서비스가 이 방식을 사용한다.

```javascript
// Stripe API: 10달러 50센트 청구
{ "amount": 1050, "currency": "usd" }  // 센트 단위 정수
```

---

### 3-2. BigDecimal — 10진법 그대로 저장

```java
// ❌ double (IEEE 754)
double a = 0.1 + 0.2;               // → 0.30000000000000004

// ✅ BigDecimal (10진법 유지)
BigDecimal a = new BigDecimal("0.1");
BigDecimal b = new BigDecimal("0.2");
a.add(b);                           // → 0.3 (정확!)
```

주의: 생성자에 반드시 **문자열**을 전달해야 한다.

```java
new BigDecimal(0.1)    // ❌ double 0.1이 먼저 만들어져 이미 오차 발생
new BigDecimal("0.1")  // ✅ 10진법 문자열을 그대로 파싱
```

---

### 3-3. 언어별 10진법 타입

```python
# Python
from decimal import Decimal
Decimal('0.1') + Decimal('0.2')  # → Decimal('0.3')
```

```csharp
// C#
decimal a = 0.1m + 0.2m;  // → 0.3 (정확!)
```

```javascript
// JavaScript — 네이티브 미지원, 라이브러리 사용
import { Decimal } from 'decimal.js';
new Decimal('0.1').plus('0.2');  // → '0.3'
```

---

### 3-4. Epsilon 비교 — 오차를 인정하는 방식

금융이 아닌 일반 프로그래밍에서는 **"충분히 가까운지"** 를 비교한다.

```java
// ❌ 절대 하면 안 됨
if (0.1 + 0.2 == 0.3)  // → false!

// ✅ 오차 허용 범위를 두고 비교
double epsilon = 0.0000001;
if (Math.abs(result - expected) < epsilon)  // → true!
```

게임 물리 엔진, 그래픽 렌더링, 과학 시뮬레이션에서 주로 사용한다.

---

### 3-5. 실무 선택 기준

```
돈 계산 (금융, 결제, 회계)
  → BigDecimal 또는 정수 변환 (오차 절대 불가)

과학/공학 계산
  → double + epsilon 비교 (속도 중요, 약간의 오차 허용)

일반 웹 개발
  → 대부분 double로 충분, 금액만 BigDecimal/정수 처리

게임/그래픽
  → float(32비트)로도 충분, 속도가 최우선
```

## 🔍 Part 4. BigDecimal은 내부에서 어떻게 동작하는가?

### 4-1. 핵심 구조: 정수 + 소수점 위치

```
BigDecimal = unscaledValue × 10^(-scale)

unscaledValue: 소수점을 무시한 정수값
scale: 소수점을 왼쪽으로 몇 칸 옮길지
```

```
BigDecimal("0.1")     → {unscaledValue: 1,   scale: 1}  → 1 × 10⁻¹
BigDecimal("3.14")    → {unscaledValue: 314, scale: 2}  → 314 × 10⁻²
BigDecimal("0.00025") → {unscaledValue: 25,  scale: 5}  → 25 × 10⁻⁵
```

2진법 변환을 하지 않고, **10진법 숫자를 정수로 보관하면서 소수점 위치만 따로 기억**하는 구조다.

---

### 4-2. Java 내부 필드

```java
public class BigDecimal extends Number {
    private final long intCompact;   // 작은 수 (long 범위 이내)
    private BigInteger intVal;       // 큰 수 (long 범위 초과)
    private final int scale;         // 소수점 위치
}
```

작은 수는 `long`(64비트 정수)으로 빠르게, 큰 수는 `BigInteger`(int 배열)로 무한 확장 가능하게 처리한다.

---

### 4-3. 연산: 정수끼리만 계산

**덧셈 (scale이 같은 경우)**

```
0.1 + 0.2:
  {unscaledValue: 1, scale: 1} + {unscaledValue: 2, scale: 1}
  → 정수 덧셈: 1 + 2 = 3
  → 결과: {unscaledValue: 3, scale: 1} → 0.3
```

**덧셈 (scale이 다른 경우)**

```
0.1 + 0.25:
  {unscaledValue: 1, scale: 1} + {unscaledValue: 25, scale: 2}

  1단계: scale 맞추기 (작은 쪽을 큰 쪽에 맞춤)
    1 × 10 = 10 → {unscaledValue: 10, scale: 2}

  2단계: 정수 덧셈
    10 + 25 = 35

  결과: {unscaledValue: 35, scale: 2} → 0.35
```

**곱셈**

```
3.14 × 0.5:
  {unscaledValue: 314, scale: 2} × {unscaledValue: 5, scale: 1}

  정수 곱셈: 314 × 5 = 1570
  scale 덧셈: 2 + 1 = 3 (지수법칙: 10⁻² × 10⁻¹ = 10⁻³)

  결과: {unscaledValue: 1570, scale: 3} → 1.570
```

모든 연산이 **정수 연산 + scale 정수 연산**뿐이다. 2진법 소수가 개입할 여지가 없다.

---

### 4-4. 출력: 문자열 조작

`{unscaledValue: 3, scale: 1}`을 "0.3"으로 표시할 때, `3 ÷ 10`을 계산하지 않는다. 나눗셈을 하면 다시 2진법 소수가 등장하기 때문이다.

```
toString() 동작:

1단계: 정수를 문자열로 변환
  3 → "3"

2단계: scale만큼 소수점 문자를 찍어넣음
  scale=1 → 끝에서 1칸 앞에 '.' 삽입
  "3" → "0.3"
```

```
{unscaledValue: 314, scale: 2}
  314 → "314" → 끝에서 2칸 앞에 '.' → "3.14"

{unscaledValue: 25, scale: 5}
  25 → "25" → 자릿수 부족, 앞에 0 채움 → "00025" → "0.00025"
```

수학적 나눗셈 없이 **순수 문자열 조작**으로 소수점을 표현한다.

---

### 4-5. 전체 생애주기

```
입력:   "3.14" → '.' 위치 파싱 → scale=2, 정수 314
저장:   {unscaledValue: 314(2진법 정수, 정확), scale: 2(2진법 정수, 정확)}
연산:   정수끼리 연산 → 2진법 정수 영역 → 정확
출력:   314 → "314" → '.' 삽입 → "3.14"

전 과정에서 2진법 "소수"가 한 번도 등장하지 않음
```

### 4-6. BigDecimal의 단점: 느리다

```
double 연산:
  CPU 내부 전용 하드웨어(FPU)가 한 클럭에 처리 → 나노초

BigDecimal 연산:
  scale 비교 → 10의 거듭제곱 곱셈 → 정수 덧셈 → 객체 생성
  → double 대비 수십~수백 배 느림
```

정확한 대신 느리다. 금융처럼 정확도가 생명인 곳에서만 사용하고, 게임이나 그래픽처럼 속도가 중요한 곳에서는 double을 쓰는 이유다.

---

### 4-7. 비유로 정리

```
double = 진짜 피자를 칼로 자르는 요리사
  → 자를 때마다 미세한 오차 발생

BigDecimal = 피자를 안 자르는 회계사
  → 메모장에 "1판, 10등분 중 3조각"이라고 기록만 함
  → 계산할 때도 메모끼리 연산
  → 보여줄 때도 메모를 읽어서 글자로 표시
  → 피자를 실제로 자르는 순간이 없음
```

---

## 💡 한 줄 정리

> 0.1 + 0.2 ≠ 0.3은 **10진법 소수가 2진법에서 무한소수가 되기 때문**이다.  
> BigDecimal은 소수를 정수로 분해하여 **2진법 소수가 등장할 여지를 원천 차단**하며, 입력부터 출력까지 문자열과 정수만으로 동작한다.
