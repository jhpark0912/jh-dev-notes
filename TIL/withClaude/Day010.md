# Day 010. HTTPS는 어떻게 데이터를 지켜줄까?

> 🗓️ 2026-03-09  
> 🏷️ Tags: `HTTPS` `TLS` `대칭키` `비대칭키` `핸드셰이크` `Let'sEncrypt` `SSL인증서`

---

## 🤔 질문

HTTPS는 어떻게 데이터를 암호화할까?  
대칭키와 비대칭키는 뭐가 다르고, TLS 핸드셰이크는 왜 둘을 조합할까?  
Let's Encrypt는 왜 무료이고, Google/Meta는 왜 이를 후원할까?

---

## 📌 핵심 요약

HTTPS(TLS)는 **비대칭키로 대칭키를 안전하게 교환**하고, **대칭키로 실제 데이터를 빠르게 암호화**하는 하이브리드 설계다.  
비대칭키의 느림을 처음 1번만 감수하고, 이후 모든 통신은 1,000배 빠른 대칭키를 사용한다.  
Let's Encrypt는 인증서를 무료화하여 HTTPS 보급률을 30%에서 95%로 끌어올렸으며, 이는 후원사들의 **광고 수익 증가 전략**과 맞닿아 있다.

---

## 🔍 Part 1. 두 가지 암호화 방식

### 1-1. 대칭키 암호화 — 같은 키로 잠그고 연다

```
암호화: "안녕하세요" + 키(ABC123) → "xK9#mL2$"
복호화: "xK9#mL2$" + 키(ABC123) → "안녕하세요"

→ 잠글 때와 열 때 같은 키 사용
→ 빠르다! 대량 데이터 처리에 적합
→ 문제: 그 키를 어떻게 안전하게 전달하지?
```

해커가 네트워크를 감청하고 있는 상태에서 키를 보내면 복사당한다. 이것이 **키 교환 문제**다.

---

### 1-2. 비대칭키 암호화 — 키가 두 개 (공개키 + 개인키)

잠그는 키와 여는 키를 다르게 만드는 발상이다.

```
공개키 (Public Key):  누구에게나 공개해도 됨
개인키 (Private Key): 서버만 보유, 절대 공개 안 함

암호화: "안녕하세요" + 공개키 → "xK9#mL2$"
복호화: "xK9#mL2$" + 개인키 → "안녕하세요"

→ 공개키로 잠근 건 개인키로만 열 수 있음
→ 공개키가 유출돼도 상관없음
```

비유하면 **우체통** 방식이다. 누구나 편지를 넣을 수 있지만(공개키로 암호화), 꺼내는 건 열쇠를 가진 서버(개인키)만 가능하다.

단점은 **느리다**는 것. 대칭키보다 약 1,000배 느려서 대량 데이터 통신에는 부적합하다.

```
대칭키:   빠르다 ✅  키 전달이 위험 ❌
비대칭키:  느리다 ❌  키 전달이 안전 ✅

→ 둘을 조합하면 양쪽의 장점만 취할 수 있다
```

---

## 🔍 Part 2. TLS 핸드셰이크 — 둘을 조합한 하이브리드 설계

### 2-1. 왜 하이브리드인가?

비대칭키로 모든 데이터를 암호화하면 매 요청이 느리다. 하이브리드 방식은 **비대칭키를 1번만 사용**하여 대칭키를 안전하게 교환하고, 이후 모든 통신은 빠른 대칭키로 처리한다.

```
1KB 데이터 암호화 기준:
  대칭키 (AES-256):     약 0.001ms
  비대칭키 (RSA-2048):   약 1ms → 1,000배 느림

웹 페이지 로딩 시 요청 100개:
  전부 비대칭키: 100 × 1ms = 100ms
  하이브리드:   1ms(핸드셰이크) + 100 × 0.001ms = 1.1ms → 약 100배 차이
```

비유하면, 비대칭키로만 통신하는 건 **매번 금고에서 돈을 꺼내는 것**이고, 하이브리드는 **금고에서 지갑에 한 번 옮겨 담고 이후는 지갑에서 쓰는 것**이다.

---

### 2-2. 핸드셰이크 전체 과정

**Step 1: Client Hello**

브라우저가 서버에 인사한다. 지원하는 TLS 버전, 암호화 알고리즘 목록, 랜덤 숫자(Client Random)를 전송한다.

**Step 2: Server Hello + 인증서 전달**

서버가 알고리즘을 선택하고, SSL 인증서와 랜덤 숫자(Server Random)를 보낸다. 인증서 안에 **서버의 공개키**가 들어있다.

```
SSL 인증서 내용:
┌──────────────────────────────┐
│ 도메인: www.google.com       │
│ 발급자: Let's Encrypt        │
│ 유효기간: 2025-01-01 ~ 12-31 │
│ 서버 공개키: MIIBIjANBgk...  │ ← 핵심
│ 디지털 서명: SHA256RSA...     │
└──────────────────────────────┘
```

**Step 3: 인증서 검증**

브라우저가 인증서의 진위를 확인한다. 브라우저에 내장된 신뢰할 수 있는 CA(Certificate Authority, 인증 기관) 목록에서 발급자를 찾고, CA의 공개키로 서명을 검증한다.

비유하면, 인증서는 **신분증**, CA는 **정부(신분증 발급 기관)**, 브라우저는 **경비원(정부 발급인지 확인)**이다. 해커가 가짜 인증서를 만들어도 CA의 서명이 없으면 거부된다.

**Step 4: 대칭키 생성 및 교환 (Diffie-Hellman)**

TLS 1.3에서는 Diffie-Hellman 키 교환을 사용한다. 브라우저와 서버가 각각 비밀 값을 가진 채 공개 값만 교환하여 **같은 대칭키**를 만들어낸다.

```
Diffie-Hellman 비유 — 색깔 섞기:

공개 색: 노란색 (해커도 앎)
브라우저 비밀 색: 빨간색 / 서버 비밀 색: 파란색

교환:
  브라우저: 노란+빨간 = 주황색 → 서버에 전송
  서버:    노란+파란 = 초록색 → 브라우저에 전송

해커가 보는 것: 노란색, 주황색, 초록색
  → 섞인 색에서 원래 색을 분리하는 건 불가능!

최종 대칭키:
  브라우저: 초록색 + 빨간색 = 갈색
  서버:    주황색 + 파란색 = 갈색
  → 둘 다 같은 "갈색"(대칭키) 생성! 해커는 만들 수 없음
```

실제로는 매우 큰 소수와 수학적 연산(이산 로그 문제)을 사용하여, 공개값에서 비밀값을 역산하는 것이 극도로 어렵다.

**Step 5: 대칭키로 실제 통신 시작**

이제 둘만 아는 대칭키로 모든 데이터를 빠르게 암호화하여 주고받는다.

---

### 2-3. 전체 흐름 요약

```
브라우저                          서버
   │──── Client Hello ──────────→│  "암호화 하고 싶어"
   │←── Server Hello + 인증서 ── │  "좋아, 여기 내 인증서"
   │  인증서 검증 (CA 확인)        │
   │──── 키 교환 데이터 ─────────→│  Diffie-Hellman
   │←── 키 교환 데이터 ──────────│
   │  [대칭키 생성 완료!]          │  [대칭키 생성 완료!]
   │════ 암호화된 데이터 ═══════→ │  대칭키로 빠르게 통신
   │←═══ 암호화된 데이터 ═══════ │
```

핸드셰이크(비대칭키)는 약 50ms로 처음 1번만, 이후 모든 통신은 빠른 대칭키로 처리된다.

## 🔍 Part 3. 핸드셰이크 실패와 경고

### 3-1. 접속 자체가 차단되는 경우

```
TLS 버전 불일치:    브라우저(TLS 1.3) ↔ 서버(TLS 1.0만 지원)
알고리즘 불일치:    안전한 알고리즘으로 합의 불가
인증서 도메인 불일치: 접속 주소와 인증서 도메인이 다름
→ ERR_SSL_VERSION_OR_CIPHER_MISMATCH 등 에러로 차단
```

### 3-2. 경고만 표시되는 경우 (사용자 선택)

```
인증서 만료:        유효기간이 지남 → 암호화는 동작, 보증이 안 됨
자체 서명 인증서:   CA가 아닌 서버가 직접 만든 인증서 → 개발 환경에서 흔함
HTTP 사이트:       인증서 자체가 없음 → "주의 요함" 표시
```

자체 서명 인증서는 암호화 자체는 정상 동작한다. 단지 "이 서버가 진짜 맞는지" CA가 보증하지 않을 뿐이다. 내가 아는 서버면 괜찮지만, 모르는 사이트의 경고는 절대 무시하면 안 된다.

---

### 3-3. 실무에서 자주 만나는 상황

```
개발 서버 자체 서명 인증서:  → 매번 경고, 운영에서는 정식 인증서 필수
인증서 갱신 깜빡함:         → Let's Encrypt 90일 만료, 자동 갱신 설정 필수
잘못된 인증서 설치:         → 도메인과 인증서 불일치로 경고/차단
```

## 🔍 Part 4. Let's Encrypt — HTTPS 대중화의 핵심

### 4-1. Let's Encrypt 이전

SSL 인증서는 원래 유료였다. 연 $60~$1,500 수준으로, 소규모 사이트에는 부담이 됐다. 2015년 기준 HTTPS 사이트 비율은 약 30%에 불과했다.

---

### 4-2. Let's Encrypt의 등장 (2015년)

비영리 단체 ISRG가 운영하는 **무료 인증서 발급 기관(CA)**이다. Mozilla, Google, Meta, Cisco 등이 후원한다.

```
유료 CA: DV(도메인 검증), OV(조직 검증), EV(확장 검증) 제공
         → 사람이 심사 → 인건비 발생 → 유료

Let's Encrypt: DV만 제공
         → 전 과정 자동화(ACME 프로토콜) → 인건비 제로
         → 후원사가 인프라 비용 지원 → 무료 가능
```

---

### 4-3. 도메인 검증 방식 (ACME 프로토콜)

```
HTTP-01 챌린지:

1. "example.com 인증서 발급해줘"
2. Let's Encrypt: "서버의 이 경로에 이 파일을 올려봐"
3. 서버에 해당 파일 배치
4. Let's Encrypt가 URL로 접속해서 파일 확인
   → "이 서버를 제어할 수 있는 사람이구나" → 인증서 발급!
```

---

### 4-4. 실무에서 사용

```bash
# certbot으로 인증서 발급 (1분)
$ sudo certbot --nginx -d example.com

# 자동 갱신 설정 (90일마다 필요)
$ sudo certbot renew --dry-run
```

---

### 4-5. Let's Encrypt의 한계

```
Let's Encrypt:
  ✅ 무료, 자동 발급/갱신
  ✅ 암호화 품질은 유료와 동일
  ❌ DV(도메인 검증)만 가능, 조직 실체 확인 안 됨
  ❌ 유효기간 90일, 자동 갱신 필수

유료 CA:
  ✅ OV/EV 인증으로 조직 실체 보증
  ✅ 유효기간 1년, 배상 보험 포함
  ❌ 비쌈
```

```
선택 기준:
  개인/스타트업/소규모:  → Let's Encrypt (무료, 충분)
  기업 공식/금융/결제:   → 유료 OV/EV 인증서 (조직 검증, 신뢰도)
  사내 개발 서버:       → 자체 서명 인증서
```

## 🔍 Part 5. Google/Meta가 Let's Encrypt를 후원하는 진짜 이유

### 5-1. 인증서 장사보다 큰 이익이 있다

```
Google 수익의 약 80%: 광고
Meta 수익의 약 97%:  광고

사용자가 인터넷을 많이, 안전하게 쓸수록 → 광고 수익 증가
HTTPS 보급 = 인터넷 신뢰도 증가 = 사용량 증가 = 광고 수익 증가
```

비유하면, 쇼핑몰 주인이 주차장을 무료로 제공하는 것과 같다. 주차비로 버는 것보다 손님이 많아져서 매장 매출이 올라가는 게 훨씬 크다.

---

### 5-2. Google의 구체적 이익

**데이터 독점 강화:** HTTP 시대에는 ISP가 사용자 검색 데이터를 중간에서 수집할 수 있었다. HTTPS가 보편화되면 ISP가 내용을 볼 수 없으므로, 검색 데이터는 Google만 갖게 된다.

**HTTPS 강제 생태계 조성:** Google은 2014년 HTTPS를 검색 순위 시그널로 채택하고, Chrome에서 HTTP 사이트에 "안전하지 않음" 경고를 표시했다. Let's Encrypt 후원으로 전환 장벽을 제거하고, 검색 순위와 브라우저 경고로 전환 압박을 동시에 가했다.

---

### 5-3. Meta의 구체적 이익

Facebook/Instagram에서 공유되는 외부 링크가 HTTP면 보안 사고 위험이 있고, 이는 플랫폼 신뢰도 하락으로 이어진다. 광고 클릭 후 HTTPS 쇼핑몰로 연결되면 결제 전환율이 높아져 광고 효과가 개선되고, 광고주가 더 많은 예산을 집행한다.

---

### 5-4. 후원 비용 대비 이익

```
Let's Encrypt 연간 운영: 약 $3.6M → 후원사끼리 분담 → 회사당 수억 원
Google 연간 광고 수익:   약 $300B
Meta 연간 광고 수익:     약 $130B

→ 후원 비용은 수익의 0.001%도 안 됨
→ 커피값으로 인터넷 전체 보안을 높여 자기 사업 환경을 개선하는 것
```

---

### 5-5. 같은 논리의 다른 예시

```
Google이 무료로 공개/후원하는 것들:
  Chrome (브라우저)    → 사용자가 Google 검색을 더 많이
  Android (모바일 OS)  → 모바일에서 Google 서비스 사용
  Kubernetes (컨테이너) → 클라우드 사용 증가 → GCP 수익
  Let's Encrypt (인증서) → 웹 보안 강화 → 인터넷 사용 증가

→ 전부 "본업의 파이를 키우는" 전략
→ 직접 돈 버는 것보다 생태계를 키우는 게 더 큰 이익
```

---

### 5-6. Let's Encrypt의 영향

```
HTTPS 사이트 비율:
  2015년 (이전): 약 30%
  2020년:       약 80%
  2025년:       약 95%+
```

현재 전 세계 3억 개 이상의 인증서를 발급한 세계 최대 CA이며, "HTTPS가 기본"이 된 데 가장 큰 공헌을 했다.

---

## 💡 한 줄 정리

> HTTPS는 **비대칭키의 안전함으로 대칭키를 교환하고, 대칭키의 속도로 데이터를 보호**하는 하이브리드 설계다. Let's Encrypt가 인증서를 무료화하여 HTTPS를 대중화했고, 이를 후원하는 빅테크 기업들은 자선이 아니라 **인터넷 전체의 보안 강화가 자사 광고 생태계의 성장으로 직결**되기 때문이다.
