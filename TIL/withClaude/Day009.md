# Day 009. Docker는 가상머신과 뭐가 다를까? — 컨테이너의 진짜 원리

> 🗓️ 2026-03-06  
> 🏷️ Tags: `Docker` `컨테이너` `네임스페이스` `cgroups` `UnionFS` `VM` `Linux커널` `macOS`

---

## 🤔 질문

Docker는 가상머신과 뭐가 다르고, 내부에서 어떻게 동작할까?  
왜 모든 OS에서 결국 Linux를 바라보며, macOS 터미널은 Linux인가?  
네임스페이스와 cgroups는 chmod 같은 권한 설정과 뭐가 다를까?

---

## 📌 핵심 요약

Docker는 새로운 기술을 발명한 게 아니라, Linux 커널의 **네임스페이스(격리)와 cgroups(자원 제한)** 을 누구나 쓸 수 있게 포장한 것이다.  
VM이 OS를 통째로 올리는 것과 달리, 컨테이너는 **커널을 공유하고 앱만 격리**하기 때문에 가볍고 빠르다.  
Docker가 Linux를 바라보는 이유는 **컨테이너의 핵심 기술이 Linux 커널에만 존재**하기 때문이다.

---

## 🔍 Part 1. Docker의 탄생 — 역사와 배경

### 1-1. Docker 이전의 고통

```
개발자 PC:   Java 11, MySQL 5.7, Ubuntu 20.04
테스트 서버:  Java 8, MySQL 8.0, CentOS 7
운영 서버:   Java 11, MySQL 5.7, RHEL 8

→ "내 컴퓨터에서는 되는데?" 반복
```

가상머신(VM)으로 해결했지만, OS를 통째로 올려야 해서 무거웠다. VM 하나에 수 GB 메모리, 수십 GB 디스크, 부팅 수 분이 필요했다.

---

### 1-2. 컨테이너 기술의 역사 — Docker 이전

컨테이너 아이디어는 Docker보다 수십 년 전부터 있었다.

```
1979년  chroot (Unix) → 파일 시스템 경로를 제한하는 "감옥" 개념의 시초
2000년  FreeBSD Jails → chroot 확장, 네트워크·프로세스까지 격리
2002년  Mount 네임스페이스 → 호스팅 서비스의 파일 격리 목적
2006년  cgroups (Google 개발) → 서버 자원 관리 목적
2008년  LXC → 네임스페이스 + cgroups를 결합한 최초의 Linux 컨테이너
2013년  Docker 등장!
```

---

### 1-3. Docker의 등장과 혁신

dotCloud이라는 PaaS 회사의 내부 프로젝트로 시작. 창업자 솔로몬 하익스(Solomon Hykes)가 2013년 PyCon에서 처음 공개했다.

Docker의 혁신은 기술이 아니라 **UX와 생태계**에 있었다. 컨테이너 기술 자체는 이미 있었지만, Dockerfile 몇 줄이면 환경이 구성되고 Docker Hub로 이미지를 공유할 수 있게 만들었다.

```dockerfile
FROM openjdk:17
COPY app.jar /app.jar
CMD ["java", "-jar", "/app.jar"]
```

비유하면, Linux 컨테이너 기술은 **반제품 상태의 엔진**이었고, Docker가 거기에 핸들, 의자, 바퀴를 달아서 **자동차로 만든** 것이다.

---

### 1-4. Docker 이후의 생태계 변화

```
2013년  Docker 0.1 (내부적으로 LXC 사용)
2014년  자체 컨테이너 런타임(libcontainer) 개발, LXC 탈피
2015년  OCI(Open Container Initiative) 설립 → 컨테이너 표준 규격 제정
2016년  containerd 분리 → 컨테이너 런타임을 독립 프로젝트로
2017년  Kubernetes가 컨테이너 오케스트레이션의 승자로 확정
2020년  Kubernetes가 dockershim 지원 중단 (Docker 이미지는 계속 사용)
```

## 🔍 Part 2. 컨테이너의 핵심 원리

### 2-1. 네임스페이스(Namespace) — "뭘 볼 수 있는가"를 격리

프로세스가 시스템의 어디까지 볼 수 있는지를 제한한다.

```
호스트에서: PID 1(init), PID 345(nginx), PID 5847(java), PID 6103(python)
컨테이너 A: PID 1(java) ← 자기 프로세스만 보임, nginx·python 존재를 모름
컨테이너 B: PID 1(python) ← 자기만 보임, A의 존재조차 모름
```

Linux가 제공하는 네임스페이스 종류:

```
PID  → 프로세스 목록 격리 (자기 프로세스만 보임)
NET  → 네트워크 격리 (독립적인 IP, 포트)
MNT  → 파일 시스템 격리 (독립적인 디렉토리 구조)
UTS  → 호스트명 격리 (컨테이너마다 다른 hostname)
IPC  → 프로세스 간 통신 격리
USER → 사용자 ID 격리 (컨테이너 root ≠ 호스트 root)
```

네임스페이스는 컨테이너를 위해 만든 게 아니라, 2002년부터 2013년까지 **각각의 격리 필요에 의해 독립적으로 하나씩 추가**된 것이다.

---

### 2-2. cgroups(Control Groups) — "얼마나 쓸 수 있는가"를 제한

프로세스의 하드웨어 자원 사용량을 제한한다.

```
CPU:       "이 컨테이너는 CPU 50%까지만"
메모리:     "이 컨테이너는 512MB까지만"
디스크 I/O: "이 컨테이너는 초당 100MB까지만"
네트워크:   "이 컨테이너는 초당 10Mbps까지만"
```

2006년 Google이 내부 서버 운영을 위해 개발했다. 수만 개의 프로세스 중 하나가 자원을 독식하는 것을 방지하기 위한 목적이었으며, **컨테이너와 무관하게 탄생**했다. Docker 없이도 systemd, Android 등에서 단독으로 사용된다.

---

### 2-3. chmod와는 다른 층위의 개념

```
chmod 755 (파일 권한):
  → "이 파일을 누가 읽고/쓰고/실행할 수 있나?"
  → 파일 단위의 접근 제어
  → 권한 거부 시 "Permission denied" (파일 존재는 앎)

네임스페이스:
  → "이 프로세스가 시스템의 뭘 볼 수 있나?"
  → 프로세스의 시야 자체를 제한
  → 다른 컨테이너의 존재 자체를 모름

cgroups:
  → "이 프로세스가 자원을 얼마나 쓸 수 있나?"
  → CPU, 메모리 등 사용량 상한 설정
  → 초과 시 프로세스 강제 종료(OOM Kill)
```

비유하면, chmod는 **각 방의 출입 카드**, 네임스페이스는 **층별로 다른 엘리베이터(다른 층 존재를 모름)**, cgroups는 **부서별 전기·수도 사용량 제한**이다.

---

### 2-4. 독립적으로 탄생해 합쳐진 것

```
밀가루(네임스페이스): 원래 보안 격리용으로 개발
설탕(cgroups): 원래 서버 자원 관리용으로 개발
달걀(Union FS): 원래 파일 시스템 보조용으로 개발

LXC (2008): "이 세 개를 합치면 컨테이너가 되겠는데?" → 최초 결합
Docker (2013): "그 컨테이너를 누구나 쓸 수 있게 하자!" → 대중화
```

## 🔍 Part 3. VM vs 컨테이너

### 3-1. 구조 비교

```
가상머신:                          컨테이너:
┌─────────┐ ┌─────────┐          ┌─────────┐ ┌─────────┐
│   App   │ │   App   │          │   App   │ │   App   │
│  Libs   │ │  Libs   │          │  Libs   │ │  Libs   │
│ Guest OS│ │ Guest OS│ ← OS 2개  └────┬────┘ └────┬────┘ ← OS 없음
└────┬────┘ └────┬────┘          ─────┴──────────┴─────
─────┴──────────┴─────           │   컨테이너 엔진 (Docker) │
│      하이퍼바이저      │          │       Host OS            │
│       Host OS        │          │       하드웨어             │
│       하드웨어        │          └──────────────────────────┘
└──────────────────────┘
```

```
                VM              컨테이너
OS              각각 별도 OS     호스트 커널 공유
시작 시간       수 분            수 초 (밀리초 가능)
이미지 크기     수 GB            수십~수백 MB
메모리 사용     OS당 수백 MB     앱이 쓰는 만큼만
격리 수준       강함 (완전 분리)  상대적으로 약함 (커널 공유)
밀도            서버당 수~수십 개 서버당 수백~수천 개
```

---

### 3-2. 컨테이너가 호스트 커널을 공유한다는 의미

OS는 **커널(하드웨어 제어)** 과 **사용자 영역(앱, 라이브러리, 유틸리티)** 으로 나뉜다.

컨테이너가 공유하는 건 커널이고, 컨테이너마다 다른 건 사용자 영역이다. Docker 이미지에 "Ubuntu"가 들어있는 건 OS 전체가 아니라 **사용자 영역만** 포함된 것이다.

```dockerfile
FROM ubuntu:22.04
# → Ubuntu 커널이 아니라 사용자 영역(apt, bash, 라이브러리 등)만 포함 → 약 78MB
# → VM의 Ubuntu는 커널 + 드라이버 + 부트로더 포함 → 약 2~4GB
```

---

## 🔍 Part 4. Docker가 Linux를 바라보는 이유

### 4-1. 컨테이너 기술이 Linux 커널에만 있다

```
Linux 커널:  ✅ 네임스페이스 ✅ cgroups ✅ Union FS → 컨테이너 가능
macOS 커널:  ❌ 네임스페이스 ❌ cgroups → 컨테이너 불가
Windows 커널: ❌ Linux 네임스페이스/cgroups 없음 (자체 컨테이너는 별도 구현)
```

가벼워서가 아니라, **컨테이너를 만드는 부품이 Linux에만 존재**한다. 서버 시장의 77%가 Linux이므로, 서버 기술인 컨테이너가 자연스럽게 Linux 중심으로 발전했다.

---

### 4-2. Mac/Windows에서 Docker가 돌아가는 진짜 구조

```
Linux:   Linux → Docker → 컨테이너 (VM 불필요, 가장 효율적)
Mac:     macOS → 경량 Linux VM (숨겨져 있음) → Docker → 컨테이너
Windows: Windows → WSL2 (Linux) → Docker → 컨테이너
```

Mac/Windows에서는 **백그라운드에서 Linux VM이 자동으로 뜨고**, Docker는 그 안에서 실행된다. 그래서 Linux 서버에서보다 약간 느리고 메모리를 더 사용한다.

---

### 4-3. Mac 터미널은 Linux가 아니다

Mac 터미널은 **macOS 자체의 Unix 쉘**이다. Docker의 Linux VM과는 무관하다.

```bash
# Mac 터미널에서
$ uname -s
Darwin         ← macOS 커널

# Docker 컨테이너 안에서
$ docker run ubuntu uname -s
Linux          ← Linux 커널

→ 두 개가 서로 다른 커널 위에서 동작
```

명령어가 비슷한 이유는 macOS와 Linux가 **같은 조상(Unix)** 을 갖고 있기 때문이다.

```
Unix (1969)
  ├── BSD → macOS (Darwin) : Unix 직계 후손
  └── Linux (1991) : Unix를 참고해 새로 만듦
```

---

### 4-4. macOS가 개발에 유리한 이유

macOS와 Linux가 Unix 계열이므로 경로 체계, 셸, 권한, 명령어가 자연스럽게 호환된다.

```
Mac → Linux 서버 배포:
  셸 스크립트: 거의 그대로 동작 ✅
  파일 경로: /로 시작하는 구조 동일 ✅
  권한 체계: chmod, chown 동일 ✅

Windows → Linux 서버 배포:
  경로 구분자: \ vs / 충돌 ❌
  줄바꿈: CRLF vs LF 충돌 ❌
  셸 문법: PowerShell ≠ bash ❌
```

Windows는 WSL2로 Linux를 품어 격차를 줄이고 있지만, "Windows 안에 Linux를 따로 설치"하는 구조라 macOS처럼 자연스럽지는 않다.

## 🔍 Part 5. Docker 이미지의 레이어 구조

### 5-1. 이미지는 읽기 전용 레이어의 스택

```dockerfile
FROM ubuntu:22.04          # 레이어 1: Ubuntu 사용자 영역 (78MB)
RUN apt-get install java   # 레이어 2: Java 설치 (+200MB)
COPY app.jar /app.jar      # 레이어 3: 앱 복사 (+50MB)
```

```
이미지:                      컨테이너 실행 시:
┌─────────────────────┐     ┌─────────────────────┐
│ 레이어 3: app.jar   │     │ 쓰기 가능 레이어     │ ← 컨테이너 전용
├─────────────────────┤     ├─────────────────────┤
│ 레이어 2: Java      │     │ 레이어 3: app.jar   │ ← 읽기 전용
├─────────────────────┤     ├─────────────────────┤
│ 레이어 1: Ubuntu    │     │ 레이어 2: Java      │
└─────────────────────┘     ├─────────────────────┤
                            │ 레이어 1: Ubuntu    │
                            └─────────────────────┘
```

---

### 5-2. 레이어 공유로 디스크 절약

같은 베이스 이미지를 쓰는 컨테이너가 여러 개면, 공통 레이어를 한 번만 저장하고 공유한다.

```
앱 A: Ubuntu(78MB) + Java(200MB) + app-a(50MB) = 328MB
앱 B: Ubuntu(78MB) + Java(200MB) + app-b(30MB) = 308MB

레이어 없이: 328 + 308 = 636MB
레이어 공유: 78 + 200 + 50 + 30 = 358MB → 278MB 절약
```

---

### 5-3. 빌드 캐싱으로 속도 향상

변경되지 않은 레이어는 캐시를 재사용한다. 그래서 **자주 바뀌는 것을 아래에** 두는 것이 중요하다.

```dockerfile
# ❌ 코드 변경 시 의존성 설치부터 재빌드
COPY . /app
RUN pip install -r requirements.txt

# ✅ 의존성은 캐시, 코드만 재빌드
COPY requirements.txt /app/
RUN pip install -r requirements.txt  # 캐시 재사용!
COPY . /app                          # 이것만 재빌드
```

---

### 5-4. Union File System과 Copy-on-Write

여러 레이어를 하나의 파일 시스템으로 합치는 기술이 Union FS(overlay2)다.

파일을 읽을 때는 위 레이어부터 아래로 탐색하고, 수정할 때는 읽기 전용 레이어를 건드리지 않고 **쓰기 레이어에 복사한 뒤 수정(Copy-on-Write)** 한다.

이 덕분에 같은 이미지로 컨테이너를 100개 띄워도 이미지를 100번 복사하지 않는다. 읽기 전용 레이어는 공유하고, 각 컨테이너의 쓰기 레이어만 별도로 갖는다.

---

## 🔍 Part 6. Docker의 기술 스택 전체 그림

```
1. 네임스페이스  → "뭘 볼 수 있나" 격리 (프로세스, 네트워크, 파일 등)
2. cgroups      → "얼마나 쓸 수 있나" 제한 (CPU, 메모리, I/O)
3. Union FS     → 레이어를 겹겹이 쌓아 하나의 파일 시스템으로 합성
4. 이미지       → 읽기 전용 레이어의 묶음 (Dockerfile로 정의)
5. 컨테이너     → 이미지 + 쓰기 가능 레이어 + 네임스페이스 + cgroups

이 중 Docker가 새로 만든 것: 없음!
전부 Linux 커널의 기존 기능을 편하게 포장한 것.
Docker의 진짜 혁신: Dockerfile, 이미지 빌드/공유 생태계(Docker Hub)
```

---

## 💡 한 줄 정리

> Docker는 Linux 커널의 **네임스페이스·cgroups·Union FS**를 조합해 만든 컨테이너를 누구나 쓸 수 있게 포장한 것이다. 이 기술들은 각각 보안 격리, 자원 관리, 파일 시스템이라는 **별개의 목적으로 탄생**했고, 합쳐져서 컨테이너가 되었다. Docker가 Linux를 바라보는 이유는 가벼움이 아니라 **이 부품들이 Linux에만 존재**하기 때문이다.
