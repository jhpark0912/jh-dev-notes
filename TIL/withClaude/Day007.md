# Day 007. 왜 비밀번호를 DB에 그대로 저장하면 안 될까?

> 🗓️ 2026-03-04  
> 🏷️ Tags: `해싱` `솔트` `bcrypt` `Argon2` `scrypt` `Jasypt` `비밀번호보안`

---

## 🤔 질문

비밀번호를 DB에 왜 그대로 저장하면 안 될까?  
해싱과 암호화는 뭐가 다르고, bcrypt에서 Argon2까지 어떻게 진화해왔을까?  
Jasypt는 어디에 쓰는 거고, 온프레미스 환경에서는 어떤 대안이 있을까?

---

## 📌 핵심 요약

비밀번호 보안은 **평문 → 단순 해시 → 솔트 + 해시 → bcrypt → Argon2** 순으로 진화했다.  
핵심은 "DB가 뚫려도 비밀번호를 알 수 없게" 만드는 것이며, 해싱(단방향)과 암호화(양방향)는 용도가 다르다.  
Jasypt는 **설정값 암호화** 도구이지 비밀번호 해싱 도구가 아니며, 사용자 비밀번호에 쓰면 위험하다.

---

## 🔍 Part 1. 비밀번호 저장의 진화

### 1-1. 평문 저장 — 최악의 방법

```sql
users 테이블:
┌──────────┬──────────────┐
│ username │ password     │
├──────────┼──────────────┤
│ 홍길동    │ mypassword123│
│ 김철수    │ qwerty       │
└──────────┴──────────────┘
```

DB가 유출되면 모든 사용자의 비밀번호가 즉시 노출된다. DB 유출은 놀라울 정도로 자주 발생한다. 2012년 LinkedIn 650만 건, 2016년 1억 1천만 건 등. **DB는 언젠가 뚫린다고 가정하고, 뚫려도 비밀번호를 알 수 없게** 만드는 것이 핵심이다.

---

### 1-2. 단순 해시 — 좋아 보이지만 부족

SHA-256은 단방향이라 해시값에서 원본을 역산할 수 없다. 그러나 해커는 역산이 아니라 **대조**로 뚫는다.

**공격 방법 1: 레인보우 테이블** — 흔히 쓰는 비밀번호 수억 개를 미리 해싱해둔 사전. DB에서 훔친 해시값을 사전에서 검색하면 즉시 원본이 나온다.

**공격 방법 2: Brute Force** — 후보를 하나씩 해싱해서 비교. SHA-256은 초당 수십억 번 계산 가능하여 짧은 비밀번호는 몇 초 만에 뚫린다.

**추가 문제:** 같은 비밀번호를 쓰는 사용자들의 해시값이 동일하여, 한 명을 뚫으면 같은 비밀번호 쓰는 사람이 전부 노출된다.

---

### 1-3. 솔트(Salt) — 같은 비밀번호도 다른 해시로

솔트는 비밀번호마다 붙이는 **랜덤 문자열**이다.

```
홍길동: SHA-256("123456" + "x9f2k1") = a7b3f9e2...
이영희: SHA-256("123456" + "m3p7q8") = 5c1d8e4f...

같은 비밀번호인데 해시가 완전히 다름!
```

솔트는 DB에 함께 저장하며, 해커가 볼 수 있어도 괜찮다. 솔트의 목적은 비밀 유지가 아니라 **레인보우 테이블 무력화**이기 때문이다.

```
솔트 없을 때: 레인보우 테이블 1개로 100만 명 전체 공격 가능
솔트 있을 때: 사용자마다 테이블을 새로 만들어야 함 → 사실상 불가능
```

솔트는 **한 명을 더 안전하게 만드는 게 아니라, 전체를 한꺼번에 뚫는 것을 막는 장치**다. 특정 한 명을 타겟으로 한 brute force는 여전히 가능하며, 이를 막는 것이 bcrypt의 역할이다.

---

### 1-4. bcrypt — 일부러 느리게 만든 해시

핵심 아이디어: **"해싱을 의도적으로 느리게 만들자."** 정상 사용자는 로그인 시 해시를 1번만 계산하니까 0.1초 걸려도 무관하지만, 해커는 수십억 번을 시도해야 하므로 치명적이다.

```
SHA-256: 초당 수십억 번 → 6자리 비밀번호: 몇 초 만에 뚫림
bcrypt:  초당 약 1만 번 → 6자리 비밀번호: 수십 년 이상 소요
→ 속도 차이 약 100만 배
```

**cost factor**로 난이도를 조절할 수 있으며, 값을 올릴수록 해싱 시간이 2배씩 늘어난다.

```
cost = 10 → 2¹⁰ = 1,024 라운드 → 약 0.1초
cost = 12 → 2¹² = 4,096 라운드 → 약 0.4초
cost = 13 → 2¹³ = 8,192 라운드 → 약 0.8초
```

하드웨어 성능이 발전하면 cost를 올리면 된다. 2025년 기준 cost 12~13을 권장한다.

bcrypt 해시값에는 **버전, cost, 솔트, 해시가 모두 내장**되어 있어 별도 솔트 컬럼이 불필요하다.

```
$2b$12$LJ3m4ys3FTXOPsQpj1sTYeRMpp1X1rPyE.DFAbCdE3456789012

$2b    → bcrypt 버전
$12    → cost factor
$LJ3m4ys3FTXOPsQpj1sTYe → 솔트 (22자)
RMpp1X1rPyE.DFAbCdE3456789012 → 해시값 (31자)
```

## 🔍 Part 2. 차세대 해싱 — GPU 공격에 대응하다

### 2-1. bcrypt의 약점: GPU 공격

bcrypt는 CPU를 많이 쓰게 만들지만, GPU는 수천 코어가 각각 다른 후보를 동시에 시도할 수 있다.

```
CPU 1개:         초당 약 50번 시도
GPU (RTX 4090):  초당 약 10만 번 시도 → 약 2,000배 빠름
```

---

### 2-2. scrypt — 메모리까지 많이 쓰게 하자

2009년, Colin Percival이 설계. CPU뿐 아니라 **메모리(RAM)도 대량으로 사용**하게 만든다.

```
bcrypt = "연필만 있으면 풀 수 있는 문제"
  → 5,000명이 연필 하나씩 들고 동시에 풀 수 있음

scrypt = "문제 하나에 책상 하나가 필요한 문제"
  → 교실에 책상이 100개뿐이면 100명만 동시 가능
  → GPU 코어 5,000개여도 메모리 부족으로 동시 실행 제한
```

---

### 2-3. Argon2 — 현재 최강의 비밀번호 해싱

2015년 **Password Hashing Competition(PHC)** 우승 알고리즘. 전 세계 암호학자들이 참가한 공개 경쟁에서 선정되어 신뢰도가 높다.

세 가지를 모두 조절할 수 있다.

```
bcrypt:  시간(CPU 비용)만 조절
scrypt:  시간 + 메모리 조절
Argon2:  시간 + 메모리 + 병렬도 조절
```

Argon2의 세 가지 변형:

```
Argon2d  → GPU 공격에 가장 강함, 사이드 채널 공격에 취약
Argon2i  → 사이드 채널 공격에 강함, GPU 방어력은 약간 낮음
Argon2id → d와 i의 하이브리드 → 비밀번호 해싱에 권장!
```

---

### 2-4. 진화의 전체 그림

```
시대        방식              방어 대상

1990년대   평문 저장          → 방어 없음
2000년대   SHA-256 해싱       → 역산 방어 (but 레인보우 테이블에 취약)
2000년대   솔트 + SHA-256     → 레인보우 테이블 방어 (but brute force에 취약)
2000년대   bcrypt             → CPU brute force 방어 (but GPU 공격에 취약)
2009년    scrypt             → GPU 공격 방어 (메모리 요구)
2015년    Argon2             → CPU + GPU + 사이드 채널 모두 방어
```

### 2-5. 실무 코드

```java
// bcrypt (Java, Spring Security)
String hash = new BCryptPasswordEncoder().encode("mypassword123");

// Argon2 (Java, Spring Security 5.3+)
String hash = new Argon2PasswordEncoder(16, 32, 4, 65536, 3)
    .encode("mypassword123");
```

```python
# bcrypt (Python)
import bcrypt
hash = bcrypt.hashpw(b"mypassword123", bcrypt.gensalt(rounds=12))

# Argon2 (Python)
from argon2 import PasswordHasher
ph = PasswordHasher(time_cost=3, memory_cost=65536, parallelism=4)
hash = ph.hash("mypassword123")
```

### 2-6. 2025년 기준 권장

```
신규 프로젝트     → Argon2id (최신, 가장 안전)
Spring Security  → bcrypt가 기본값, Argon2도 지원
기존 프로젝트     → bcrypt (cost 12 이상)도 충분히 안전
SHA-256 + 솔트만  → bcrypt 또는 Argon2로 마이그레이션 권장
```

## 🔍 Part 3. 해싱 vs 암호화 — Jasypt의 위치

### 3-1. 근본적으로 다른 도구

```
해싱 (bcrypt, Argon2):
  원본 → 해시값 (단방향, 복원 불가)
  용도: 사용자 비밀번호 저장 (일치 여부만 확인)

암호화 (Jasypt, AES):
  원본 → 암호문 → 원본 복원 가능 (양방향)
  용도: 설정 파일의 DB 비밀번호, API 키 (원본이 필요한 데이터)
```

사용자 비밀번호에 Jasypt(암호화)를 쓰면, DB 유출 시 복호화로 원본이 노출될 수 있어 위험하다.

---

### 3-2. Jasypt의 용도와 주의점

Jasypt는 주로 **application.yml의 민감한 설정값**을 암호화하는 데 쓰인다.

```yaml
# Jasypt 적용
spring:
  datasource:
    password: ENC(a8G3kL9mN2pQ5rT8wX1yZ4bD7fH0jK3m)
```

기본 알고리즘(PBEWithMD5AndDES)은 현재 기준으로 취약하므로 반드시 변경해야 한다.

```yaml
# ✅ 강한 알고리즘으로 변경
jasypt:
  encryptor:
    algorithm: PBEWithHMACSHA512AndAES_256
    iv-generator-classname: org.jasypt.iv.RandomIvGenerator
```

근본적 문제는 **암호화 키 관리**다. 키를 설정 파일에 평문으로 두면 암호화한 의미가 없으므로, 최소한 환경 변수로 분리해야 한다.

---

### 3-3. 용도별 정리

```
사용자 비밀번호 (로그인용):
  → 해싱 (bcrypt, Argon2) — 원본 복원 불필요
  → Jasypt 쓰면 안 됨!

설정 파일 민감 정보 (DB 비밀번호, API 키):
  → 암호화 (Jasypt, Vault) — 원본 복원 필요

데이터 전송 보안:
  → 전송 암호화 (TLS/SSL) — 또 다른 영역
```

## 🔍 Part 4. 온프레미스 환경의 대안

### 4-1. 클라우드를 못 쓸 때의 선택지

**옵션 1: HashiCorp Vault 온프레미스 설치** — Vault는 클라우드 서비스가 아니라 직접 서버에 설치하는 소프트웨어다. 오픈소스(Community) 버전으로 무료 사용 가능하며, 완전 폐쇄망에서 운영할 수 있다.

**옵션 2: 환경 변수 + CI/CD 연동** — 가장 현실적인 방법. 비밀 값을 환경 변수로 분리하고 Jenkins/GitLab CI의 Credentials 기능으로 관리한다.

```yaml
spring:
  datasource:
    password: ${DB_PASSWORD}     # 환경 변수 참조
jasypt:
  encryptor:
    password: ${JASYPT_KEY}      # 키도 환경 변수로 분리
```

**옵션 3: Jasypt 유지 + 보완** — 이미 Jasypt를 쓰고 있다면 알고리즘 강화 + 키를 환경 변수로 분리하는 것만으로도 보안 수준이 크게 올라간다.

---

### 4-2. 단계적 개선 전략

```
지금 당장 (1일):
  → Jasypt 알고리즘 강화 (PBEWithHMACSHA512AndAES_256)
  → 암호화 키를 환경 변수로 분리

다음 단계 (1~2주):
  → CI/CD Credentials로 비밀 값 관리
  → 코드·설정 파일에서 비밀 값 완전 제거

장기적 (1~3개월):
  → Vault 온프레미스 도입 검토
  → 비밀 값 자동 갱신, 접근 로그, 권한 관리
```

```
진화 방향:

설정 파일에 평문      → "비밀번호가 노출되네?"
Jasypt로 암호화       → "키 관리가 문제네?"
환경 변수 + CI/CD     → "체계적으로 관리하자"
Vault                → "비밀 값 자체를 설정 파일에서 없애자"
```

## 💡 한 줄 정리

> 비밀번호 보안의 핵심은 **"공격자에게 더 많은 자원을 요구하는 것"** 이다. 솔트는 전체 공격 비용을, bcrypt는 개인 공격 비용을, Argon2는 CPU·메모리·병렬도 모두를 무기로 삼는다. 해싱(단방향)과 암호화(양방향)는 용도가 다르며, 사용자 비밀번호에는 반드시 해싱을 써야 한다.
