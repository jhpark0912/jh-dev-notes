# Day 006. Java HashMap은 내부에서 어떻게 동작할까?

> 🗓️ 2026-03-03  
> 🏷️ Tags: `HashMap` `해시충돌` `레드블랙트리` `ConcurrentHashMap` `volatile` `Java8`

---

## 🤔 질문

HashMap은 어떻게 문자열 키로 값을 순식간에 찾아올까?  
Java 8에서 해시 충돌 해결 방식이 어떻게 바뀌었고, 멀티스레드에서는 어떤 문제가 생길까?

---

## 📌 핵심 요약

HashMap은 키의 해시값을 배열 인덱스로 변환하여 **O(1)** 탐색을 구현한다.  
Java 8은 충돌이 8개 이상이면 **연결 리스트를 레드-블랙 트리로 전환**하여 최악의 경우를 O(n)→O(log n)으로 개선했다.  
멀티스레드 환경에서는 **ConcurrentHashMap**이 버킷 단위 락 + volatile + ForwardingNode로 동시성과 성능을 모두 확보한다.

---

## 🔍 Part 1. 해시 충돌 해결 — Java 7 vs Java 8

### 1-1. Java 7: 연결 리스트

해시 충돌이 나면, 같은 인덱스에 연결 리스트로 줄줄이 매달아두는 방식이었다.

```
배열[3]에 "name", "age", "phone"이 충돌한 경우:

배열[3] → ("name","홍길동") → ("age",25) → ("phone","010-1234")

get("phone") 호출:
  배열[3] 도착 → "name"? 아님 → "age"? 아님 → "phone"? 맞다!
```

한 인덱스에 데이터가 몰리면 사실상 리스트 순차 탐색이 되어 **O(n)** 까지 성능이 떨어진다.

이 취약점을 노린 공격도 있었다. 의도적으로 같은 해시값을 만드는 키를 대량으로 보내 서버를 마비시키는 **HashDoS 공격**이 2011년에 보고되었다.

---

### 1-2. Java 8: 트리화(Treeification)

충돌이 많아지면 연결 리스트를 **레드-블랙 트리**로 전환하는 전략을 도입했다.

```
충돌 ≤ 7개: 연결 리스트 유지
충돌 ≥ 8개: 레드-블랙 트리로 전환 (TREEIFY_THRESHOLD = 8)
충돌 ≤ 6개: 다시 연결 리스트로 복원 (UNTREEIFY_THRESHOLD = 6)
```

```
Java 7 (연결 리스트):
배열[3] → A → B → C → D → E → F → G → H
         하나씩 순회 → O(n)

Java 8 (8개 이상 → 트리 전환):
배열[3] →      D
             /   \
            B     F
           / \   / \
          A   C E   G
                     \
                      H
         트리 탐색 → O(log n)
```

---

### 1-3. 성능 차이

```
충돌 1,000개:   리스트 최대 1,000번 비교 / 트리 최대 10번 → 100배 차이
충돌 100만 개:  리스트 최대 1,000,000번 / 트리 최대 20번 → 50,000배 차이
```

---

### 1-4. 왜 처음부터 트리를 안 쓰나?

트리 노드는 리스트 노드보다 **메모리를 약 2배** 더 사용한다.

```
연결 리스트 노드: hash, key, value, next          → 4개 필드
트리 노드:       hash, key, value, left, right, parent, color → 7개 필드
```

충돌 7개 이하일 때는 리스트 순회도 충분히 빠르므로, 트리의 메모리 오버헤드가 이득보다 크다.

---

### 1-5. 왜 하필 임계값이 8인가?

Java 소스코드 주석에 근거가 명시되어 있다. 해시 함수가 잘 분산된다면, 한 버킷에 노드가 쌓이는 수는 **포아송 분포**를 따른다.

```
한 버킷에 쌓이는 개수별 확률:
0개: 60.65%
1개: 30.33%
2개: 7.58%
3개: 1.26%
4개: 0.16%
5개: 0.01%
6개: 0.001%
7개: 0.00006%
8개: 0.00000006%  ← 여기부터 트리 전환
```

8개 이상이 쌓일 확률은 천만 분의 6으로, 정상 상황에서는 트리 전환이 거의 일어나지 않는다. 발생한다면 해시 함수가 나쁘거나 공격을 받고 있다는 신호이므로 그때만 트리로 방어한다.

---

### 1-6. 트리→리스트 복원이 6인 이유: 진동 방지

```
임계값이 둘 다 7이라면:
  7개 → 트리 전환 → 삭제 → 6개 → 리스트 복원 → 추가 → 7개 → 트리 전환...
  → 계속 왔다 갔다! (thrashing)

실제 설계 (8과 6):
  8개 → 트리 전환 → 삭제 → 7개 → 아직 트리 유지 → 삭제 → 6개 → 리스트 복원
  → 안정적
```

2의 간격을 두어, 한 번 트리가 되면 확실히 줄어들 때만 복원한다.

## 🔍 Part 2. 리사이징 — 배열 확장

### 2-1. 언제 리사이징이 발생하나?

```
기본 배열 크기: 16
기본 로드 팩터: 0.75

리사이징 시점 = 배열 크기 × 로드 팩터 = 16 × 0.75 = 12개 차면 발동
```

로드 팩터 0.75는 **시간(탐색 속도)과 공간(메모리)의 최적 균형점**이다. 해시 테이블 이론에서 0.7~0.8 사이일 때 평균 탐색 횟수가 1.5~2회 수준으로 유지된다.

---

### 2-2. 리사이징은 새 배열로 이사하는 것

기존 배열 안에서 자리를 바꾸는 게 아니라, **새 배열을 할당하고 거기에 옮기는** 방식이다.

```
기존 배열 (크기 16)              새 배열 (크기 32)
┌──────────────┐               ┌──────────────┐
│ [5] → A → B │──── 옮김 ────→│ [5] → A      │
│ [7] → C → D │──── 옮김 ────→│ [7] → C      │
│ ...          │               │ [21] → B     │  (5+16)
└──────────────┘               │ [23] → D     │  (7+16)
     ↑                         └──────────────┘
  옮긴 후 버림                        ↑
                               새로운 내부 배열이 됨
```

읽는 곳(기존)과 쓰는 곳(새 배열)이 분리되어 있어, 위치 충돌이 발생하지 않는다.

---

### 2-3. Java 7의 리사이징 문제

**순서 역전 (head insertion):** 리스트 노드를 앞쪽에 삽입하여 순서가 뒤집혔다. 멀티스레드에서 두 스레드가 동시에 리사이징하면 **순환 참조**가 생겨 무한 루프로 CPU가 멈출 수 있었다.

```
스레드 1: A → B를 옮기는 중
스레드 2: B → A로 뒤집는 중
결과: A → B → A → B → ...  ← 무한 루프!
```

---

### 2-4. Java 8의 리사이징 개선

**개선 1: tail insertion으로 순서 보존** — 순환 참조 가능성을 제거했다.

**개선 2: 비트 연산으로 빠른 재배치** — 배열 크기가 항상 2의 거듭제곱이라는 점을 이용한다.

```
배열 크기 16 → 32 확장 시:

인덱스 결정: hash & (배열크기 - 1)
  16일 때: hash & 01111(2)  → 하위 4비트
  32일 때: hash & 011111(2) → 하위 5비트

차이: 5번째 비트 하나만 확인하면 됨!
  5번째 비트 = 0 → 원래 위치 그대로
  5번째 비트 = 1 → 원래 위치 + 16(이전 배열 크기)
```

나머지 연산(%)은 느리지만, 비트 AND(&)는 한 클럭에 끝나므로 대량 재배치 시 성능 차이가 크다.

---

### 2-5. 리사이징 비용과 최적화

```
리사이징 비용:
  새 배열 메모리 할당 + 전체 노드 재배치 → O(n)
  순간적으로 기존 + 새 배열 = 약 3배 메모리 필요
```

데이터 양을 미리 안다면 초기 용량을 지정하여 리사이징을 최소화할 수 있다.

```java
// ❌ 기본 크기 16 → 데이터 10만 개 넣으면 리사이징 여러 번
Map<String, String> map = new HashMap<>();

// ✅ 예상 크기 지정 → 리사이징 최소화
Map<String, String> map = new HashMap<>(200000);
```

## 🔍 Part 3. 멀티스레드 — HashMap은 왜 위험한가?

### 3-1. HashMap은 싱글 스레드 전용

HashMap에는 동기화 장치가 없다. 리사이징 중 다른 스레드가 접근하면 데이터 유실, 무한 루프, NullPointerException 등이 발생할 수 있다.

---

### 3-2. 동시성 Map의 진화

**1세대: Hashtable** — 모든 메서드에 synchronized. 한 번에 하나의 스레드만 접근 가능. 안전하지만 느리다.

**2세대: Collections.synchronizedMap** — HashMap을 감싸서 synchronized 추가. 본질적으로 Hashtable과 같은 문제.

**3세대: ConcurrentHashMap** — 완전히 다른 접근. 실무에서의 정답.

```
Hashtable / synchronizedMap:
  ┌─────────────────────────────┐
  │ 전체를 하나의 락으로 잠금     │
  │ [0] [1] [2] ... [15]       │
  │ 🔒 하나의 큰 자물쇠          │
  └─────────────────────────────┘
  → 누구 하나 작업하면 전원 대기

ConcurrentHashMap:
  ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐
  │ [0]  │ │ [1]  │ │ [2]  │ │ [3]  │ ...
  │ 🔒   │ │ 🔒   │ │ 🔒   │ │ 🔒   │
  └──────┘ └──────┘ └──────┘ └──────┘
  → 각 버킷마다 독립적인 락
  → 다른 버킷은 동시에 작업 가능
```

---

### 3-3. ConcurrentHashMap의 핵심 특성

**버킷 단위 락:** 같은 버킷일 때만 대기, 다른 버킷은 동시 접근 가능.

**읽기는 락 없음(lock-free):** volatile과 CAS 연산으로 락 없이도 안전한 읽기를 보장. 대부분의 웹 애플리케이션은 읽기가 쓰기보다 훨씬 많으므로 실무에서 큰 성능 차이를 만든다.

**병렬 리사이징:** 배열을 구간별로 나눠서 여러 스레드가 분담하며, 리사이징 중에도 읽기가 가능하다.

---

### 3-4. 리사이징 중 읽기: ForwardingNode

리사이징 중에는 기존 배열과 새 배열이 동시에 존재한다. ConcurrentHashMap은 버킷 이동이 완료되면 기존 배열에 **ForwardingNode(전달 노드)** 를 남긴다.

```
리사이징 과정:

1단계: 배열[2]의 데이터를 새 배열로 이동
2단계: 기존 배열[2]에 ForwardingNode 설치 → "이사 완료" 표식

get() 호출 시:
  기존 배열[2]에 일반 노드 → 여기서 탐색
  기존 배열[2]에 ForwardingNode → 새 배열로 안내 → 새 배열에서 탐색
```

```
시간순:

T1: 배열[0],[1],[2] 이동 완료, FWD 설치
    기존: [0:FWD] [1:FWD] [2:FWD] [3:데이터] [4:데이터] ...

T2: get() 인덱스 1번 → FWD 발견 → 새 배열에서 탐색 → 찾음!
T3: get() 인덱스 3번 → 일반 노드 → 기존 배열에서 탐색 → 찾음!
```

순서도 중요하다. **"새 배열에 데이터 배치 → 기존 배열에 FWD 설치"** 순서가 보장되어, FWD를 따라갔는데 데이터가 아직 없는 상황은 발생하지 않는다.

비유하면, 이사한 집 문에 **"이사했습니다. 새 주소: ○○아파트"** 쪽지를 붙여두는 것이다.

## 🔍 Part 4. volatile — 스레드 간 가시성 보장

### 4-1. 왜 필요한가?

각 CPU 코어에는 캐시가 있어, 변수를 메인 메모리가 아닌 자기 캐시에서 읽는다. 한 스레드가 값을 바꿔도 다른 스레드의 캐시에는 반영되지 않을 수 있다.

```
volatile 없이:

  CPU 코어 1             CPU 코어 2
  ┌──────────┐          ┌──────────┐
  │ 캐시     │          │ 캐시     │
  │ x = 1 ✏️ │          │ x = 0   │ ← 옛날 값!
  └──────────┘          └──────────┘
       메인 메모리: x = 0 (아직 미반영)
```

비유하면, 메인 메모리는 **도서관의 원본 장부**이고, 각 CPU 캐시는 **각자 책상 위에 복사해둔 메모**다. 한 직원이 메모를 고쳐도 다른 직원의 메모에는 반영되지 않는다.

---

### 4-2. volatile의 동작

변수에 `volatile`을 붙이면 **캐시를 무시하고 항상 메인 메모리에서 직접 읽고 쓴다.**

```
volatile 있으면:

  CPU 코어 1             CPU 코어 2
  │ 쓰기 ↓                ↑ 읽기 │
  ─────────────────────────────
  │     메인 메모리: x = 1      │ ← 즉시 반영, 즉시 읽음
  ─────────────────────────────
```

---

### 4-3. 대표적인 사용 예시: 플래그 변수

```java
// ❌ volatile 없으면 — 루프가 끝나지 않을 수 있음
boolean running = true;
// 스레드 1: while(running) { 작업 }
// 스레드 2: running = false  → 스레드 1이 변경을 못 볼 수 있음!

// ✅ volatile 있으면 — 즉시 반영
volatile boolean running = true;
// 스레드 2가 false로 바꾸면, 스레드 1이 다음 루프에서 즉시 감지
```

---

### 4-4. volatile의 한계: 복합 연산은 보장 안 됨

volatile은 **가시성(최신 값 보기)** 만 보장하고, **원자성(읽기-수정-쓰기의 안전성)** 은 보장하지 않는다.

```java
volatile int count = 0;

// 두 스레드가 동시에 count++를 실행하면:
// 스레드 1: 읽기(0) → 더하기(1) → 쓰기(1)
// 스레드 2: 읽기(0) → 더하기(1) → 쓰기(1)
// 결과: 1 (2가 되어야 함!)
```

```
volatile로 충분한 경우:
  단순 읽기/쓰기 (플래그, 상태값)

volatile로 부족한 경우 (다른 도구 필요):
  읽기-수정-쓰기 → AtomicInteger, synchronized 사용
```

---

### 4-5. ConcurrentHashMap에서 volatile의 역할

```java
transient volatile Node<K,V>[] table;  // 내부 배열이 volatile
```

리사이징 스레드가 ForwardingNode를 설치하면 volatile에 의해 메인 메모리에 즉시 반영되고, 읽기 스레드가 해당 버킷을 확인할 때 항상 최신 상태(FWD 노드)를 보게 된다. 이것이 **락 없이도 안전한 읽기**가 가능한 핵심 원리다.

---

## 💡 한 줄 정리

> HashMap은 **해시 → 배열 인덱스 → O(1) 탐색**이라는 단순한 원리 위에, Java 8의 **트리화(충돌 방어)** + **비트 연산 리사이징(성능)** + ConcurrentHashMap의 **버킷 락·volatile·ForwardingNode(동시성)** 이 겹겹이 쌓인 정교한 설계다.
