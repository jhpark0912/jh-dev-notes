# Day 008. DB 인덱스는 왜 빠를까? — B-Tree와 커버링 인덱스

> 🗓️ 2026-03-05  
> 🏷️ Tags: `B-Tree` `인덱스` `커버링인덱스` `레드블랙트리` `복합인덱스` `EXPLAIN`

---

## 🤔 질문

DB에 인덱스를 걸면 왜 빨라질까? 왜 HashMap이 아닌 B-Tree를 쓸까?  
Java는 왜 B-Tree가 아닌 레드-블랙 트리를 택했을까?  
커버링 인덱스는 왜 빠르고, 인덱스 설계 기준은 어떻게 잡아야 할까?

---

## 📌 핵심 요약

B-Tree는 **디스크 페이지에 키를 최대한 채워 I/O 횟수를 줄이는 설계**이고, Java의 레드-블랙 트리는 **메모리에서 노드가 작을수록 CPU 캐시 효율이 좋기 때문**에 선택됐다.  
커버링 인덱스는 **인덱스에 컬럼 값이 실제로 복사되어 있어** 테이블 접근 없이 답을 반환하며,  
인덱스 설계는 **추측이 아닌 슬로우 쿼리 기반**으로 판단해야 한다.

---

## 🔍 Part 1. DB가 B-Tree를 선택한 진짜 이유

### 1-1. 핵심: 디스크 I/O가 병목이다

DB 데이터는 메모리가 아닌 디스크(SSD/HDD)에 있다. 디스크 읽기는 메모리보다 수만~수십만 배 느리다.

```
메모리 읽기:  약 100 나노초
SSD 읽기:    약 100,000 나노초 (메모리의 1,000배)
HDD 읽기:    약 10,000,000 나노초 (메모리의 100,000배)

→ 디스크를 한 번이라도 덜 읽는 게 핵심
```

---

### 1-2. 디스크는 "페이지" 단위로 읽는다

디스크에서 1바이트만 필요해도 최소 4KB~16KB 단위(페이지)로 읽어와야 한다. 어차피 4KB를 읽을 거면, 그 안에 데이터를 최대한 많이 넣는 게 이득이다.

```
이진 트리 (레드-블랙 트리):
  한 노드 = 키 1개
  4KB 읽었는데 키 1개만 확인 → 낭비

B-Tree:
  한 노드 = 키 수백 개
  4KB 읽으면 수백 개를 한 번에 비교 → 효율적
```

비유하면, 레드-블랙 트리는 **한 페이지에 단어 1개만 적힌 사전**이고, B-Tree는 **한 페이지에 단어 500개씩 적힌 사전**이다.

---

### 1-3. 실제 숫자로 보는 차이

```
데이터 1,000만 건, 디스크 페이지 16KB:

이진 트리: 높이 ≈ log₂(10,000,000) ≈ 23 → 디스크 23번 읽기
B-Tree:   높이 ≈ log₅₀₀(10,000,000) ≈ 3 → 디스크 3번 읽기

SSD 기준: 23 × 0.1ms = 2.3ms vs 3 × 0.1ms = 0.3ms
HDD 기준: 23 × 10ms = 230ms vs 3 × 10ms = 30ms
```

시간 복잡도는 둘 다 O(log n)이지만, 로그의 밑이 다르다. 이진 트리는 log₂, B-Tree는 log₅₀₀. 디스크 환경에서 이 차이가 결정적이다.

---

### 1-4. B-Tree의 구조

```
[루트 노드]
┌──────────────────────────────────────────┐
│ 100 │ 200 │ 300 │ 400 │ 500 │ ...│ 900 │  ← 키 수백 개
└─┬────┬────┬────┬────┬────┬─────────┬────┘
  ↓    ↓    ↓    ↓    ↓    ↓         ↓
 <100  100  200  300  400  500      >900
       ~199 ~299 ~399 ~499 ~899

[리프 노드]
  실제 데이터(또는 데이터 위치)가 있음
  리프끼리 연결 리스트로 이어져 있음 ← 범위 검색에 핵심!
```

리프 노드끼리 연결 리스트로 이어져 있어, `WHERE age BETWEEN 20 AND 30`에서 20을 찾은 뒤 리프를 따라가기만 하면 30까지 순서대로 나온다. HashMap은 이게 불가능하다.

## 🔍 Part 2. Java가 레드-블랙 트리를 택한 이유

### 2-1. 메모리에서는 B-Tree의 장점이 사라진다

```
DB (디스크 기반):
  디스크 읽기는 페이지 단위 → 한 번에 많이 담을수록 유리 → B-Tree

Java TreeMap (메모리 기반):
  메모리는 어디든 동일한 속도로 접근 → "한 번에 많이 읽기" 이점 없음
  → B-Tree의 장점이 사라짐
```

---

### 2-2. 메모리에서는 오히려 B-Tree의 단점이 부각

```
B-Tree 노드 (키 수백 개):
  → 노드 내부에서 이진 탐색 약 9번 비교
  → 삽입/삭제 시 키 이동, 노드 분할/병합 오버헤드 큼
  → 노드 크기가 커서 CPU 캐시에 안 맞을 수 있음

레드-블랙 트리 노드 (키 1개):
  → 비교 1번 후 왼쪽/오른쪽 바로 결정
  → 삽입/삭제 시 회전 연산 최대 3번이면 균형 복구
  → 노드가 작아 CPU 캐시 라인(64바이트)에 잘 들어감
```

```
비유:

B-Tree = 대형 마트
  → 먼 곳(디스크)에서 한 번 오면 많이 볼 수 있어 효율적
  → 이미 매장 안(메모리)에 있으면 큰 매대가 필요 없음

레드-블랙 트리 = 편의점
  → 이미 가까이(메모리) 있으니 작고 빠른 게 유리
```

---

### 2-3. 정리

```
                    디스크 환경 (DB)      메모리 환경 (Java)
핵심 병목            디스크 I/O 횟수       CPU 연산 횟수
최적 전략            한 번에 많이 읽기      빠르게 비교하고 이동
유리한 구조          B-Tree               레드-블랙 트리
노드 크기            클수록 좋음           작을수록 좋음
```

## 🔍 Part 3. 커버링 인덱스 — 테이블을 안 읽는 마법

### 3-1. 인덱스는 값을 실제로 갖고 있다

인덱스는 단순한 "위치 안내판"이 아니라, 인덱스에 포함된 **컬럼의 값이 실제로 복사되어 저장**돼 있다.

```
users 테이블 (원본):
┌────┬────────┬─────┬─────────────────┬──────┐
│ id │ name   │ age │ email           │ addr │
├────┼────────┼─────┼─────────────────┼──────┤
│  1 │ 홍길동  │ 25  │ hong@test.com   │ 서울  │
│  2 │ 김철수  │ 30  │ kim@test.com    │ 부산  │
│  3 │ 이영희  │ 25  │ lee@test.com    │ 대구  │
└────┴────────┴─────┴─────────────────┴──────┘

CREATE INDEX idx_name_age ON users(name, age);

인덱스 idx_name_age (정렬된 복사본):
┌────────┬─────┬───────────┐
│ name   │ age │ PK(id)    │ ← 테이블로 돌아갈 주소
├────────┼─────┼───────────┤
│ 김철수  │ 30  │ → id=2   │
│ 이영희  │ 25  │ → id=3   │
│ 홍길동  │ 25  │ → id=1   │
└────────┴─────┴───────────┘

name과 age의 실제 값이 인덱스에 복사되어 있음!
```

즉, 인덱스는 원본 테이블의 **"특정 컬럼만 골라서 정렬해둔 축소판"** 이다.

---

### 3-2. 커버링 인덱스의 동작

쿼리가 필요로 하는 모든 컬럼이 인덱스에 포함되어 있으면, 테이블에 접근하지 않는다.

```sql
-- 커버링 O: name과 age만 필요 → 인덱스에 다 있음!
SELECT name, age FROM users WHERE name = '홍길동';
-- → 인덱스만 읽고 끝 (Using index)

-- 커버링 X: email이 필요 → 인덱스에 없음!
SELECT name, age, email FROM users WHERE name = '홍길동';
-- → 인덱스에서 PK 찾고 → 테이블에 가서 email 읽어옴
```

---

### 3-3. 성능 차이

```
100만 건에서 1,000건 조회:

일반 인덱스:
  인덱스 탐색 (3번 I/O) + 테이블 랜덤 I/O (1,000번) ≈ 1,003번

커버링 인덱스:
  인덱스 탐색 (3번 I/O) + 리프 순차 읽기 (수 번) ≈ 5~10번

→ 100배 이상 차이!
```

---

### 3-4. EXPLAIN으로 확인하는 법

```sql
EXPLAIN SELECT name, age FROM users WHERE name = '홍길동';

Extra 컬럼:
  Using index           ← 커버링 인덱스! 테이블 안 읽음
  Using index condition ← 인덱스 일부 사용, 테이블도 읽음
  NULL                  ← 인덱스 타고 테이블도 읽음
```

---

### 3-5. 실무 활용 예시

**목록 조회 최적화:**

```sql
SELECT title, created_at FROM posts
WHERE board_id = 1 ORDER BY created_at DESC LIMIT 20;

-- 커버링 인덱스
CREATE INDEX idx_board_created_title ON posts(board_id, created_at, title);
-- → 테이블 접근 없이 인덱스만으로 20건 반환
```

**COUNT 최적화:**

```sql
SELECT COUNT(*) FROM orders WHERE status = 'PENDING';

CREATE INDEX idx_status ON orders(status);
-- → 인덱스만 세면 됨, 테이블 접근 불필요
```

---

### 3-6. 비유로 정리

```
원본 테이블 = 도서관의 책 (전체 내용)
인덱스 = 책 뒤의 색인 (키워드 + 페이지 번호)
커버링 인덱스 = 색인에 요약 내용까지 적어둔 것
  → "키워드의 요약이 뭐야?" → 색인만 보면 답 나옴, 책을 안 펼쳐도 됨
  → "키워드의 전체 내용이 뭐야?" → 책을 펼쳐야 함
```

## 🔍 Part 4. 인덱스 설계 기준

### 4-1. 인덱스의 비용

```
INSERT 시: 테이블 1번 쓰기 + 인덱스 개수만큼 추가 쓰기
UPDATE 시: 변경된 컬럼이 포함된 인덱스 전부 갱신
DELETE 시: 테이블 + 모든 인덱스에서 삭제

숨겨진 비용:
  → 디스크 용량 증가 (컬럼 값이 복사됨)
  → 메모리(버퍼 풀) 경쟁 (캐시에 다 안 올라감 → 디스크 읽기 증가)
  → 옵티마이저 혼란 (어떤 인덱스를 쓸지 고르는 시간 증가)
```

---

### 4-2. 읽기/쓰기 비율로 판단

```
읽기 위주 (조회 90%, 쓰기 10%):
  → 인덱스 넉넉하게, 커버링 인덱스 적극 활용
  → 예: 블로그, 뉴스, 상품 카탈로그

쓰기 위주 (조회 30%, 쓰기 70%):
  → 인덱스 최소화, 꼭 필요한 것만
  → 예: 로그 저장, IoT 센서 데이터, 실시간 채팅

균형형 (조회 60%, 쓰기 40%):
  → 자주 실행되는 쿼리 기준으로 선별
  → 예: 일반 웹 서비스, 쇼핑몰
```

---

### 4-3. 슬로우 쿼리 기반으로 판단

추측이 아닌, 실제로 느린 쿼리를 찾아서 필요한 인덱스만 건다.

```
1단계: 슬로우 쿼리 로그 활성화
  → MySQL: slow_query_log = ON, long_query_time = 1

2단계: 자주 실행되고 느린 쿼리 식별
  → "하루 10만 번, 매번 2초" ← 1순위
  → "하루 3번, 5초" ← 후순위

3단계: EXPLAIN으로 분석

4단계: 필요한 인덱스 추가

5단계: 추가 후 성능 변화 확인
```

---

### 4-4. 선택도(Cardinality)로 판단

선택도 = 해당 컬럼의 고유 값 비율. 높을수록 인덱스 효과가 크다.

```
선택도 높음 (효과 큼):
  user_id, email, phone → 거의 모든 값이 고유

선택도 낮음 (효과 적음):
  gender (M/F), is_active (true/false)
  → 인덱스 타도 전체의 50%를 읽어야 함 → 풀 스캔이 나을 수 있음
```

---

### 4-5. 복합 인덱스 설계 원칙

개별 인덱스 여러 개보다 **복합 인덱스 하나**가 효율적인 경우가 많다.

**왼쪽부터 순서대로 사용되는 규칙:**

```sql
CREATE INDEX idx_a_b_c ON table(A, B, C);

✅ WHERE A = 1                      → A 사용
✅ WHERE A = 1 AND B = 2             → A, B 사용
✅ WHERE A = 1 AND B = 2 AND C = 3   → A, B, C 전부 사용
❌ WHERE B = 2                       → A를 건너뜀, 인덱스 못 탐
❌ WHERE C = 3                       → A, B를 건너뜀
```

비유하면, 전화번호부가 "시 → 구 → 동" 순서로 정렬되어 있을 때, "강남구"만으로는 찾을 수 없고 "서울시 강남구"로 시부터 시작해야 하는 것과 같다.

**컬럼 순서 원칙:**

```
1순위: = 조건으로 자주 쓰이는 컬럼 (앞에)
2순위: 범위 조건 (>, <, BETWEEN) (뒤에)
3순위: ORDER BY / GROUP BY 컬럼
4순위: SELECT 조회 컬럼 (커버링 목적)
```

```sql
-- 자주 실행되는 쿼리
SELECT name, created_at FROM users
WHERE status = 'ACTIVE' AND age > 25
ORDER BY created_at DESC;

-- 최적 복합 인덱스
CREATE INDEX idx_optimized ON users(status, age, created_at, name);
--                                   ①=     ②범위  ③정렬       ④커버링
```

---

### 4-6. 실무 가이드라인 요약

```
테이블당 인덱스 수: 일반적으로 3~5개 이내 권장

인덱스를 걸어야 하는 곳:
  ✅ WHERE절에 자주 쓰이는 컬럼
  ✅ JOIN 조건 컬럼 (FK)
  ✅ ORDER BY / GROUP BY 컬럼
  ✅ 선택도가 높은 컬럼

인덱스를 걸지 말아야 하는 곳:
  ❌ 선택도가 낮은 컬럼 (gender, boolean)
  ❌ 거의 조회 안 되는 컬럼
  ❌ 자주 UPDATE되는 컬럼
  ❌ 테이블이 작을 때 (수백 건이면 풀 스캔이 더 빠름)

반드시 해야 할 것:
  → 슬로우 쿼리 로그 기반으로 판단
  → EXPLAIN으로 반드시 확인
  → 사용되지 않는 인덱스는 정기적으로 정리
```

---

## 💡 한 줄 정리

> B-Tree는 **디스크 I/O를 줄이기 위해** 한 노드에 키를 수백 개 담는 설계이고, 커버링 인덱스는 **컬럼 값이 인덱스에 복사되어 있어** 테이블 접근을 없앤다. 인덱스 설계는 추측이 아닌 **슬로우 쿼리 → EXPLAIN → 선택도·복합 인덱스 설계 → 검증** 순서로 해야 한다.
