# Day 004. 컴퓨터는 어떻게 '가'와 'A'를 구분할까?

> 🗓️ 2026-02-27  
> 🏷️ Tags: `문자인코딩` `유니코드` `UTF-8` `ASCII` `EUC-KR`

---

## 🤔 질문

컴퓨터는 0과 1밖에 모르는데, 어떻게 "가"와 "A"를 구분할까?  
UTF-8, UTF-16, EUC-KR은 뭐가 다르고, 왜 글자가 깨지는 걸까?

---

## 📌 핵심 요약

컴퓨터가 문자를 다루는 과정은 **번호 부여(유니코드) → 바이트 변환(UTF-8) → 화면 표시(폰트)** 3단계다.  
UTF-8은 ASCII 호환성 + 공간 효율 + 견고한 설계로 웹의 98%를 장악했으며,  
글자가 깨지는 건 **"쓸 때와 읽을 때 서로 다른 번역 사전을 쓰기 때문"** 이다.

---

## 🔍 Part 1. 문자에 번호를 매기다 — 유니코드

### 1-1. 문자 인코딩이란?

컴퓨터는 숫자만 이해하므로, 문자를 저장하려면 숫자로 바꿔야 한다. 이 과정이 두 단계다.

```
1단계: "가"에 번호를 붙이자     → 유니코드가 담당 (가 = 44032번)
2단계: 44032를 0과 1로 저장하자 → UTF-8/16/32가 담당
```

---

### 1-2. 유니코드 = 전 세계 문자 번호 사전

유니코드는 전 세계 모든 문자에 고유 번호를 부여한 사전이다.

```
사전 내용:
  U+0041 → A
  U+0042 → B
  U+AC00 → 가
  U+AC01 → 각
  U+1F600 → 😀
```

`U+`로 시작하는 번호를 **코드 포인트(Code Point)** 라고 부른다. 문자의 주민등록번호 같은 것이다.

---

### 1-3. 누가 관리하는가?

**유니코드 컨소시엄(Unicode Consortium)** 이라는 비영리 단체가 관리한다. 1991년에 설립되었고, Apple, Google, Microsoft, Meta 등이 회원사로 참여한다.

새 문자(이모지 포함) 추가는 **누구나 제안서를 제출**할 수 있다. "이 문자가 왜 필요한지, 얼마나 쓸 것인지" 근거를 작성하면 위원회가 심사하여 승인하거나 거절한다.

2024년 기준 유니코드 16.0, 약 15만 자 이상이 등록되어 있다.

---

### 1-4. 어디에 저장돼 있는가?

유니코드 테이블은 우리 컴퓨터 안에 이미 들어있다.

```
유니코드 사전이 존재하는 곳:

1. 운영체제 (OS)
   → Windows, macOS, Linux 모두 유니코드 테이블 내장

2. 폰트 파일
   → 각 폰트가 "U+AC00(가)을 이런 모양으로 그린다"를 정의

3. 프로그래밍 언어/런타임
   → Java, Python 등이 문자열 처리를 위해 유니코드 테이블 내장
```

OS 업데이트에 새 이모지가 추가되는 이유가 바로 이것이다. 유니코드 컨소시엄이 번호를 정하면, OS 제작사와 폰트 제작사가 이를 반영하여 배포한다.

---

### 1-5. 문자가 화면에 표시되는 전체 과정

```
"가"를 입력
  ↓
유니코드 테이블에서 "가 = U+AC00" 확인
  ↓
UTF-8로 인코딩 → 0xEA 0xB0 0x80 (3바이트)으로 저장
  ↓
화면 표시 시, 폰트에서 U+AC00의 글자 모양(글리프) 탐색
  ↓
픽셀로 렌더링 → 화면에 "가"가 보임
```

이 과정에서 **유니코드 = 번호 약속**, **UTF-8 = 저장 방식**, **폰트 = 그리는 방식**으로 역할이 분리되어 있다.

---

## 🔍 Part 2. 번호를 저장하는 방법 — UTF-8 vs UTF-16 vs UTF-32

### 2-1. UTF-32: 단순하지만 낭비

모든 문자를 **4바이트(32비트) 고정**으로 저장한다.

```
A   → 00 00 00 41  (4바이트)
가  → 00 00 AC 00  (4바이트)
😀 → 00 01 F6 00  (4바이트)
```

장점은 모든 문자가 4바이트로 같으니 N번째 문자를 바로 찾을 수 있다는 것. 단점은 영어 `A`가 1바이트면 충분한데 4바이트를 써야 해서 **영어 문서의 75%가 쓸데없는 0으로 채워진다**는 것이다.

---

### 2-2. UTF-16: 절충안이지만 애매

대부분의 문자를 **2바이트**, 특수 문자만 **4바이트**로 저장한다.

```
A   → 00 41        (2바이트)
가  → AC 00        (2바이트)
😀 → D8 3D DE 00  (4바이트, 서로게이트 페어 처리)
```

역사적으로 중요한 의미가 있다. 유니코드 초기(1991년)에 "전 세계 문자가 65,536자(2바이트) 안에 들어갈 것"이라 예상했다. 그래서 **Windows NT(1993년)와 Java(1995년)가 내부 문자열을 UTF-16으로 설계**했다. 이후 이모지 등이 추가되어 2바이트를 넘겼지만 설계가 이미 굳은 뒤였다.

이것이 Java에서 이모지 길이가 2로 나오는 이유다.

```java
"A".length()    // → 1
"가".length()   // → 1
"😀".length()  // → 2 (서로게이트 페어!)
```

---

### 2-3. UTF-8: 천재적인 가변 길이 설계

켄 톰프슨(Ken Thompson)과 롭 파이크(Rob Pike)가 1992년에 설계했다. 핵심 아이디어는 **문자 번호 크기에 따라 1~4바이트를 쓰자**는 것이다.

```
범위                    바이트 수    해당 문자
U+0000 ~ U+007F        1바이트     영어, 숫자, ASCII 기호
U+0080 ~ U+07FF        2바이트     유럽 문자 (악센트 등)
U+0800 ~ U+FFFF        3바이트     한글, 한자, 일본어
U+10000 ~ U+10FFFF     4바이트     이모지, 고대 문자
```

실제 바이트:

```
A   → 41              (1바이트 — ASCII와 완전히 동일!)
가  → EA B0 80        (3바이트)
😀 → F0 9F 98 80     (4바이트)
```

비유하면 **물건 크기에 딱 맞는 박스를 쓰는 것**이다. UTF-32가 반지도 냉장고 박스에 넣는 것이라면, UTF-8은 반지는 소형 박스, 가구는 대형 박스를 쓴다.

---

### 2-4. UTF-8이 천재적인 이유 3가지

**첫째, ASCII 완벽 호환**

영어와 숫자는 ASCII 시절(1963년)과 바이트가 완전히 동일하다. 60년 전에 만든 텍스트 파일도 그대로 읽힌다. UTF-16은 `A`가 `00 41`이 되어 ASCII 시스템에서 중간의 `00`을 문자열 끝(null)으로 오인하여 깨진다.

**둘째, 웹에서 압도적으로 효율적**

웹 콘텐츠의 상당 부분은 영어 + HTML 태그 + JSON 키값이다. ASCII 문자가 1바이트로 처리되니 전체 데이터 크기가 작다. 한글이 3바이트인 게 아쉬워 보이지만, 실제로는 HTML 태그와 코드가 차지하는 비중이 높아 전체적으로 UTF-8이 유리하다.

**셋째, 자기 동기화(self-synchronizing)**

각 바이트만 보면 "문자의 시작인지, 중간인지"를 즉시 판단할 수 있다.

```
0xxxxxxx → 1바이트 문자의 시작 (ASCII)
110xxxxx → 2바이트 문자의 시작
1110xxxx → 3바이트 문자의 시작
11110xxx → 4바이트 문자의 시작
10xxxxxx → 중간 바이트 (이어지는 바이트)
```

데이터가 깨져도 `10`으로 시작하지 않는 바이트를 찾으면 다음 문자의 시작점이 된다. 복구가 쉽다.

---

### 2-5. UTF-8은 어떻게 모든 문자를 커버하는가?

각 바이트에는 "역할 표식"이 붙어있어 실제 데이터를 담을 비트 수가 줄어든다.

**1바이트: `0xxxxxxx`**

표식 1비트, 데이터 **7비트** → 2⁷ = **128개**

**2바이트: `110xxxxx 10xxxxxx`**

표식 5비트, 데이터 **11비트** → 2¹¹ = **2,048개**

**3바이트: `1110xxxx 10xxxxxx 10xxxxxx`**

표식 8비트, 데이터 **16비트** → 2¹⁶ = **65,536개**

**4바이트: `11110xxx 10xxxxxx 10xxxxxx 10xxxxxx`**

표식 11비트, 데이터 **21비트** → 2²¹ = **2,097,152개 (약 210만)**

```
총 수용력:           약 2,097,152개 (210만)
유니코드 상한선:      약 1,114,111개 (111만) → 넉넉히 커버
현재 등록 문자:       약 150,000개 (15만)  → 전체의 7%만 사용 중
여유 공간:           약 93% 이상 비어있음
```

수백 년간 새 문자를 추가해도 충분한 여유가 있다. 5~6바이트로 확장하지 않은 이유는, 유니코드 컨소시엄이 상한선을 U+10FFFF(약 111만)로 정했고 4바이트의 21비트면 이를 충분히 포함하기 때문이다.

---

### 2-6. 결과: UTF-8의 승리

2024년 기준 전 세계 웹사이트의 약 **98%** 가 UTF-8을 사용한다.

```
인코딩 사용 비율 (웹 기준):

UTF-8   ████████████████████████████████████████ 98%
기타     ██ 2%
```

## 🔍 Part 3. 왜 아직도 글자가 깨지는가?

### 3-1. UTF-8 이전의 세계: 나라마다 다른 번역 사전

UTF-8(1992년) 이전에 각 나라는 이미 자국어 인코딩을 만들어 수십 년간 사용하고 있었다.

```
1963년  ASCII 탄생 (영어만, 128자)
1987년  EUC-KR 등장 (한국)
1990년  Shift_JIS 정착 (일본)
1992년  UTF-8 설계
2000년대  웹에서 UTF-8 확산 시작
```

---

### 3-2. EUC-KR의 동작 방식

EUC-KR은 한글 2,350자를 2바이트에 매핑한 인코딩이다.

```
1바이트 영역 (0x00~0x7F): ASCII 그대로 사용
  A = 0x41

2바이트 영역 (0xA1~0xFE): 한글 배치
  가 = 0xB0 0xA1
  나 = 0xB3 0xAA
```

첫 바이트가 0x80 이상이면 다음 바이트와 합쳐서 한글로 판단하는 방식이다.

---

### 3-3. 글자가 깨지는 원리

**"쓸 때의 약속"과 "읽을 때의 약속"이 다르기 때문**이다.

```
저장할 때: "가"를 EUC-KR로 저장 → 0xB0 0xA1로 기록됨
읽을 때:  UTF-8로 해석 → 0xB0 0xA1은 UTF-8 규칙에 맞지 않음
          → "???" 깨진 문자 출력
```

한국어로 쓴 편지를 받은 사람이 **일본어 사전을 펴고 해석**하는 것과 같다. 같은 글자를 보고 전혀 다른 뜻으로 읽으니 말이 안 되는 것이다.

---

### 3-4. 왜 아직도 EUC-KR이 남아있나?

**레거시 시스템** — 1990~2000년대에 만들어진 한국 정부·은행·기업 시스템이 EUC-KR 기반이다. 수십 년치 데이터를 전부 UTF-8로 변환하는 데는 막대한 비용과 리스크가 따른다.

**윈도우의 영향** — 한국 Windows는 오랫동안 기본 인코딩이 CP949(EUC-KR 확장)였다. Windows 10부터 UTF-8 기본 전환이 시작됐지만 아직 완전하지 않다.

**웹 초기 한국 사이트** — 2000년대 초반 한국 웹사이트 대부분이 EUC-KR로 제작됐다.

```html
<!-- 2000년대 한국 웹사이트 -->
<meta http-equiv="Content-Type" content="text/html; charset=EUC-KR">

<!-- 현재 표준 -->
<meta charset="UTF-8">
```

---

### 3-5. 실무에서 깨지는 대표적 상황

```
상황 1: 윈도우 메모장 CSV → Mac에서 열기
  → 윈도우(CP949)로 저장, Mac(UTF-8)으로 읽음 → 한글 깨짐

상황 2: 오래된 DB → 새 시스템으로 이관
  → DB가 EUC-KR, 새 시스템이 UTF-8 → 변환 누락 시 깨짐

상황 3: 이메일 제목/본문
  → 발신 서버(EUC-KR)와 수신 클라이언트(UTF-8) 불일치 → 깨짐
```

해결 방향은 하나다. **새로 만드는 건 무조건 UTF-8, 기존 시스템은 점진적으로 전환.**

```
인코딩 깨짐의 구조:

저장: "가" → [인코딩 A 사전] → 바이트 기록
읽기: 바이트 → [인코딩 B 사전] → "???"

해결: A와 B를 같은 인코딩(UTF-8)으로 맞추면 끝
```

---

## 💡 한 줄 정리

> 문자 인코딩은 **"숫자 ↔ 문자" 번역 사전**이다. 유니코드가 전 세계 통일 사전을 만들었고, UTF-8이 그 사전을 가장 효율적으로 저장하는 방법이 되었다. 글자가 깨지는 건 아직 옛 사전이 남아있기 때문이다.
