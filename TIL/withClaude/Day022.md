# Day 022. 시간은 왜 이렇게 어려운가

> 2026-04-07 #시간 #타임존 #java.time #윤초 #API설계

## 🤔 질문
왜 시간을 다루는 건 모든 언어에서 그렇게 어려운 일이 되었을까? 그리고 Java는 왜 제대로 된 시간 API를 만드는 데 18년이나 걸렸을까?

## 📌 핵심 요약
- 시간은 천문학(지구 자전)과 정치(타임존)와 역사(윤초)가 뒤엉킨 카오스라서, 단순한 코딩 문제가 아니다.
- Java의 `Date`/`Calendar`는 C의 실수를 수입한 결과물이고, `java.time`은 20년 만에 처음부터 다시 설계해 청산한 명작이다.
- 거의 모든 언어가 시간 API에서 같은 실수를 반복했고, 결국 **불변·타입 분리·월 1-indexed**로 수렴하고 있다.

## 🔍 상세 설명

### Part 1. 시간이 어려운 진짜 이유

대부분 "1분은 60초"라고 믿지만, 윤초가 들어가는 날엔 1분이 61초가 되고 서머타임 전환일엔 하루가 23시간 또는 25시간이 된다. 시간은 본질적으로 균일하지 않다.

시간에는 두 명의 심판이 존재한다.

**지구 자전 기준 (UT1)**
- 태양이 머리 위에 오면 정오라는 식의, 인류가 수천 년간 써온 기준
- 문제: 지구 자전 속도가 달의 조석력, 지진, 빙하 융해 등으로 미세하게 흔들림
- 비유: **손으로 돌리는 레코드판** — 속도가 일정하지 않다

**원자시계 기준 (TAI)**
- 세슘-133 원자의 9,192,631,770번 진동 = 1초
- 3000만 년에 1초 오차의 정밀도
- 비유: **디지털 메트로놈** — 절대 흔들리지 않는다

이 두 심판이 어긋나는 것을 보정한 게 **UTC**(협정 세계시)다. 기본은 원자시계를 따르되, 지구 자전과 0.9초 이상 벌어지면 윤초를 삽입해 맞춘다.

```
TAI (원자시계):  |----|----|----|----|----|---->  완벽히 균일
UT1 (지구자전):  |----|----|---|----|-----|--->  미세하게 불규칙
UTC (타협안):    |----|----|----|----|----!--->  윤초로 보정 (!)
```

### Part 2. 윤초와 타임존 — 카오스의 실체

**윤초가 일으킨 실제 사고**
- 2012년: 윤초 삽입 직후 Reddit, LinkedIn, Mozilla 등 줄줄이 다운. Linux 커널 hrtimer가 윤초를 처리 못해 CPU 100%로 폭주.
- 2017년: Cloudflare DNS가 음수 시간차를 계산해버려 장애.

```java
// 윤초가 위험한 이유 — 단순한 시간차 계산
long elapsed = endTime - startTime;
// 윤초 삽입 시 시계가 뒤로 갈 수 있음 → elapsed가 음수!
if (elapsed < 0) {
    // 💥 아무도 예상 못 한 상황
}
```

Google은 이를 피하려고 **Leap Smear**라는 꼼수를 쓴다. 윤초 1초를 한 번에 넣지 않고, 24시간 동안 매 초를 ~11.6μs씩 늘려 슬쩍 흡수해버리는 방식이다.

**타임존은 과학이 아니라 정치다**
- 네팔: UTC+5:45 — 인도(+5:30)와 다르다는 것을 강조하려고 일부러 15분 차이
- 북한: 2015년 "일제가 우리 시간을 빼앗았다"며 +8:30으로 변경 → 2018년 남북정상회담 후 +9로 복귀
- 중국: 5개 타임존 분량의 땅인데 정치적으로 UTC+8 하나만 사용. 신장에서는 오전 10시에 해가 뜬다.
- 사모아: 2011년 12월 29일 → 12월 31일로 하루를 통째로 건너뜀(날짜변경선 반대편으로 이동)

**서머타임의 함정**
- 봄(Spring Forward): 2:00 AM → 3:00 AM. 2시 30분은 존재하지 않는 시간
- 가을(Fall Back): 2:00 AM → 1:00 AM. 1시 30분이 두 번 존재

```java
// 그래서 cron job은 UTC 기준으로 돌려야 한다
// ❌ 0 30 2 * * *     (로컬 시간 2:30 → 봄에는 실행 안 됨, 가을에는 두 번 실행)
// ✅ 0 30 17 * * *    (UTC 기준)
```

### Part 3. Java 시간 API의 3세대 흑역사

**1세대: `java.util.Date` (1996) — 거의 모든 게 잘못됨**

```java
// 월이 0부터, 연도는 1900 기준
Date date = new Date(126, 2, 15);  // 2026년 3월 15일 ?!
```

C 언어의 `struct tm`을 그대로 베낀 결과다. C에서 `tm_year`는 1900 기준, `tm_mon`은 0부터 시작한다.

문제점:
- 월 0-indexed, 연도 1900 기준
- Mutable — 누구나 수정 가능, 방어적 복사 필수
- 이름은 Date인데 시·분·초·밀리초까지 다 들어있어 순수 날짜 표현 불가
- 타임존 지원 없음

JDK 1.1에서 메서드 대부분이 출시 2년 만에 Deprecated 처리되었다.

**2세대: `java.util.Calendar` (1998) — 고치려다 더 꼬임**

```java
Calendar cal = Calendar.getInstance();
cal.set(2026, 1, 30);  // 2월 30일?!
// 에러 안 남. 자동으로 3월 2일로 변환 (lenient mode)
```

호환성 때문에 0-indexed 월을 그대로 유지했고, 잘못된 값을 조용히 보정해버리는 lenient 모드는 버그를 추적 불가능하게 만들었다. 비유하면 **낡은 집을 리모델링하면서 잘못된 기초는 그대로 두고 벽지만 바꾼 격**이다.

**3세대: `java.time` (2014) — 처음부터 다시**

Joda-Time의 저자 Stephen Colebourne이 직접 JSR-310 스펙 리드를 맡아 백지에서 재설계한 결과물.

```java
LocalDate date = LocalDate.of(2026, 4, 6);          // 날짜만
LocalTime time = LocalTime.of(14, 30);               // 시간만
LocalDateTime dt = LocalDateTime.of(date, time);     // 둘 다, 타임존 없음
ZonedDateTime zoned = ZonedDateTime.of(dt, ZoneId.of("Asia/Seoul"));  // + 타임존

// 불변
LocalDate today = LocalDate.of(2026, 4, 6);
LocalDate nextWeek = today.plusDays(7);  // today는 그대로

// 잘못된 값은 즉시 예외
LocalDate.of(2026, 2, 30);  // DateTimeException
```

핵심 원칙: **불변, 명확한 타입 분리, 월 1-indexed, fail-fast.**

```java
// 서울 시간 → 뉴욕 시간 변환
ZonedDateTime seoul = ZonedDateTime.of(2026, 4, 6, 14, 0, 0, 0, ZoneId.of("Asia/Seoul"));
ZonedDateTime ny = seoul.withZoneSameInstant(ZoneId.of("America/New_York"));
// "같은 순간을, 다른 타임존으로" — 메서드명만으로 의도가 보인다
```

### Part 4. 왜 7년이나 걸렸나 — JSR-310의 험난한 여정

JSR-310은 2007년 제출되어 2014년 Java 8에야 포함되었다. 단순히 설계가 복잡해서가 아니라, 세 가지가 겹친 결과다.

```
JSR-310 타임라인 vs Java 세계의 격변
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
2007  JSR-310 제출. Sun이 운영
2008  금융위기 → Sun 재정 악화
2009  Oracle이 Sun 인수 발표
2010  인수 완료. Java의 주인이 바뀜
2011  Java 7 출시 (3년 지연). JSR-310은 탈락
2012  JSR-310 재설계 시작
2014  Java 8 출시. 드디어 포함
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**① 외부 환경 (~3년)**
Sun의 재정난으로 Java 7 자체가 3년 지연되며 JSR-310이 잘려나갔다. Oracle 인수 후에는 거버넌스가 혼란에 빠졌고, Apache 재단이 JCP 이사회에서 탈퇴할 정도로 긴장이 컸다.

**② 기술적 재설계 (~2년)**
초기에는 Joda-Time을 표준화하는 방향이었지만, 진행하면서 Joda-Time 자체의 한계(null 관대 처리, Chronology 시스템의 과도한 복잡성, 너무 많은 클래스)도 보였다. Colebourne은 결국 백지에서 재설계를 결정했다.

| Joda-Time의 교훈 | java.time의 결정 |
|---|---|
| null 관대 처리 | null 즉시 거부 (fail-fast) |
| Chronology 복잡 | ISO 기본, 나머지 별도 패키지 |
| 클래스 너무 많음 | 핵심 5개로 압축 |
| Interval 내장 | 의도적으로 제외 |

**③ 우선순위 경쟁 (~2년)**
Java 8의 최우선 과제는 람다(JSR-335)였다. 람다는 언어 스펙·컴파일러·라이브러리 전체를 손대야 했고, 핵심 팀 리소스 대부분이 여기에 투입됐다. java.time은 람다가 확정된 후에야 최종 다듬기가 가능했다 — 두 JSR은 사실상 의존 관계였다.

결과적으로 이 7년의 기다림 덕분에 java.time은 **Joda-Time의 복잡성을 청산하고 람다와 잘 어우러지는 API**가 될 수 있었다. 만약 2009년에 서둘러 나왔다면 또 다른 흑역사가 됐을지도 모른다.

### Part 5. 다른 언어들은 어떻게 다루는가

**JavaScript — Java의 실수를 복붙한 비극**

Brendan Eich가 1995년 JS를 10일 만에 만들 때 Java 1.0의 `Date`를 거의 그대로 복사했다. 월 0-indexed, mutable, 타임존 미지원이라는 Java의 모든 실수가 유전됐다.

```javascript
new Date(2026, 2, 15)  // 3월 15일. 2 = 3월

// 파싱이 브라우저마다 다른 지옥
new Date("2026-04-06")  // Safari는 UTC, 옛 Chrome은 로컬 — 같은 코드 다른 결과
```

Moment.js가 사실상 표준이었으나 mutable + 300KB의 무게로 2020년 공식 은퇴 선언. 현재는 Day.js(2KB), date-fns(함수형), Luxon(Moment 팀의 후속작)이 대안이고, **TC39 Temporal**이 JS의 java.time이 될 예정이다.

```javascript
// Temporal — JS의 미래
Temporal.PlainDate.from('2026-04-06')           // ≈ LocalDate
Temporal.ZonedDateTime.from('2026-04-06T14:30[Asia/Seoul]')  // ≈ ZonedDateTime
// 불변, 월 1-indexed, java.time과 거의 1:1 대응
```

**Python — 출발은 좋았으나 타임존이 지옥**

처음부터 월 1-indexed, 불변, 타입 분리(`date`/`time`/`datetime`)를 갖췄다. Java가 18년 걸린 것을 Python은 처음부터 한 셈이다.

문제는 **naive vs aware** 구분과 표준 라이브러리에 타임존 DB가 없었다는 점.

```python
# Python 3.8 이전에는 표준에 UTC만 있었음
# pytz의 유명한 함정
tz = pytz.timezone("Asia/Seoul")
wrong = datetime(2026, 4, 6, 14, 30, tzinfo=tz)  # ❌ LMT(+8:28) 적용 버그
correct = tz.localize(datetime(2026, 4, 6, 14, 30))  # ✅ localize() 필수

# Python 3.9+ 드디어 zoneinfo 표준 모듈
from zoneinfo import ZoneInfo
dt = datetime(2026, 4, 6, 14, 30, tzinfo=ZoneInfo("Asia/Seoul"))
```

pytz의 `localize()` 함정은 **전원 버튼이 뒤에 숨겨진 가전제품**과 같다. 동작은 하는데 올바르게 켜는 방법이 직관적이지 않다.

**Go — 이상한 천재의 포맷팅**

Rob Pike가 설계한 Go의 `time` 패키지는 기본기는 탄탄하지만 포맷팅이 유별나다.

```go
// 다른 언어: 패턴 문자
// Java   "yyyy-MM-dd HH:mm:ss"
// Python "%Y-%m-%d %H:%M:%S"

// Go: 기준 시각의 각 부분을 직접 씀
t.Format("2006-01-02 15:04:05")
// 2006=년, 01=월, 02=일, 15=시, 04=분, 05=초
// → "1월 2일 3시 4분 5초 2006년"이라는 기준 시각
```

"패턴 약어를 외울 필요가 없다"는 철학이지만, 호불호가 극단적으로 갈리는 설계다.

**Rust — 타입 시스템으로 컴파일 타임에 차단**

표준 라이브러리는 `Instant`/`Duration`만, 나머지는 `chrono` 크레이트에 맡겼다. 핵심은 타입 시스템.

```rust
let utc_time: DateTime<Utc> = Utc::now();
let local_time: DateTime<Local> = Local::now();
// utc_time == local_time  // 컴파일 에러! 타입이 다름
let converted: DateTime<Utc> = local_time.with_timezone(&Utc);  // 명시적 변환 필요
```

Java는 런타임에 알아서 변환하지만, Rust는 **타입이 다르면 컴파일 자체가 안 된다**. 시간 버그를 컴파일 타임에 잡겠다는 철학.

**전체 비교**

| 언어 | 월 시작 | 불변성 | 타임존 내장 | 설계 시기 |
|---|---|---|---|---|
| Java Date | 0 | Mutable | ✗ | 1996 (실패) |
| Java Time | 1 | Immutable | ✓ | 2014 (재설계) |
| JS Date | 0 | Mutable | ✗ | 1995 (복붙) |
| JS Temporal | 1 | Immutable | ✓ | ~2025 (진행 중) |
| Python | 1 | Immutable | △ (3.9+) | 2003 |
| Go | 1 | Immutable | ✓ | 2009 |
| Rust chrono | 1 | Immutable | ✓ (타입 강제) | 2014 |

1990년대 언어들(Java, JS)은 C의 실수를 그대로 물려받았고, 2000년대 이후 언어들은 선배의 실수를 보고 배웠다. 결국 모든 언어가 **불변·타입 분리·월 1-indexed**로 수렴하고 있다.

## 💡 한 줄 정리
> 시간 API의 역사는 "C의 실수를 물려받은 언어들이 20년에 걸쳐 같은 교훈에 도달하는" 과정이며, 그 교훈은 항상 동일하다 — 불변, 타입 분리, 그리고 fail-fast.
