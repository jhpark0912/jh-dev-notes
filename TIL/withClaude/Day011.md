# Day 011. OAuth로 "구글로 로그인"은 어떻게 동작할까?

> 🗓️ 2026-03-11  
> 🏷️ Tags: `OAuth` `JWT` `RBAC` `Zanzibar` `IdP` `RelationTuple` `권한관리`

---

## 🤔 질문

OAuth는 왜 탄생했고, "구글로 로그인"은 내부에서 어떤 흐름으로 동작할까?  
JWT로 권한을 관리하는 방식은 무엇이고, 대규모 서비스에서는 어떻게 확장할까?  
Google은 수조 개의 권한 관계를 어떻게 밀리초 안에 판단할까?

---

## 📌 핵심 요약

OAuth는 **"비밀번호를 넘기지 않고도 제3자에게 필요한 권한만 위임"** 하기 위해 탄생했다.  
JWT는 토큰 자체에 권한 정보를 담아 **DB 조회 없이 인증**하는 stateless 방식이며, 실무에서는 **짧은 Access Token + Refresh Token** 조합으로 보안과 성능의 균형을 맞춘다.  
대규모 서비스는 인증을 **IdP로 분리**하고, Google Zanzibar는 **Relation Tuple과 그래프 탐색**으로 수조 개의 관계를 밀리초 안에 판단한다.

---

## 🔍 Part 1. OAuth는 왜 탄생했는가?

### 1-1. OAuth 이전의 세계: 비밀번호를 직접 넘겨야 했다

2006년쯤, 웹 서비스들이 서로 연동되기 시작했다. "트위터 친구 목록을 다른 앱에서 보고 싶다"면, 그 앱에 **트위터 아이디와 비밀번호를 직접 넘겨야** 했다.

```
사용자: "이 앱에서 내 트위터 친구 목록을 보고 싶어"
앱: "그러면 트위터 아이디랑 비밀번호 줘"
사용자: hong / mypassword123
앱: 이 비밀번호로 트위터에 직접 로그인 → 친구 목록 가져옴

→ 제3자 앱이 내 비밀번호를 갖고 있음!
```

---

### 1-2. 이 방식의 위험성

```
1. 과도한 권한: 친구 목록만 보겠다고 했지만, 비밀번호가 있으니
   → 글 삭제, 비밀번호 변경, 계정 탈취까지 가능

2. 취소 불가: 이 앱의 접근을 끊으려면 비밀번호를 바꾸는 수밖에 없음
   → 다른 정상적인 서비스도 전부 끊김

3. 신뢰 문제: 앱이 비밀번호를 평문 저장할 수도 있음
   → 앱이 해킹당하면 내 비밀번호 유출
```

비유하면, 호텔 룸서비스를 시키려고 **방 마스터키를 직원에게 통째로 넘기는** 것과 같다. 음식만 놓고 나오길 기대하지만, 서랍을 뒤질 수도, 열쇠를 복사할 수도 있다.

실제로 2007년 Twitter 연동 앱들이 해킹당하면서 계정 대량 탈취, 스팸 DM, 피싱 트윗이 퍼지는 사고가 발생했다.

---

### 1-3. OAuth의 탄생

Twitter의 블레인 쿡(Blaine Cook)과 Google의 크리스 메시나(Chris Messina) 등이 모여 해결책을 만들었다.

핵심 아이디어: 비밀번호 대신 **제한된 권한의 토큰**을 발급한다.

```
OAuth 이전: 앱에게 마스터키(비밀번호) 넘김 → 모든 권한
OAuth 이후: 앱에게 카드키(토큰) 발급 → 특정 권한만, 특정 시간만
```

비유하면, 마스터키 대신 **"301호에만, 오늘만, 음식 배달 목적으로"** 라고 적힌 임시 카드키를 발급하는 것이다. 취소하고 싶으면 카드키만 비활성화하면 되고, 마스터키(비밀번호)는 안전하게 갖고 있다.

```
2007년  OAuth 1.0 (서명 기반, 구현 복잡)
2009년  OAuth 1.0a (보안 취약점 수정)
2012년  OAuth 2.0 (현재 표준, 토큰 기반, HTTPS 필수)
```

## 🔍 Part 2. "구글로 로그인" 동작 흐름

### 2-1. 참여자와 흐름

```
참여자 3명:
  사용자 (브라우저)
  우리 서비스 (Client)
  Google (Authorization Server + Resource Server)
```

```
1. 사용자가 "구글로 로그인" 클릭
2. 우리 서비스 → Google 로그인 페이지로 리다이렉트
   (client_id, redirect_uri, scope=email+profile 포함)
3. 사용자가 Google에서 로그인 + 권한 동의
4. Google → 우리 서비스에 Authorization Code 전달
5. 우리 서비스 → code + client_secret으로 Google에 Access Token 요청
6. Google → Access Token 반환
7. 우리 서비스 → Access Token으로 사용자 정보(email, name) 조회
8. 우리 서비스 → 자체 JWT 발급 → 사용자에게 전달
```

핵심: Authorization Code는 브라우저를 거쳐 전달되지만(노출 가능), code만으로는 아무것도 못 하고 **code + client_secret(서버에만 있음)이 합쳐져야** token이 발급된다.

## 🔍 Part 3. JWT와 권한 관리

### 3-1. JWT의 핵심: 서버가 저장하지 않는다 (Stateless)

JWT는 서버에 토큰을 저장하지 않는다. 토큰 자체에 정보가 들어있다.

```
세션 방식: 요청 → 서버 DB에서 세션 조회 → 권한 확인 (매번 DB 접근)
JWT 방식:  요청 → JWT 서명 검증 → Payload에서 바로 권한 확인 (DB 불필요)
```

```
JWT Payload 예시:
{
  "user_id": "user_001",
  "role": "ADMIN",
  "iat": 1709881200,
  "exp": 1709884800
}
```

비유하면, 세션은 **매번 데스크에서 예매 확인**하는 것이고, JWT는 **입장권에 좌석·영화·시간이 다 적혀있어** 문 앞에서 입장권만 확인하면 끝이다.

---

### 3-2. 권한 관리 방식: RBAC

**RBAC(Role-Based Access Control)** 은 "사용자 → 역할 → 권한" 구조로, 역할을 부여하면 해당 권한이 자동으로 따라오는 방식이다.

```
역할 정의:
  ADMIN:   사용자 관리, 글 CRUD, 시스템 설정
  MANAGER: 글 CRUD, 사용자 조회
  USER:    글 작성/조회

JWT에 role만 넣음:
  { "user_id": "001", "role": "MANAGER" }

서버에서:
  @PreAuthorize("hasRole('ADMIN')")      → ADMIN만
  @PreAuthorize("hasAnyRole('ADMIN', 'MANAGER')") → 둘 다 가능
```

장점은 직관적이고 구현이 간단하다는 것. 한계는 "이 특정 문서에 대해서만 편집 가능" 같은 리소스별 세밀한 제어가 어렵고, 역할 조합이 늘어나면 역할이 폭발적으로 증가한다는 것이다.

---

### 3-3. JWT의 근본적 문제와 해결

JWT가 stateless이기 때문에, 이미 발급된 토큰을 **즉시 무효화할 방법이 없다.**

```
상황: ADMIN 직원이 퇴사
  세션 방식: 서버에서 세션 삭제 → 즉시 차단
  JWT 방식: 토큰에 "role: ADMIN"이 적혀있고, 무효화할 방법 없음
            → 만료될 때까지 ADMIN 접근 가능!
```

**해결법: 짧은 Access Token + Refresh Token**

```
Access Token:  만료 15분 (짧게, stateless)
Refresh Token: 만료 7일 (길게, 서버 DB에 저장)

정상 흐름: Access Token으로 인증 → 만료 시 Refresh Token으로 재발급
퇴사자 차단: DB에서 Refresh Token 삭제 → 최대 15분 후 완전 차단
즉시 차단: Redis 블랙리스트에 토큰 등록 → 즉시 차단
```

## 🔍 Part 4. 대규모 서비스의 권한 관리

### 4-1. 인증을 별도 서비스로 분리 — IdP

대규모 서비스(수억 사용자, 앱 수십~수백 개)는 인증/권한을 **독립된 전문 서비스(IdP, Identity Provider)** 로 분리한다.

```
작은 서비스: 앱 하나가 인증+권한+비즈니스 전부 담당
대규모 서비스:
  Gmail, YouTube, Drive... → 각각 비즈니스만 담당
  IdP (인증 전문 서비스) → 로그인, 토큰, 권한을 전담
```

비유하면, 작은 회사는 각 부서에서 방문자를 확인하지만, 대기업은 1층 보안 데스크에서 통합 확인 후 출입증을 발급하는 것이다.

---

### 4-2. 캐시 전략 — DB 조회를 줄인다

대규모에서 DB 조회를 **없애는 건 불가능**하고, **횟수를 극적으로 줄이는** 것이 핵심이다.

```
다단계 캐시 구조:

1차: 로컬 캐시 (서버 메모리, Caffeine)  → 히트 80%, 0.1ms
2차: 분산 캐시 (Redis)                → 히트 19%, 1ms
3차: DB (PostgreSQL)                  → 히트 1%, 10ms

→ 99%의 요청이 DB까지 가지 않음
```

캐시의 한계와 대응:

```
캐시 불일치: 권한 변경이 캐시에 즉시 반영 안 됨
  → TTL(자동 만료) 또는 이벤트 기반 캐시 무효화로 대응

캐시 장애: Redis가 죽으면 DB에 부하 폭주
  → 로컬 캐시 + 분산 캐시 2단계 구조로 방어

캐시 메모리: 전체 사용자를 캐싱할 수 없음
  → LRU로 활성 사용자만 캐싱 (히트율 95%+ 유지)
```

---

### 4-3. Google Zanzibar — 관계 기반 권한 시스템

2019년 USENIX Security에서 발표된 논문. Google 전 서비스(Gmail, Drive, YouTube 등)의 권한을 통합 관리하는 시스템이다.

**Relation Tuple** — 모든 권한을 "누가, 무엇에, 어떤 관계"로 표현한다.

```
형식: object#relation@user

doc:보고서#owner@user:홍길동       → "홍길동은 보고서의 소유자"
folder:프로젝트#viewer@group:개발팀#member → "개발팀 멤버는 프로젝트 폴더의 뷰어"
doc:설계서#parent@folder:프로젝트   → "설계서는 프로젝트 폴더에 속함"
```

핵심은 **튜플 연쇄에 의한 자동 추론**이다.

```
저장된 튜플:
  ① 박민수는 개발팀의 멤버다
  ② 개발팀 멤버는 프로젝트 폴더의 뷰어다
  ③ 설계서는 프로젝트 폴더에 속한다
  ④ 규칙: 폴더의 뷰어는 폴더 안 문서도 볼 수 있다

질문: "박민수가 설계서를 볼 수 있는가?"
추론: ① → ② → ④ + ③ → Yes!

→ 직접 "박민수 → 설계서" 관계를 저장하지 않았는데 자동 추론
→ 멤버 50명, 문서 1,000개여도 튜플 몇 개로 관리 가능
```

RBAC이면 50명 × 1,000개 = 50,000개의 매핑이 필요하지만, Relation Tuple은 최소한의 튜플 + 규칙으로 자동 추론된다.

**Zookie** — 분산 환경에서 캐시 불일치를 해결하는 독창적 메커니즘이다.

```
권한 변경 시 "Zookie(타임스탬프 토큰)" 반환
→ 이후 Check 요청에 Zookie 포함
→ "이 시점 이후의 데이터로 판단해" 보장
→ 오래된 캐시 노드는 자동으로 건너뜀
```

**실제 성능:**

```
초당 처리량: 수백만 건
응답 시간: p50=5ms, p99=50ms
저장된 관계: 수조 개
가용성: 99.999%
```

---

### 4-4. RBAC vs Relation Tuple 비교

```
              RBAC                    Relation Tuple
권한 표현     역할 기반                 관계 기반
리소스별 권한  표현 어려움               자연스러움
그룹 권한     별도 구현 필요             튜플 연쇄로 자동
상속/추론     직접 구현해야 함           규칙 정의하면 자동
적합한 규모   소~중규모                  대규모, 복잡한 권한 구조
```

대부분의 서비스는 RBAC로 충분하고, Google Drive 같은 복잡한 공유 체계가 필요할 때 관계 기반으로 진화한다.

---

### 4-5. 규모별 권장 구조

```
스타트업 (사용자 수천 명):
  → JWT + DB RBAC 테이블 → 충분

성장기 (사용자 수십만, 앱 여러 개):
  → Keycloak(오픈소스 IdP) 도입 → 인증 서비스 분리

대규모 (사용자 수백만+, 앱 수십 개+):
  → 전용 IdP + 분산 캐시 + 관계 기반 권한
  → SpiceDB, Ory Keto, OpenFGA 등 Zanzibar 영감 오픈소스 활용
```

---

## 💡 한 줄 정리

> OAuth는 **비밀번호 없이 권한을 위임**하고, JWT는 **토큰에 정보를 담아 DB 없이 인증**하며, 대규모에서는 **IdP + 다단계 캐시 + 관계 기반 권한(Zanzibar)**으로 수조 개의 관계를 밀리초 안에 판단한다. DB 조회를 없애는 마법은 없고, **줄이는 전략과 트레이드오프의 선택**이 아키텍처의 본질이다.
