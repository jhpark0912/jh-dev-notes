# Day 017. JVM 메모리와 GC — 왜 서버가 가끔 멈출까?

> 🗓️ 2026-03-27  
> 🏷️ Tags: `JVM` `GC` `Heap` `Stack` `Eden` `G1GC` `ZGC` `StopTheWorld` `메모리누수`

---

## 🤔 질문

JVM 메모리는 어떻게 나뉘어 있고, 각 영역의 이름은 왜 그렇게 정해졌을까?  
GC는 "쓰레기"를 어떻게 찾고, Stop-The-World는 왜 발생할까?  
실무에서 GC를 신경 써야 하는 순간은 언제이고, 어떻게 모니터링할까?

---

## 📌 핵심 요약

JVM의 Heap은 **Young/Old 세대로 나뉘어** 관리되며, "대부분의 객체는 금방 죽는다"는 세대별 가설에 기반한다.  
GC는 **GC Root에서 참조 체인을 따라가 도달 불가능한 객체를 수거**하며, 이 과정에서 Stop-The-World가 발생한다.  
실무에서는 **잘 돌아가면 GC를 직접 분석할 필요 없지만**, Actuator + Grafana로 Heap 추이와 GC Pause를 모니터링하다가 징후가 보일 때만 분석하면 된다.

---

## 🔍 Part 1. JVM 메모리 구조

### 1-1. 전체 구조

```
┌─────────────────────────────────────────┐
│              JVM 메모리                   │
│                                         │
│  Heap (힙):     객체, 배열 저장 → GC가 관리│
│  Stack (스택):   지역 변수, 메서드 호출 정보│
│                 → 스레드마다 하나, 자동 정리│
│  Method Area:   클래스 정보, static, 바이트코드│
│  PC Register, Native Stack              │
└─────────────────────────────────────────┘
```

### 1-2. Heap vs Stack

```java
public void createUser() {
    int age = 25;                       // Stack
    String name = new String("홍길동");  // 객체는 Heap, 참조는 Stack
    User user = new User(name, age);    // 객체는 Heap, 참조는 Stack
}
```

Stack은 메서드가 끝나면 자동 정리(포인터만 이동). Heap은 누군가 참조하고 있을 수 있어 **"아무도 참조 안 하는 객체"를 찾아서 정리**해야 한다. 이게 GC의 일이다.

비유하면, Stack은 **카페 쟁반**(나가면 쟁반째 치움)이고, Heap은 **뷔페 테이블**(아무도 안 먹는 음식만 골라서 치워야 함)이다.

---

### 1-3. 이름의 유래

**Stack:** 자료구조 스택과 같은 LIFO 원리. 메서드 호출 시 쌓이고(push), 끝나면 빠진다(pop).

**Heap:** 자료구조 힙과는 무관. 영어 "heap(무더기, 더미)"에서 유래. 객체가 순서 없이 쌓이는 공간이라 "정리 안 된 더미"라는 의미.

**Method Area:** 클래스의 메서드 정보(바이트코드)가 저장되는 곳. Java 7 이전에는 PermGen(Permanent Generation, 영구 세대)이라 불렸으나, 크기 고정 문제로 Java 8부터 **Metaspace**로 대체. OS 네이티브 메모리를 사용하여 크기 제한이 유연해졌다.

---

## 🔍 Part 2. GC가 쓰레기를 찾는 방법

### 2-1. GC Roots에서 참조 체인 따라가기

```
GC Roots (출발점):
  → 현재 실행 중인 스레드의 Stack 변수들
  → static 변수
  → JNI 참조

GC Root에서 참조를 따라 도달할 수 있는 객체 = 살아있음
도달할 수 없는 객체 = 쓰레기 → 수거!
```

비유하면, GC는 **도서관 사서**이고 GC Root는 **대출 기록**이다. 대출 기록에 없는 책은 아무도 안 읽고 있으므로 폐기 가능하다.

---

## 🔍 Part 3. Heap의 세대별 구조 — Generational GC

### 3-1. 세대별 가설

1984년 David Ungar의 논문 "Generation Scavenging"에서 제안. **"대부분의 객체는 젊어서 죽는다(Infant Mortality)"** — 실제로 80~95%의 객체가 생성 직후 사라진다.

### 3-2. Heap 구조와 이름의 유래

```
┌──────────────────────────────────────────┐
│ Young Generation (젊은 세대)               │
│  Eden: 모든 객체가 처음 태어나는 곳         │
│        (성경의 에덴동산에서 유래)            │
│  Survivor S0, S1: GC를 살아남은(survived)  │
│        객체가 가는 곳 (두 개가 핑퐁)        │
├──────────────────────────────────────────┤
│ Old Generation (늙은 세대)                 │
│  여러 번의 GC를 살아남아 "늙은" 객체        │
│  DB 커넥션 풀, 캐시, Spring Bean 등        │
│  "Tenured(종신 재직)"라고도 불림           │
└──────────────────────────────────────────┘
```

Survivor가 2개인 이유: 한쪽에서 다른 쪽으로 살아남은 것만 복사하면 메모리가 연속적으로 정리되어 단편화가 없다. 방 2개로 이사하면서 필요한 물건만 옮기고 나머지를 통째로 비우는 것과 같다.

---

### 3-3. Minor GC — Young Generation 청소

```
1. Eden이 가득 참 → Minor GC 발동
2. 살아있는 객체만 Survivor로 복사
3. Eden 전체를 한 번에 비움 (통째로 리셋, 빠름!)
4. S0 ↔ S1 핑퐁하며 반복
5. age가 임계값(기본 15)을 넘으면 → Old로 승격
```

Minor GC가 빠른 이유: 대부분(80~95%) 죽으므로 살아있는 소수만 복사하면 되고, Eden을 개별 삭제가 아닌 영역 전체 리셋으로 비운다. 보통 수 밀리초~수십 밀리초.

### 3-4. Major GC (Full GC) — Old Generation 청소

Old Generation이 가득 차면 전체 Heap을 대상으로 청소. 수백 밀리초~수 초가 걸리며, 여기서 Stop-The-World가 문제된다.

---

## 🔍 Part 4. Stop-The-World

GC가 참조 상태를 정확히 파악하려면 앱이 참조를 바꾸면 안 되므로, **모든 애플리케이션 스레드를 일시 정지**시킨다.

```
STW 10ms:  대부분 못 느낌
STW 100ms: "앱이 멈칫했는데?"
STW 500ms: "요청이 0.5초간 응답 없었어!"
STW 1초+:  "서버가 죽은 줄 알았어!"

→ Heap이 크면 정리할 게 많아서 STW가 길어짐
→ 이걸 줄이는 게 GC 알고리즘 발전의 핵심
```

---

## 🔍 Part 5. GC 알고리즘의 진화

**Serial GC:** 싱글 스레드 GC. 코어 1개 환경용. 지금은 거의 안 씀.

**Parallel GC:** 여러 GC 스레드가 동시 청소. STW는 있지만 Serial보다 빠름. Java 8 기본.

**G1GC:** Heap을 작은 Region으로 분할하여 "쓰레기가 많은 곳(Garbage First)"부터 우선 청소. 전체를 청소하지 않아 STW가 짧음. `-XX:MaxGCPauseMillis=200`으로 목표 시간 설정 가능. Java 9부터 기본. 대부분의 실무 환경에서 사용 중.

**ZGC:** STW를 1ms 이내로 줄이는 것이 목표. 대부분의 GC 작업을 앱과 동시에(Concurrent) 수행. Heap이 수 TB여도 STW 1ms 미만. Java 15 정식 도입. 지연 시간이 중요한 서비스에 적합.

```
진화 요약:

Serial (1990년대):  싱글 스레드, STW 길음
Parallel (2000년대): 멀티 스레드, STW 줄어듦
G1 (2012~):        Region 단위, 목표 시간 설정 가능
ZGC (2020~):       거의 STW 없음, 대규모 Heap 적합
```

## 🔍 Part 6. 실무에서 GC 모니터링

### 6-1. 안 봐도 되는 경우

트래픽이 적고, 응답 시간이 안정적이고, OOM이 안 나고, 메모리가 넉넉하면 GC를 신경 안 써도 된다. "잘 돌아가면 건드리지 마라"는 실무 원칙.

### 6-2. 봐야 하는 징후

```
🚨 응답 시간이 간헐적으로 튀는 경우: 평소 50ms → 가끔 2000ms → Full GC의 STW 의심
🚨 메모리 사용량이 계속 증가: 톱니 모양이 아니라 우상향 → 메모리 누수 의심
🚨 OOM 발생: Heap이 꽉 참 → 반드시 분석 필요
🚨 CPU가 갑자기 100%: GC Thrashing (GC가 돌아도 공간이 안 나옴) 의심
```

### 6-3. 최소한의 모니터링 체계

Actuator + Prometheus + Grafana로 자동 감지:

```
Actuator 메트릭:
  jvm.memory.used     → 현재 Heap 사용량
  jvm.gc.pause        → GC 멈춤 시간 (STW)
  jvm.gc.memory.promoted → Old로 승격된 양

Grafana 대시보드:
  Heap 그래프가 톱니 모양 → 정상 (GC가 주기적으로 정리)
  Heap 그래프가 우상향   → 메모리 누수!

알림 설정:
  Heap 80% 초과, GC Pause 500ms 초과 → Slack 알림
```

### 6-4. 문제 발생 시 대응 순서

```
1단계: Grafana에서 징후 감지 (자동 알림)
2단계: GC 로그 확인 (Full GC 빈도, 메모리 감소 여부)
  → 줄어들면: Heap이 작음 → -Xmx 늘리기
  → 안 줄어들면: 메모리 누수 → 3단계로
3단계: 힙 덤프 분석 (/actuator/heapdump → Eclipse MAT)
4단계: 원인 수정 + 재배포
```

## 🔍 Part 7. 메모리 누수 — GC가 있는데도 메모리가 새는 경우

GC는 "아무도 참조 안 하는 객체"를 수거한다. 실수로 참조를 안 끊으면 GC가 수거 못 한다.

```
대표적 메모리 누수 패턴:

1. static 컬렉션에 계속 추가만 하고 삭제 안 함
   → static이라 앱 종료까지 참조 유지 → GC 수거 불가

2. 이벤트 리스너 해제 안 함
   → 리스너가 외부 객체를 참조 → 해제 전까지 GC 안 됨

3. Connection 닫지 않음
   → try-with-resources로 방지
```

## 🔍 Part 8. 실무 JVM 설정

```
java -jar app.jar \
  -Xms512m                    # 초기 Heap 크기
  -Xmx2g                      # 최대 Heap 크기 (Xms와 같게 권장)
  -XX:+UseG1GC                # G1GC 사용 (Java 9+ 기본)
  -XX:MaxGCPauseMillis=200    # 목표 STW 시간
  -Xlog:gc*:file=gc.log:time  # GC 로그 활성화
```

Xms와 Xmx를 같게 설정하면 Heap이 늘어날 때마다 OS에 메모리를 요청하는 오버헤드를 방지하여 안정적이다.

---

## 💡 한 줄 정리

> JVM의 Heap은 **"대부분의 객체는 금방 죽는다"는 세대별 가설**에 기반해 Eden/Survivor/Old로 나뉘며, GC 알고리즘은 Serial → G1 → ZGC로 **Stop-The-World를 줄이는 방향**으로 진화했다. 실무에서 GC는 **잘 돌아가면 신경 안 써도 되지만**, Actuator + Grafana로 **Heap 추이와 GC Pause를 모니터링하는 체계**를 갖추고 징후가 보일 때만 분석하는 것이 현실적이다.
