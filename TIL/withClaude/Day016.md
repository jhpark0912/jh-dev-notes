# Day 016. 코드 한 줄이 CPU에서 실행되기까지 — JVM과 컴파일의 세계

> 🗓️ 2026-03-26  
> 🏷️ Tags: `JVM` `바이트코드` `JIT` `AOT` `GraalVM` `컴파일` `인터프리터` `Kotlin` `Go` `Rust`

---

## 🤔 질문

Java 코드가 .java에서 화면 출력까지 어떤 과정을 거칠까?  
바이트코드는 왜 존재하고, JIT 컴파일러는 어떻게 코드를 점점 빠르게 만들까?  
컴파일 언어와 인터프리터 언어는 왜 나뉘어 있고, 프로그래밍 언어는 왜 계속 만들어질까?

---

## 📌 핵심 요약

Java 코드는 **.java → 바이트코드(.class) → JVM 인터프리터 → JIT 컴파일 → 기계어** 순서로 실행된다.  
바이트코드는 **"어디서든 실행"** 을 위한 중간 언어이고, JIT는 자주 실행되는 코드를 감지하여 **런타임에 점점 더 빠르게 만드는** 적응형 최적화다.  
모든 프로그래밍 언어는 **기존 언어의 불만에서 탄생**하며, 컴퓨팅 환경이 바뀌는 한 새 언어는 계속 나올 것이다.

---

## 🔍 Part 1. Java 코드의 여정

### 1-1. 소스코드에서 CPU 실행까지 5단계

```
1단계: .java (소스코드) → 개발자가 작성한 텍스트 파일
2단계: .class (바이트코드) → javac 컴파일러가 변환, JVM이 이해하는 중간 언어
3단계: JVM 인터프리터 → 바이트코드를 한 줄씩 실행
4단계: JIT 컴파일러 → 자주 쓰는 코드를 기계어로 변환
5단계: CPU가 기계어를 실행 → 화면에 결과 출력
```

---

### 1-2. 바이트코드가 존재하는 이유: "Write Once, Run Anywhere"

C/C++은 소스코드를 바로 기계어로 컴파일하는데, 기계어가 OS와 CPU마다 달라서 **각 환경에서 따로 컴파일**해야 한다.

Java는 바이트코드라는 중간 언어를 거치므로, **.class 파일 하나가 어느 OS의 JVM에서든 실행**된다.

```
C: hello.c → Windows .exe / Mac 바이너리 / Linux 바이너리 (각각 빌드)
Java: Hello.java → Hello.class (하나만 빌드) → 어디서든 JVM으로 실행
```

비유하면, C는 **각 나라 언어로 따로 번역한 책**이고, Java 바이트코드는 **에스페란토(공용어)로 쓴 책**이다. 통역사(JVM)만 있으면 어느 나라에서든 읽을 수 있다.

Docker 이미지에 JVM을 포함시키면, Java의 "어디서든 동일" + Docker의 "어디서든 동일"이 결합되어 완벽한 이식성을 달성한다.

---

### 1-3. JVM 내부 구조

```
┌───────────────────────────────────┐
│                JVM                 │
│                                   │
│  Class Loader: .class를 메모리에 로드│
│         ↓                          │
│  Runtime Data Areas (메모리 영역)    │
│    Heap (객체), Stack (지역변수)     │
│    Method Area, PC Register        │
│         ↓                          │
│  Execution Engine (실행 엔진)       │
│    Interpreter (한 줄씩 실행)       │
│    JIT Compiler (기계어 변환)       │
│         ↓                          │
│  네이티브 기계어 → CPU 실행          │
└───────────────────────────────────┘
```

---

## 🔍 Part 2. 인터프리터 vs JIT — 실행 방식의 핵심

### 2-1. 인터프리터: 한 줄씩 번역

```
바이트코드 한 줄 읽기 → 기계어 변환 → 실행 → 다음 줄...

장점: 시작이 빠름 (바로 실행)
단점: 같은 코드를 만날 때마다 매번 다시 번역 → 느림
```

### 2-2. JIT: 자주 쓰는 코드를 감지하여 기계어로 컴파일

```
처음에는 인터프리터로 실행하면서 모니터링

"이 메서드가 10,000번째 호출이네? Hot Code다!"
→ JIT가 통째로 기계어로 컴파일
→ 다음부터는 기계어를 직접 실행!

1~9,999번: 인터프리터 (느림)
10,000번:  JIT 컴파일 발동 → 기계어 생성
10,001번~: 기계어 직접 실행 (빠름!)
```

비유하면, 자주 시키는 음식의 레시피를 처음엔 보면서 만들다가(인터프리터), 외워버린 뒤 레시피 없이 만드는 것(JIT)이다.

---

### 2-3. JIT가 하는 최적화들

**메서드 인라이닝:** 자주 호출되는 작은 메서드의 코드를 호출 지점에 직접 삽입하여 호출 오버헤드를 제거.

**루프 최적화:** 반복문 안에서 매번 계산할 필요 없는 값을 밖으로 꺼내고, 배열 경계 체크를 제거.

**탈출 분석:** 메서드 밖으로 나가지 않는 객체는 힙이 아닌 스택에 할당하거나, 아예 객체 생성을 제거하여 GC 부담 감소.

---

### 2-4. C1과 C2 — 단계적 컴파일 (Tiered Compilation)

```
JVM 내부에 컴파일러가 2개:

C1 (Client): 빠르게 컴파일, 가벼운 최적화 → 시작 속도 중시
C2 (Server): 느리게 컴파일, 강력한 최적화 → 실행 속도 중시

단계적 실행:
  1단계: 인터프리터 (즉시 실행)
  2단계: C1 컴파일 (빠른 개선)
  3단계: 프로파일링 (어디가 느린지 분석)
  4단계: C2 컴파일 (최대 성능)

→ 시작은 빠르게, 시간이 지나면 점점 빨라짐
```

---

### 2-5. Spring Boot vs Quarkus와의 연결

Spring Boot의 시작이 느린 이유: 앱 시작 시 수천 개의 .class 로드, 클래스 스캔, Bean 생성이 "인터프리터 + C1" 단계에서 실행되어 아직 C2 최적화가 안 됐기 때문. 한 번 뜬 후 시간이 지나면 C2가 최적화하여 점점 빨라진다(워밍업).

Quarkus/GraalVM Native Image의 접근: JIT가 런타임에 하는 일을 **빌드 시점에 미리 처리(AOT)**. JVM 없이 직접 실행되는 네이티브 바이너리를 생성하여 시작 0.05초를 달성. 대신 JIT의 런타임 적응형 최적화가 없어, 지속적 부하에서는 JVM이 더 빠를 수 있다.

```
시작 시간:       Native 압승 (0.05초 vs 1.91초)
지속 부하 처리량: JVM이 비슷하거나 더 빠름 (C2 최적화 후)

→ 자주 껐다 켜는 환경 (서버리스, K8s): Native 유리
→ 한 번 띄워두는 환경 (일반 서버): JVM도 충분
```

## 🔍 Part 3. 컴파일 언어 vs 인터프리터 언어 — 왜 나뉘어 있는가?

### 3-1. 트레이드오프

```
                  컴파일 (C, Rust)   인터프리터 (Python, JS)   하이브리드 (Java)
실행 속도          가장 빠름           느림                     빠름 (JIT 후)
시작/개발 속도     느림 (빌드 대기)     빠름 (즉시 실행)          중간
이식성             낮음 (OS별 빌드)     높음                     높음 (바이트코드)
동적 기능          제한적              강력                     제한적
주요 용도          OS, 게임, 임베디드   스크립트, 데이터, 웹      엔터프라이즈, 백엔드
```

### 3-2. 용도가 다르다

**최대 성능:** C, C++, Rust → OS 커널, 게임 엔진, 임베디드. 1ms의 차이가 중요한 곳.

**빠른 개발:** Python, JavaScript → 데이터 분석, 프로토타입, 스크립트. 실행 속도보다 개발 속도가 중요한 곳.

**균형:** Java, Kotlin, C# → 엔터프라이즈 백엔드. 충분히 빠르면서 개발 생산성도 필요한 곳.

### 3-3. 동적 언어의 유연함은 컴파일로 구현하기 어렵다

Python의 `exec(code)`처럼 실행 중에 코드 자체를 생성하고 실행하거나, 변수 타입이 런타임에 바뀌는 동적 기능은 컴파일 타임에 처리하는 것이 불가능하거나 매우 어렵다.

### 3-4. 경계가 흐려지고 있다

```
Python: CPython이 바이트코드로 컴파일, PyPy는 JIT 탑재
JavaScript: V8 엔진이 JIT 컴파일 (Node.js가 서버로 쓸 만큼 빨라짐)
Go: 컴파일 속도가 수 초 (인터프리터처럼 빠른 피드백)
Java: GraalVM으로 AOT 추가 (시작 속도 극적 향상)

→ 인터프리터 언어는 JIT를 추가하고
   컴파일 언어는 빌드 속도를 줄이고
   하이브리드는 AOT를 추가하며
   모든 언어가 서로의 장점을 흡수하는 중
```

## 🔍 Part 4. 프로그래밍 언어는 왜 계속 만들어지는가?

### 4-1. 모든 언어는 "기존 언어의 불만"에서 탄생

**Kotlin (2011, JetBrains):** "Java가 너무 장황해. JVM 생태계는 버릴 수 없으니, JVM에서 돌아가면서 간결한 언어를 만들자." → Java의 30줄 데이터 클래스가 `data class User(val name: String, val age: Int)` 한 줄로. Google이 2019년 Android 공식 언어로 채택.

**Go (2009, Google):** "C++ 컴파일이 45분 걸려. 서버에 동시성이 중요한데 C++로는 어려워." → 수 초 컴파일, `go` 키워드 하나로 동시 실행. Docker, Kubernetes가 Go로 작성됨.

**Rust (2010, Mozilla):** "C++만큼 빠르면서 메모리 안전한 언어는 없을까?" → 소유권 시스템으로 컴파일 타임에 메모리 버그 차단. 2024년 Linux 커널에 공식 포함.

**TypeScript (2012, Microsoft):** "JavaScript가 대규모 프로젝트에서 너무 불안해." → 타입 시스템 추가, JavaScript와 100% 호환. Kotlin이 Java에 한 것 = TypeScript가 JavaScript에 한 것.

**Swift (2014, Apple):** "Objective-C가 1980년대 문법이라 불편해." → 현대적이고 안전한 iOS 개발 언어.

### 4-2. 누가 만드는가?

**대기업:** Google(Go, Dart), Apple(Swift), Microsoft(TypeScript, C#), JetBrains(Kotlin), Mozilla(Rust). 자기 문제를 해결하려고 만듦.

**개인:** Guido van Rossum(Python, 크리스마스 휴가에 취미로 시작), Brendan Eich(JavaScript, 10일 만에 설계), Yukihiro Matsumoto(Ruby, "프로그래밍이 즐거워야 해").

### 4-3. 새 언어의 생존 전략

**기존 생태계에 올라타기:** Kotlin → JVM(Java 라이브러리 전부 사용), TypeScript → npm(JavaScript 생태계 그대로). 생태계를 새로 만들 필요 없음.

**킬러 유스케이스 확보:** Python → 머신러닝, Go → 인프라 도구(Docker), Rust → 시스템 프로그래밍, Swift → iOS. "이 분야에서는 이 언어가 최고"라는 영역을 확보.

### 4-4. 언어가 계속 나오는 이유

```
1970~80년대: 메모리가 비쌈 → C (직접 제어)
1990년대:   인터넷 등장 → Java (이식성), JavaScript (브라우저)
2000년대:   웹 스케일 → Python (생산성)
2010년대:   클라우드/모바일 → Go (동시성), Kotlin (Android), Swift (iOS)
2020년대:   보안/안전 → Rust (메모리 안전)

→ 환경이 바뀌면 기존 언어의 한계가 드러남
→ 한계를 해결하는 새 언어가 등장
→ 이 사이클은 계속됨
```

하지만 오래된 언어도 잘 죽지 않는다. C(1972), Java(1995), Python(1991) 모두 수십 년째 현역이며, 새 언어가 기존 영역을 완전히 대체하기는 어렵다. 각자의 영역에서 공존한다.

---

## 💡 한 줄 정리

> Java 코드는 **바이트코드(이식성) → JIT(적응형 최적화)** 를 거쳐 실행되며, 이 설계가 Spring Boot의 워밍업과 Quarkus AOT의 트레이드오프를 만든다. 컴파일과 인터프리터가 나뉘어 있는 이유는 **실행 속도 vs 개발 속도의 트레이드오프** 때문이며, 모든 언어는 **"기존 언어의 불만"에서 탄생**한다. 컴퓨팅 환경이 바뀌는 한 새 언어는 계속 나올 것이고, 현대 언어들은 JIT·AOT 등으로 서로의 장점을 흡수하며 경계가 흐려지고 있다.
