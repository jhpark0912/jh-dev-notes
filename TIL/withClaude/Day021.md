# Day 021. 왜 레거시 코드는 죽지 않을까? — 기술 부채와 마이그레이션의 현실

> 🗓️ 2026-04-02  
> 🏷️ Tags: `기술부채` `레거시` `리팩토링` `교살자패턴` `마이그레이션` `COBOL` `.NET→Spring`

---

## 🤔 질문

기술 부채란 무엇이고, 왜 "나중에 고치자"는 대부분 실현되지 않을까?  
COBOL이 67년째 은행에서 돌아가는 이유는 뭘까?  
원 제작자 없는 레거시 시스템을 새 기술 스택으로 전환할 때 가장 주의할 점은?

---

## 📌 핵심 요약

기술 부채는 **"나중에 고치자"가 쌓여 개발 속도를 점점 느리게 만드는 것**이며, 개발자 시간의 23~42%가 부채 처리에 소비된다.  
COBOL이 67년째 은행에서 돌아가는 이유는 **교체 비용·리스크·검증 불가능성** 때문이며, 재작성은 대부분 실패한다.  
원 제작자 없는 레거시 전환에서 가장 위험한 건 **코드에 숨겨진 비즈니스 규칙을 놓치는 것**이며, **교살자 패턴으로 점진적 전환** + **Shadow Traffic으로 결과 비교 검증**하는 것이 핵심이다.

---

## 🔍 Part 1. 기술 부채란 무엇인가

### 1-1. 금융 부채에서 따온 비유

Ward Cunningham(위키 발명가)이 1992년 제안한 용어. 의도적으로 금융 개념을 빌려왔다.

```
금융 부채: 돈을 빌려 씀 → 이자가 쌓임 → 갚지 않으면 복리로 증가
기술 부채: 빠른 방법을 택함 → 유지보수 비용이 쌓임 → 방치하면 복잡도가 기하급수 증가
```

개발자 시간의 23~42%가 기술 부채 처리에 사용된다. 주 5일 중 1~2일은 "빚 갚는 데" 쓰는 셈.

### 1-2. 기술 부채의 4가지 유형

```
                    신중한                    무모한
 의도적           전략적 부채                 위험한 부채
 (알고 함)        "일단 이렇게 하고           "설계? 시간 없어
                  나중에 고치자"              일단 돌아가게 해"

 비의도적         학습 부채                   가장 위험
 (모르고 함)      "더 좋은 방법이             "뭐가 문제인지도
                  있었네"                     모름"
```

전략적 부채는 비즈니스 판단으로 합리적일 수 있지만, "뭐가 문제인지도 모르는" 비의도적+무모한 부채는 재작성 외에 답이 없을 수 있다.

## 🔍 Part 2. COBOL이 아직도 은행에서 돌아가는 이유

1959년에 탄생한 COBOL은 전 세계 금융 거래의 약 95%를 처리하며, 코드 총량은 약 8,000억 줄로 추정된다.

```
교체할 수 없는 이유:

규모: 수십 년간 쌓인 비즈니스 규칙이 수만 개, 문서화 안 된 것도 많음
리스크: 은행 시스템은 99.99%가 아닌 100% 동일해야 함
비용: 코어 뱅킹 교체 비용 수천억~수조 원, 기간 5~10년
안정성: 67년간 버그가 수정되어온 시스템, 새 시스템은 새 버그를 가져옴
```

---

## 🔍 Part 3. 실제 대규모 마이그레이션 사례

### 성공: Twitter (Ruby on Rails → Java/Scala, 2010~2013)

사용자 폭증으로 Rails의 성능 한계 + 기술 부채가 동시 폭발. 전체를 한 번에 바꾸지 않고 핵심 서비스부터 Java/Scala로 하나씩 전환. **점진적 전환이 성공의 핵심.**

### 성공: Amazon (모놀리스 → MSA, 2001~2006)

Jeff Bezos의 "모든 팀은 서비스 인터페이스를 통해서만 통신하라" 메모. 모놀리스를 수백 개 독립 서비스로 분리. 이 전환이 AWS 탄생의 기반이 됨. **경영진의 강력한 의지 + 명확한 원칙.**

### 실패: TSB 은행 (2018)

530만 고객의 코어 뱅킹 시스템을 한 번에 전환(빅뱅 방식). 190만 명이 접속 불가, 일부 고객에게 다른 사람의 계좌가 보이는 사고. CEO 사임, 약 5,500억 원 비용 발생. **빅뱅 전환의 위험성을 보여주는 대표 사례.**

## 🔍 Part 4. 리팩토링 vs 재작성

Joel Spolsky(StackOverflow 창업자): "재작성은 소프트웨어 회사가 할 수 있는 가장 최악의 전략적 실수다." 기존 코드에는 수년간의 버그 수정이 녹아있고, 재작성하면 이 지식을 잃는다.

```
재작성이 필요한 경우:
  → 기술 스택이 더 이상 지원되지 않을 때
  → 기존 코드를 이해하는 사람이 없을 때
  → 비즈니스 요구사항이 근본적으로 바뀌었을 때

리팩토링으로 충분한 경우 (대부분!):
  → "코드가 더럽다" 정도의 문제
  → 성능 개선, 테스트 추가가 필요할 때
```

## 🔍 Part 5. 교살자 패턴 (Strangler Fig Pattern)

레거시 시스템 앞에 프록시를 세우고, 새 기능은 새 시스템에 구현하며, 기존 기능을 하나씩 이전하여 레거시를 점점 줄여나가는 전략. Amazon, Twitter가 사용한 방식.

```
Phase 0: 사용자 → [프록시] → 레거시 100%
Phase 1: 사용자 → [프록시] → 레거시 80% + 새 시스템 20% (새 기능)
Phase 2: 사용자 → [프록시] → 레거시 30% + 새 시스템 70% (이전 중)
Phase 3: 사용자 → [프록시] → 새 시스템 100% → 레거시 제거

→ 언제든 프록시에서 라우팅만 바꾸면 롤백 가능
```

## 🔍 Part 6. 실전 — .NET Framework → Spring Boot + React 전환

### 6-1. 가장 위험한 것: 코드에 숨겨진 비즈니스 규칙

원 제작자가 없으면 코드의 의도를 알 수 없다. "일요일 할인 0%"가 비즈니스 규칙인지 테스트 잔재인지 판단할 수 없으며, 새 시스템에서 빼면 비즈니스 로직이 망가질 수 있다.

### 6-2. .NET 특유의 숨겨진 로직

```
Global.asax / Startup.cs: 요청 파이프라인의 인증, 로깅, 리다이렉트
Web.config: URL Rewrite, 커스텀 에러, 세션/인증 설정
Stored Procedure: .NET 레거시는 비즈니스 로직이 DB에 있는 경우가 많음
IIS 설정: 인증 방식, HTTPS, 정적 파일 규칙
DB Trigger: 코드에서 안 보이지만 INSERT/UPDATE 시 자동 실행
```

이것들은 코드 리뷰만으로 발견할 수 없으므로 별도 조사가 필수다.

### 6-3. 전환 전 필수 조사

```
1. API/URL 전수 조사: IIS 로그 분석으로 실제 사용 URL만 추출
2. DB 조사: 스키마 + SP + 트리거 전수 확인
3. 사용자 흐름: 현업 담당자에게 화면 흐름 시연받기
4. 외부 연동: PG사, SMS 등 HttpClient/WebRequest grep 검색
```

### 6-4. 빅뱅 전환 금지 — 교살자 패턴 적용

Nginx 프록시를 앞에 배치하고, URL 패턴별로 IIS와 Spring Boot에 라우팅. 새 기능은 Spring Boot에 구현하고, 기존 기능을 하나씩 이전. Nginx에서 라우팅만 바꾸면 즉시 롤백 가능.

### 6-5. 동일 결과 보장 — 테스트 전략

**API 비교 테스트:** 기존 IIS와 새 Spring Boot에 같은 요청을 보내 응답을 자동 비교.

**Shadow Traffic:** 실제 트래픽을 두 시스템에 동시에 보내되, 사용자에게는 IIS 응답만 전달. Spring Boot 응답은 기록만 하여 차이를 분석. 차이가 0이 될 때까지 반복 후 전환.

**DB 결과 비교:** 같은 작업 수행 후 DB 상태를 비교. 날짜 포맷, NULL 처리, 소수점, 인코딩 차이에 특히 주의.

### 6-6. C# → Java 전환 시 함정

```
날짜/시간: DateTime vs LocalDateTime/ZonedDateTime, 타임존 처리 차이
NULL: int?(nullable value type) vs Integer, null 동작 차이
소수점: decimal vs BigDecimal, 반올림 모드 차이
문자열: string.IsNullOrEmpty() vs StringUtils.isEmpty()
LINQ → Stream API: 비슷하지만 세부 동작(정렬, null) 차이
```

### 6-7. 데이터 마이그레이션 (SQL Server → PostgreSQL)

```
데이터 타입: nvarchar→varchar, datetime2→timestamp, bit→boolean
SP 문법: 완전히 다름 → 가능하면 Spring Boot 코드로 이전 (테스트 가능)
자동 증가 ID: IDENTITY → SERIAL, 기존 ID 값 보존 필요
인코딩: SQL Server 기본 인코딩 → PostgreSQL UTF-8, 한글 깨짐 주의
```

### 6-8. 인증 전환

```
Windows 인증: Spring Boot에서 LDAP/AD 연동 또는 OAuth2로 전환
Forms 인증: JWT로 전환 권장, 전환 기간 중 양쪽 인증 공존 필요
ASP.NET Identity: .NET과 Spring Security의 비밀번호 해시가 호환 안 됨!
  → 사용자에게 비밀번호 재설정 요청 또는 커스텀 PasswordEncoder 구현
```

## 🔍 Part 7. 기술 부채 관리 전략

**보이스카우트 규칙:** "캠핑장을 떠날 때 왔을 때보다 깨끗하게." 코드 수정 시 주변도 조금씩 개선.

**20% 규칙:** 스프린트의 20%를 기술 부채 상환에 할당. 경영진과 합의 필수.

**교살자 패턴:** 레거시를 한 번에 교체하지 않고, 프록시 뒤에서 점진적으로 대체.

## 💡 한 줄 정리

> 기술 부채는 **금융 부채처럼 이자(유지보수 비용)가 복리로 쌓이며**, 재작성은 대부분 실패한다. 원 제작자 없는 레거시 전환에서 가장 위험한 건 **SP·트리거·설정 파일에 숨겨진 비즈니스 규칙**이며, **교살자 패턴으로 점진적 전환** + **Shadow Traffic으로 결과 비교 검증** + **C#↔Java의 날짜·NULL·소수점·인코딩·인증 차이 체크**가 성공적 전환의 핵심이다.
