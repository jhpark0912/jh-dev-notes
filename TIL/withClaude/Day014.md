# Day 014. 왜 쿠팡은 주문 후 '결제 완료'가 바로 안 뜰까? — 메시지 큐와 비동기 처리

## 🤔 질문

메시지 큐는 왜 필요하고, Kafka와 RabbitMQ는 뭐가 다를까?  
대규모 서비스에서 주문-결제-재고의 일관성은 어떻게 보장할까?  
Saga 패턴의 보상(롤백)은 내부적으로 어떻게 동작할까?

---

## 📌 핵심 요약

메시지 큐는 **서비스 간 결합도를 낮추고, 트래픽 폭증을 버퍼링하고, 장애 시 메시지를 보존**한다.  
Kafka는 메시지를 삭제하지 않는 **로그 기반 설계**로 대규모 이벤트 스트리밍의 표준이 되었다.  
대규모 서비스는 **Saga 패턴**으로 각 단계를 독립 실행하고, 실패 시 개발자가 미리 만들어둔 **보상 API**를 역순으로 호출하여 결과적 일관성을 달성한다.

## 🔍 Part 1. 동기 vs 비동기 처리

### 1-1. 동기 처리의 한계

```
동기 처리 (전부 기다림):

사용자 → 주문 → 결제(2초) → 재고(1초) → 포인트(0.5초) → 알림(1초) → 배송(1초)
→ 응답까지 5.5초!
→ 중간에 알림이 느려지면 전체가 느려짐
```

### 1-2. 비동기 처리

```
비동기 처리 (핵심만 기다림):

사용자 → 주문 → 결제(2초) → 응답! "주문 완료!" (2초)
                ↓
        [메시지 큐에 이벤트 발행]
                ↓
        재고·포인트·알림·배송이 각자 처리
→ 사용자는 2초만 기다림, 나머지는 백그라운드
```

## 🔍 Part 2. 메시지 큐의 기본 개념

### 2-1. 핵심 구조

```
Producer (생산자) → [큐/토픽] → Consumer (소비자)

주문 서비스 (Producer)
    │ "주문 #1234 완료" 메시지 발행
    ↓
[  메시지 큐  ]
    ├── 재고 서비스 (Consumer): 재고 차감
    ├── 포인트 서비스 (Consumer): 포인트 적립
    ├── 알림 서비스 (Consumer): 카카오톡 발송
    └── 배송 서비스 (Consumer): 배송 접수
```

비유하면, 동기 호출은 **전화 통화**(상대방이 안 받으면 나도 대기)이고, 메시지 큐는 **문자 메시지**(보내고 내 할 일 함, 상대방은 나중에 처리)다.

### 2-2. 메시지 큐가 해결하는 문제 3가지

**결합도 감소:**  주문 서비스가 "주문 완료됐다" 이벤트만 발행하면 끝. 누가 소비하는지 관심 없음. 새 서비스 추가 시 새 Consumer만 만들면 되고 Producer 코드 변경 불필요.

**트래픽 버퍼링:**  타임세일로 순간 10배 폭증해도 큐에 쌓아두고 서버가 자기 속도로 처리. 서버가 죽지 않음.

**장애 복원력:**  알림 서비스가 30분 다운돼도 메시지가 큐에 보존. 복구 후 꺼내서 처리하면 유실 없음.

## 🔍 Part 3. Kafka vs RabbitMQ

### 3-1. 근본적 차이: 삭제 vs 보존

```
RabbitMQ (전통적 메시지 큐):
  Consumer가 ACK 보내면 → 메시지 삭제
  → 한 번 소비되면 없어짐

Kafka (이벤트 스트리밍):
  Consumer가 읽어도 → 메시지 유지 (설정 기간까지)
  → 여러 Consumer Group이 독립적으로 읽음
  → offset을 되돌려서 재처리 가능
```

비유하면, RabbitMQ는 **우체통**(꺼내면 사라짐)이고, Kafka는 **게시판**(여러 사람이 각자 읽고, 글은 남아있음)이다.

### 3-2. Kafka의 탄생 배경

2010년 LinkedIn에서 수십억 건의 이벤트를 기존 메시지 큐(ActiveMQ)로 처리할 수 없어, Jay Kreps·Neha Narkhede·Jun Rao가 "대용량 이벤트 스트리밍에 특화된 시스템"을 설계했다. 2011년 오픈소스 공개, 2014년 Confluent 창업.

### 3-3. Kafka가 표준이 된 이유

**처리량:** RabbitMQ 초당 수만 건 vs Kafka 초당 수백만 건 (약 100배). 디스크 순차 쓰기(append-only) 방식이 핵심.

**재처리:** 3일 전 포인트 로직 버그 → RabbitMQ는 메시지 삭제됨(불가능), Kafka는 offset 되돌려서 다시 처리 가능.

**파티션 병렬 처리:** 파티션 수 = 병렬 Consumer 수. 처리량 늘리려면 파티션과 Consumer를 늘리면 됨 (Scale-Out 개념).

**독립적 소비:** 하나의 토픽에서 재고·알림·포인트·배송·분석 서비스가 각자 속도로 독립적으로 읽음.

### 3-4. 선택 기준

```
Kafka: 대용량 이벤트 스트리밍, 여러 서비스 독립 소비, 재처리 필요, 데이터 파이프라인
RabbitMQ: 작업 큐, 복잡한 라우팅, 메시지 우선순위, 소규모 서비스
둘 다 쓰는 경우도 많음 (용도가 다르므로)
```

## 🔍 Part 4. 메시지 큐의 비용과 트레이드오프

### 4-1. 도입 시 생기는 비용

**인프라 복잡도:** 앱+DB(2개) → 앱+DB+Kafka 브로커(3대)+Consumer 앱(6~9개). 관리 대상이 3~4배 증가.

**디버깅 어려움:** 동기는 스택 트레이스 한 줄로 원인 파악. 비동기는 메시지가 큐에 들어갔는지, Consumer가 꺼냈는지, 어떤 파티션인지 추적 경로가 길어짐.

**데이터 일관성:** DB와 Kafka는 다른 시스템이라 하나의 트랜잭션으로 묶을 수 없음. DB 저장 성공 + Kafka 발행 실패 시 불일치 발생.

**순서 보장:** Kafka 파티션 내에서는 순서 보장, 파티션 간에는 보장 안 됨.

### 4-2. 규모별 선택 가이드

```
소규모 (일일 수천 건):     Spring Events (@Async + @EventListener), 외부 인프라 불필요
중규모 (일일 수만~수십만):  Redis Stream 또는 RabbitMQ
대규모 (일일 수백만+):     Kafka
```

## 🔍 Part 5. 대규모 서비스의 일관성 — Saga 패턴

### 5-1. 완벽한 일관성을 포기한다

서비스가 분리되면 DB가 다르므로 하나의 트랜잭션으로 묶을 수 없다. 대신 **"결과적 일관성(Eventual Consistency)"** 을 택한다. 순간적으로 불일치할 수 있지만, 보상 작업으로 결국 일치하게 만든다.

비유하면, 동기 트랜잭션은 **현금 거래**(동시에 교환)이고, 결과적 일관성은 **온라인 쇼핑**(돈 먼저, 물건 나중에. 안 오면 환불).

### 5-2. Saga의 핵심: 각 단계에 "되돌리기 API"를 미리 만든다

```
각 서비스가 갖고 있는 것:

결제 서비스:
  정방향: POST /payments        → 결제 실행
  보상:   POST /payments/refund → 환불 실행

재고 서비스:
  정방향: POST /inventory/decrease → 재고 차감
  보상:   POST /inventory/restore  → 재고 복원

배송 서비스:
  정방향: POST /delivery/request → 배송 접수
  보상:   POST /delivery/cancel  → 배송 취소
```

Saga의 보상은 프레임워크가 자동으로 해주는 게 아니라, **개발자가 모든 단계의 되돌리기를 직접 설계하고 구현**하는 것이다. 이 설계 비용이 크기 때문에, DB 트랜잭션으로 해결 가능하면 그게 최선이다.

### 5-3. Saga 실행 흐름

```
정상: 결제 ✅ → 재고 ✅ → 배송 ✅ → 완료!

3단계 실패:
  결제 ✅ → 재고 ✅ → 배송 ❌
  → 역순 보상: 재고 복원 → 결제 환불 → 주문 취소
  → 사용자: "재고 부족으로 취소되었습니다"
```

보상은 원래 메시지를 "되돌리는" 게 아니라 **"되돌려!"라는 새 보상 메시지를 발행**하는 것이다. 보상 메시지에 필요한 데이터(주문 ID, 금액 등)가 담겨있어, 원본 메시지가 삭제됐어도 독립적으로 롤백을 실행할 수 있다.

### 5-4. Saga의 두 가지 방식

**Choreography (이벤트 기반):** 각 서비스가 이벤트를 발행하고 다음 서비스가 반응. 중앙 관리자 없이 독립적이지만, 전체 흐름 파악이 어려움.

**Orchestration (중앙 조율자):** Saga Orchestrator가 전체 흐름을 관리. 한 곳에서 정방향/보상을 쌍으로 정의하여 디버깅이 쉽지만, 오케스트레이터가 SPOF가 될 수 있음.

### 5-5. RabbitMQ에서의 안전장치 계층

RabbitMQ의 메시지 삭제는 Consumer가 **ACK를 보낸 후에만** 발생한다.

```
1차 — ACK/NACK:
  성공 → ACK → 메시지 삭제
  일시적 실패 → NACK → 큐에 다시 넣어서 재시도

2차 — 보상 이벤트:
  비즈니스 실패 (재고 부족 등) → 보상 메시지 새로 발행 → 이전 단계 되돌림

3차 — Dead Letter Queue (DLQ):
  재시도도 보상도 실패한 메시지 → DLQ에 보관 → 개발자 수동 확인

4차 — 모니터링 + 알림:
  DLQ 메시지 수 등 모니터링 → 임계값 초과 시 Slack 알림
```

---

### 5-6. 현실적 주문 처리 구조

```
실무에서의 구분:

동기로 처리 (사용자가 기다림):
  → 주문 생성 + 결제 = 트랜잭션
  → 결제 실패 시 바로 "다른 카드로 시도해주세요"

비동기로 처리 (사용자가 안 기다림):
  → 재고 차감, 포인트 적립, 알림, 배송 접수
  → 결제 완료 후 Kafka에 이벤트 발행 → 각 서비스가 독립 처리
  → Saga로 보상 관리

핵심: "모든 걸 동기로" → 느리고 장애에 취약
     "모든 걸 비동기로" → 일관성 보장 어려움
     "핵심은 동기, 나머지는 비동기" → 현실적 최적!
```

### 5-7. "재고 없는데 결제됨"에 대한 대응

```
사전 검증: 주문 전 재고 확인 API 호출 → 99.9%는 여기서 걸러짐
재고 선점: 결제 전 재고 임시 차감 (5분간 선점, 결제 실패 시 복원)
극단적 상황: 자동 환불 + "재고 소진으로 주문 취소" 알림
→ 전체 시스템이 죽는 것보다 훨씬 나은 트레이드오프
```

## 🔍 Part 6. 실제 코드

### 6-1. Kafka Producer (주문 서비스)

```java
@Service
public class OrderService {
    @Autowired
    private KafkaTemplate<String, OrderEvent> kafkaTemplate;
    
    public void createOrder(Order order) {
        orderRepository.save(order);
        paymentService.charge(order);
        
        // 이벤트 발행하고 끝!
        kafkaTemplate.send("order-completed", new OrderEvent(
            order.getId(), order.getUserId(), order.getAmount()
        ));
    }
}
```

### 6-2. Kafka Consumer (알림 / 포인트 서비스)

```java
// 알림 서비스 — 독립적으로 처리
@KafkaListener(topics = "order-completed", groupId = "notification-service")
public void handleOrderCompleted(OrderEvent event) {
    kakaoAlimtalkService.send(event.getUserId(), "주문 완료!");
}

// 포인트 서비스 — 또 다른 독립적 처리
@KafkaListener(topics = "order-completed", groupId = "point-service")
public void handleOrderCompleted(OrderEvent event) {
    pointService.accumulate(event.getUserId(), (int)(event.getAmount() * 0.01));
}
```

### 6-3. Saga Orchestrator

```java
public class OrderSaga {
    private final List<SagaStep> steps = List.of(
        new SagaStep("결제",
            order -> paymentService.charge(order),
            order -> paymentService.refund(order)),
        new SagaStep("재고",
            order -> inventoryService.decrease(order),
            order -> inventoryService.restore(order)),
        new SagaStep("배송",
            order -> deliveryService.request(order),
            order -> deliveryService.cancel(order))
    );

    public void execute(Order order) {
        List<SagaStep> completed = new ArrayList<>();
        for (SagaStep step : steps) {
            try {
                step.forward(order);
                completed.add(step);
            } catch (Exception e) {
                Collections.reverse(completed);
                for (SagaStep s : completed) {
                    s.compensate(order);  // 역순 보상!
                }
                throw new OrderFailedException(e.getMessage());
            }
        }
    }
}
```

## 💡 한 줄 정리

> 메시지 큐는 만능이 아니며 **인프라 복잡도·디버깅·일관성·순서 보장** 비용이 따른다. 대규모 서비스는 **Saga 패턴**으로 각 단계에 "되돌리기 API"를 미리 만들어두고, 실패 시 **보상 메시지를 새로 발행**하여 역순으로 되돌린다. 핵심은 동기로, 나머지는 비동기로 처리하되, **DB 트랜잭션으로 해결 가능하면 그게 최선**이고 서비스 분리로 불가능할 때만 Saga를 택한다.
