# Day 015. 카카오톡 메시지는 어떻게 실시간으로 도착할까? — WebSocket과 SSE

> 🗓️ 2026-03-24  
> 🏷️ Tags: `WebSocket` `SSE` `실시간통신` `HTTP폴링` `SseEmitter` `WebFlux` `양방향통신`

---

## 🤔 질문

HTTP는 요청해야 응답이 오는데, 카톡은 안 물어봐도 메시지가 온다. 이게 어떻게 가능할까?  
WebSocket과 SSE는 뭐가 다르고, 각각 언제 써야 할까?  
SSE는 REST와 뭐가 다르며, Spring에서 어떻게 구현할까?

---

## 📌 핵심 요약

HTTP는 **요청-응답 모델**이라 서버가 먼저 보낼 수 없고, WebSocket은 **한 번 연결하면 양쪽이 자유롭게 메시지를 주고받는** 프로토콜이다.  
SSE는 **HTTP 그대로 서버→클라이언트 단방향 스트리밍**을 구현하며, 알림·대시보드·AI 스트리밍 응답 등 대부분의 실시간 기능에 WebSocket보다 적합하다.  
Spring MVC의 **SseEmitter만으로 추가 의존성 없이** SSE를 바로 사용할 수 있으며, 대규모에서는 WebFlux로 분리하는 하이브리드 방식이 현실적이다.

---

## 🔍 Part 1. HTTP의 한계 — 서버가 먼저 보낼 수 없다

### 1-1. HTTP 폴링의 문제

HTTP는 클라이언트가 물어봐야 서버가 답하는 구조다. 실시간 알림을 구현하려면 **계속 물어봐야(폴링)** 한다.

```
카카오톡이 HTTP 폴링이라면:
  매 1초마다: "새 메시지 있어?" → "없어"
  매 1초마다: "새 메시지 있어?" → "없어"
  ... 99번 ...
  매 1초마다: "새 메시지 있어?" → "1개 있어!"

사용자 1,000만 명 × 초당 1번 = 초당 1,000만 요청
→ 99%가 "없어" 응답 → 서버 자원 대량 낭비
```

비유하면, 폴링은 **매분마다 우체통 확인하러 나가기**(대부분 비어있음)이고, WebSocket/SSE는 **집 안에 초인종 설치**(편지 오면 우체부가 눌러줌)다.

---

## 🔍 Part 2. WebSocket — 양방향 실시간 통신

### 2-1. HTTP와의 핵심 차이

```
HTTP:      요청 → 응답 → 연결 끊김 → 또 요청 → 또 응답 → 끊김
WebSocket: 연결 (핸드셰이크) → 연결 유지 → 양쪽 자유롭게 메시지 교환
```

```
                    HTTP                WebSocket
통신 방향          단방향 (요청→응답)    양방향 (서로 자유롭게)
연결 유지          요청마다 연결/해제     한 번 연결하면 유지
서버→클라이언트    불가 (요청 없이)      가능!
헤더 오버헤드      매 요청 ~800바이트    프레임 2~14바이트
```

### 2-2. WebSocket 연결 과정 — HTTP로 시작한다

WebSocket은 HTTP 핸드셰이크로 시작해서 프로토콜을 전환한다. 기존 인프라(방화벽, ALB)가 HTTP를 이해하므로, HTTP로 문을 열고 들어간 뒤 전환하는 방식이다.

```
클라이언트: GET /chat (Upgrade: websocket)
서버:       101 Switching Protocols
→ 이후부터 WebSocket 프로토콜로 양방향 통신
```

### 2-3. WebSocket의 비용 문제

WebSocket의 상시 연결 유지에는 비용이 따른다.

**서버 메모리:** 연결당 약 10~15KB. 동시 100만 명이면 약 10GB.

**클라우드 과금:** AWS ALB는 연결 시간 기준으로 과금. 메시지가 없어도 연결만 유지하면 돈이 나감.

**Heartbeat 트래픽:** 연결이 살아있는지 확인하는 ping/pong을 30초~1분마다 보내야 함. 없으면 유령 연결이 메모리를 차지.

**Scale-Out 복잡도:** 서버가 여러 대면 사용자 연결이 분산되어, Redis Pub/Sub 같은 메시지 브로커로 서버 간 메시지를 연결해야 함.

### 2-4. 비용 최적화 전략

```
하이브리드:     양방향 필수인 기능만 WebSocket, 나머지는 SSE/HTTP
지연 연결:      채팅방 입장 시에만 WebSocket, 나가면 해제
타임아웃:       5분간 활동 없으면 연결 해제
서버 분리:      API 서버(MVC)와 WebSocket 서버를 별도 구성
```

## 🔍 Part 3. SSE — HTTP 그대로 단방향 스트리밍

### 3-1. SSE와 REST의 차이 — "응답을 끝내느냐 안 끝내느냐"

SSE는 새로운 프로토콜이 아니라 **같은 HTTP를 쓰는 방식이 다른 것**이다.

```
REST: 요청 1번 → 응답 1번 → 연결 종료 → 다시 요청해야 함
SSE:  요청 1번 → 응답 N번 → 연결 유지 → 서버가 계속 보냄
```

HTTP 레벨의 차이:

```
REST 응답:
  Content-Type: application/json
  Content-Length: 52              ← 크기 정해짐 → 끝이 있음
  {"message": "알림"}             ← 보내고 연결 종료!

SSE 응답:
  Content-Type: text/event-stream  ← 스트림!
  Transfer-Encoding: chunked       ← 크기 미정, 계속 보낼 거야
  
  data: {"message": "첫 번째 알림"}  ← 보내고
  data: {"message": "두 번째 알림"}  ← 또 보내고
  ...연결 유지하며 계속 전송
```

비유하면, REST는 **편지 한 통 쓰고 봉투 봉합**(끝)이고, SSE는 **봉투를 안 닫고 계속 종이를 넣는 것**(계속 추가 가능)이다.

---

### 3-2. SSE가 WebSocket보다 유리한 점

**HTTP 인프라 완벽 호환:** 프록시, 방화벽, CDN, ALB 추가 설정 없이 그대로 동작.

**자동 재연결:** 브라우저 내장 기능으로, 연결 끊기면 자동 재연결. Last-Event-ID로 끊긴 지점부터 이어받음. 개발자가 재연결 코드를 안 짜도 됨.

**구현이 간단:** 클라이언트 3줄이면 실시간 수신 가능. 별도 라이브러리 불필요.

```javascript
const es = new EventSource('/api/notifications/stream');
es.onmessage = (e) => {
    const data = JSON.parse(e.data);
    showNotification(data);  // 서버가 보낼 때마다 자동 호출!
};
```

---

### 3-3. SSE 연결 종료

```
클라이언트가 끊음 (close()):  페이지 이탈, 로그아웃 → 재연결 안 함
서버가 끊음 (complete()):    작업 완료, 타임아웃 → 브라우저 자동 재연결
네트워크 끊김:               지하철, Wi-Fi 전환 → 브라우저 자동 재연결

→ 클라이언트가 의도적으로 끊은 것만 재연결 안 함
→ 나머지는 브라우저가 알아서 재연결!
```

---

### 3-4. SSE의 한계

**단방향:** 서버→클라이언트만 가능. 클라이언트가 보내려면 별도 HTTP 요청.

**브라우저 연결 제한:** HTTP/1.1에서 같은 도메인 최대 6개. HTTP/2에서는 사실상 해소.

**텍스트만 전송:** 바이너리 데이터(이미지, 파일) 전송 불가. JSON 문자열은 가능.

---

### 3-5. SSE의 실무 활용 사례

```
ChatGPT/Claude 스트리밍 응답:
  → AI가 토큰 생성할 때마다 SSE로 전송 → 글자가 하나씩 나오는 효과

실시간 알림:
  → GitHub, Jira 등 서버 이벤트 발생 시 즉시 push

실시간 대시보드:
  → 서버 메트릭, 주가 등 데이터 변경 시 push

배송 추적:
  → 상태 변경 시 push → 새로고침 없이 실시간 업데이트
```

## 🔍 Part 4. 실시간 통신 방법 비교

```
                HTTP 폴링       롱 폴링        SSE              WebSocket
방향             단방향          단방향         단방향(서버→)      양방향
실시간성         낮음            중간           높음              높음
서버 부하        높음            중간           낮음              낮음
구현 복잡도      낮음            중간           낮음              높음
인프라 호환      완벽            완벽           완벽              일부 설정 필요
자동 재연결      직접 구현       직접 구현      브라우저 내장      직접 구현
바이너리 전송    가능            가능           불가              가능
```

```
선택 기준:

"서버→클라이언트 단방향이면 충분?" → SSE
"양방향 실시간 필수?" → WebSocket
"실시간 아니어도 돼?" → HTTP 폴링
"양방향인데 트래픽 적어?" → 롱 폴링
```

의외로 많은 "실시간" 기능이 단방향(SSE)으로 충분하다. 알림, 주가, AI 응답, 배송 추적 모두 서버가 일방적으로 보내면 되는 경우다.

## 🔍 Part 5. Spring에서의 SSE 구현

### 5-1. Spring MVC + SseEmitter (추가 의존성 없음!)

기존 Spring MVC 프로젝트에서 바로 사용 가능하다.

```kotlin
@RestController
class NotificationController {

    private val emitters = ConcurrentHashMap<String, SseEmitter>()

    @GetMapping("/api/notifications/stream")
    fun connect(@RequestParam userId: String): SseEmitter {
        val emitter = SseEmitter(60 * 60 * 1000L)
        emitters[userId] = emitter
        emitter.onCompletion { emitters.remove(userId) }
        emitter.onTimeout { emitters.remove(userId) }
        emitter.onError { emitters.remove(userId) }
        return emitter
    }

    fun sendToUser(userId: String, message: String) {
        emitters[userId]?.send(
            SseEmitter.event()
                .id(UUID.randomUUID().toString())
                .name("notification")
                .data(mapOf("message" to message))
        )
    }
}
```

장점은 추가 의존성 없이 바로 적용 가능한 점. 단점은 연결당 스레드를 점유하여 동시 연결 수백~수천 개가 한계다.

---

### 5-2. Spring WebFlux + Flux (대규모에 적합)

논블로킹 모델로 적은 스레드(수십 개)로 수만 연결을 처리할 수 있다.

```kotlin
@GetMapping(
    "/api/notifications/stream",
    produces = [MediaType.TEXT_EVENT_STREAM_VALUE]
)
fun streamNotifications(@RequestParam userId: String): Flux<ServerSentEvent<NotificationDto>> {
    return notificationService.getNotificationStream(userId)
        .map { notification ->
            ServerSentEvent.builder<NotificationDto>()
                .id(notification.id.toString())
                .event("notification")
                .data(notification)
                .build()
        }
}
```

장점은 스레드를 점유하지 않아 수만 연결이 가능한 점. 단점은 리액티브 학습 비용, 기존 MVC와 공존 불가, JPA와 궁합이 안 좋은 점(R2DBC 필요)이다.

---

### 5-3. 실무에서의 선택

```
기존 MVC + SSE 연결 수백 개 이하:
  → SseEmitter로 충분 (대부분 여기 해당)

기존 MVC + SSE 연결 수천 개 이상:
  → SseEmitter + 비동기 설정으로 버팀

대규모 실시간:
  → 하이브리드 방식이 현실적
    메인 API: Spring MVC (익숙하고 JPA 사용)
    실시간 서버: WebFlux로 별도 분리
    둘 사이: Kafka로 이벤트 연결
```

## 🔍 Part 6. WebSocket의 비용 vs SSE 비교

```
사용자 10만 명, 알림 기능 기준:

HTTP 폴링 (10초 간격):
  서버 부하: 높음 (초당 1만 요청), 대역폭 높음
  인프라: 단순, ALB 비용 중간

SSE:
  서버 부하: 낮음 (이벤트 있을 때만), 대역폭 낮음
  인프라: 단순 (HTTP 그대로), ALB 비용 중간

WebSocket:
  서버 부하: 낮음, 대역폭 가장 낮음
  인프라: 복잡 (Redis Pub/Sub 필요), ALB 비용 높음 (연결 시간 과금)

→ 비용: 폴링 < SSE < WebSocket
→ "전부 WebSocket"이 아니라 "이 기능에 정말 WebSocket이 필요한가?"가 핵심
```

---

## 💡 한 줄 정리

> WebSocket은 **양방향 실시간 통신**(채팅, 게임)에, SSE는 **서버→클라이언트 단방향 스트리밍**(알림, 대시보드, AI 응답)에 적합하다. SSE는 HTTP 그대로 동작하여 인프라 호환성·자동 재연결·구현 간결함에서 우위를 가지며, Spring MVC의 SseEmitter만으로 추가 의존성 없이 즉시 사용 가능하다. **"양방향이 필요한가?"** 만 판단하면 선택은 명확하다.
