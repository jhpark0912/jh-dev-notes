# Day 018. REST vs GraphQL vs gRPC — API 설계의 세 가지 철학

> 🗓️ 2026-03-30  
> 🏷️ Tags: `REST` `GraphQL` `gRPC` `protobuf` `API설계` `Over-fetching` `리졸버` `DataLoader`

---

## 🤔 질문

REST의 over-fetching/under-fetching 문제는 뭐고, GraphQL은 이걸 어떻게 해결할까?  
GraphQL 백엔드는 모든 조합을 미리 준비하는 건가? DB JOIN의 이점은 사라지는 건가?  
gRPC는 왜 내부 통신에서 강하고, 실무에서는 세 가지를 어떻게 조합할까?

---

## 📌 핵심 요약

REST는 **가장 보편적이고 단순한 API 설계**, GraphQL은 **클라이언트가 필요한 데이터를 정확히 골라가는** 설계, gRPC는 **바이너리 기반으로 빠르고 타입 안전한** 서비스 간 통신이다.  
GraphQL은 유연성을 얻는 대신 **DB JOIN의 효율을 잃는** 트레이드오프가 있으며, 대부분의 프로젝트는 REST로 충분하다.  
실무에서는 **외부(프론트)는 REST/GraphQL, 내부(서비스 간)는 gRPC**로 조합하는 것이 일반적 패턴이다.

---

## 🔍 Part 1. REST — 현재 API의 사실상 표준

### 1-1. 기본 구조

자원(Resource)을 URL로 표현하고, HTTP 메서드로 행위를 나타내는 설계 철학.

```
GET    /users          → 사용자 목록 조회
GET    /users/123      → 사용자 123번 조회
POST   /users          → 새 사용자 생성
PUT    /users/123      → 사용자 123번 수정
DELETE /users/123      → 사용자 123번 삭제
```

### 1-2. REST의 문제

**Over-fetching:** 모바일에서 사용자 이름만 필요한데, `GET /users/123` 하면 email, phone, address, profile 등 전체 데이터가 다 온다. 데이터 낭비.

**Under-fetching:** 게시글 상세 페이지에 글 + 작성자 + 댓글이 필요하면, `GET /posts/456` + `GET /users/123` + `GET /posts/456/comments`로 API 3번 호출. 네트워크 왕복 3번.

---

### 1-3. 실무에서 POST로 통일하는 것의 현실

복잡한 검색 조건을 GET 쿼리 파라미터로 표현하기 어렵고, PUT/PATCH 구분이 번거로워 POST로 통일하는 경우가 많다.

```
POST /orders/search (검색)
POST /users/123/update (수정)
→ 팀 규칙이 일관되면 실무적으로 문제없음
```

단, **단순 조회는 GET을 유지**하는 게 캐싱에 유리하다. GET은 브라우저·CDN이 자동 캐싱하지만, POST는 캐싱이 안 된다. "이론적 완벽함"보다 **"팀의 일관성"** 이 더 중요하다.

---

## 🔍 Part 2. GraphQL — 클라이언트가 데이터를 골라간다

### 2-1. 탄생 배경

Facebook이 2012년에 내부적으로 만들고 2015년에 오픈소스 공개. 뉴스피드 화면에 필요한 데이터를 REST로 가져오면 API 호출이 수십 번이었고, 모바일 네트워크에서 너무 느렸다.

### 2-2. 핵심: 클라이언트가 쿼리를 작성한다

```graphql
query {
  post(id: 456) {
    title
    author { name, profileImage }
    comments { content, author { name } }
  }
}
→ 요청한 필드만 정확히 응답! API 1번 호출로 전부 가져옴
```

REST에서는 화면마다 전용 API를 만들어야 했지만, GraphQL에서는 **프론트가 쿼리만 바꾸면 되고 백엔드 수정이 불필요**하다.

---

### 2-3. 백엔드 구조: 모든 조합이 아니라 "각 필드별 리졸버"

GraphQL 백엔드는 모든 파라미터 조합을 미리 처리하는 것이 아니라, **각 필드마다 "어떻게 가져오는지"를 정의한 리졸버**를 준비한다. 클라이언트가 요청한 필드의 리졸버만 실행된다.

```
클라이언트가 name만 요청:
  → user() 리졸버 실행
  → posts 리졸버 → 실행 안 됨! (요청 안 했으니까)

클라이언트가 name + posts + orderCount 요청:
  → user() 리졸버 + posts 리졸버 + orderCount 리졸버 실행
```

비유하면, REST는 **세트 메뉴**(고정 구성)이고 GraphQL은 **뷔페**(셰프별로 담당 메뉴를 준비해두고, 손님이 고른 것만 만듦)다.

---

### 2-4. N+1 문제와 DataLoader

사용자 10명의 글 목록을 요청하면, users 쿼리 1번 + 각 user의 posts 쿼리 10번 = 총 11번이 실행된다. **DataLoader**로 배치 처리하여 `WHERE author_id IN (1,2,3,...,10)` 쿼리 1번으로 합칠 수 있다.

---

### 2-5. GraphQL의 근본적 트레이드오프: DB JOIN 이점 상실

리졸버가 필드별로 독립적이라 **DB JOIN을 자연스럽게 활용할 수 없다.** REST에서 JOIN 1번이면 될 걸 GraphQL에서는 리졸버 3개, SQL 3번으로 처리하게 된다.

**완화 방법들:**

**DataLoader:** N+1을 배치로 줄이지만, 여전히 쿼리 여러 번. JOIN 1번의 효율에는 못 미침.

**Look-Ahead:** 클라이언트가 요청한 필드를 미리 확인하여 필요한 JOIN을 동적으로 구성. 리졸버 코드가 복잡해지는 단점.

**하이브리드:** 자주 함께 요청되는 관계(post+author)는 JOIN으로, 가끔 요청되는 관계(comments)는 별도 리졸버로. 현실적 타협.

---

### 2-6. 속도 비교: REST vs GraphQL

```
단순 조회 (DB가 병목):
  REST: JOIN 1번 → 빠름
  GraphQL: 리졸버 여러 번 → 느림
  → REST 승

복잡한 화면 (네트워크가 병목):
  REST: API 4번 호출, 왕복 4번 → 느림
  GraphQL: API 1번 호출, 왕복 1번 → 빠름
  → GraphQL 승

모바일/느린 네트워크:
  왕복 1번 vs 4번 차이가 극적 (3G에서 200ms vs 800ms)
  → GraphQL이 유리
```

대부분의 웹 서비스에서 서버-클라이언트가 같은 클라우드에 있으면 네트워크 왕복이 빠르므로 DB 효율이 더 중요해 REST가 유리한 경우가 많다. 모바일·글로벌 서비스에서는 GraphQL의 왕복 최소화가 가치 있다.

### 2-7. GraphQL의 기타 단점

**복잡한 쿼리 공격:** 클라이언트가 무한 중첩 쿼리를 보내면 서버 과부하. 쿼리 깊이/복잡도 제한을 별도 구현해야 함.

**캐싱 어려움:** 모든 요청이 POST /graphql로 같은 URL. URL 기반 캐싱(CDN, 브라우저)이 안 됨.

**학습 비용:** 스키마, 리졸버, N+1 해결 등 REST보다 초기 설정이 복잡.

## 🔍 Part 3. gRPC — 서비스 간 통신의 최적해

### 3-1. 핵심 특징

Google이 내부 Stubby를 오픈소스화. **protobuf(바이너리)** 를 전송 포맷으로, **HTTP/2**를 필수로 사용.

.proto 파일로 서비스 계약을 정의하면 Java, Go, Python 등의 코드가 자동 생성된다. 서버와 클라이언트가 같은 .proto를 공유하므로 **타입 불일치가 컴파일 타임에 잡힌다.**

### 3-2. 성능

전송 크기: REST+JSON 대비 약 3~10배 작음 (바이너리). 직렬화 속도: 약 5~10배 빠름. HTTP/2 멀티플렉싱으로 하나의 연결에서 여러 요청을 동시 처리.

### 3-3. 4가지 통신 패턴

```
Unary:                요청 1개 → 응답 1개 (REST와 비슷)
Server Streaming:     요청 1개 → 응답 여러 개 (SSE와 비슷)
Client Streaming:     요청 여러 개 → 응답 1개 (파일 업로드 등)
Bidirectional:        양쪽 다 스트림 (WebSocket과 비슷)
```

### 3-4. 단점

**브라우저 직접 호출 불가:** grpc-web 프록시 필요. 프론트엔드 통신에 부적합.

**디버깅 어려움:** curl로 호출 불가. 바이너리라 눈으로 읽을 수 없음. 전용 도구 필요.

**생태계:** REST보다 지원 도구가 제한적.

## 🔍 Part 4. 세 가지 비교

```
              REST              GraphQL           gRPC
전송 포맷     JSON (텍스트)      JSON (텍스트)      protobuf (바이너리)
프로토콜     HTTP/1.1 or 2      HTTP               HTTP/2
속도         보통               상황에 따라         빠름
스트리밍     제한적             Subscription       네이티브 (4가지)
캐싱         쉬움 (URL 기반)    어려움              어려움
브라우저     완벽 지원           완벽 지원           제한적
디버깅       쉬움 (curl)        중간               어려움
타입 안전    약함               중간 (스키마)       강함 (.proto)
학습 비용    낮음               중간               중간
```

## 🔍 Part 5. 실무에서의 조합

```
┌──────────┐     REST or GraphQL      ┌──────────┐
│ 브라우저   │ ←──────────────────────→ │ API GW   │
│ 모바일 앱  │   (JSON, 사람 친화적)     │          │
└──────────┘                          └────┬─────┘
                                           │ gRPC (빠르고 효율적)
                                    ┌──────┼──────┐
                                    │      │      │
                                 주문    결제    재고
```

외부(프론트): REST 또는 GraphQL. 내부(서비스 간): gRPC. 한 가지만 쓰는 게 아니라 **용도별로 조합**한다.

### 선택 기준

```
단순한 웹 서비스 → REST (대부분 여기 해당)
복잡한 화면, 클라이언트 다양 → GraphQL (프론트 자유도가 절실할 때)
MSA 서비스 간 내부 통신 → gRPC (빠르고 타입 안전)
```

대부분의 프로젝트는 REST로 충분하며, GraphQL은 클라이언트 다양성과 프론트 자유도가 절실할 때만 가치가 있다.

---

## 💡 한 줄 정리

> REST는 **JOIN 활용과 캐싱에서 강하고**, GraphQL은 **네트워크 왕복 최소화와 프론트 자유도에서 강하고**, gRPC는 **바이너리 속도와 타입 안전에서 강하다.** "어느 것이 최고"가 아니라 **DB 병목인가 네트워크 병목인가, 외부 통신인가 내부 통신인가**에 따라 선택하며, 대부분의 프로젝트는 REST로 충분하고 필요할 때 GraphQL과 gRPC를 조합한다.
