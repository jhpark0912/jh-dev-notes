# Day 024. 정규표현식은 어떻게 모든 언어에 침투했나

> 2026-04-15 #정규표현식 #regex #ReDoS #PCRE #RE2 #HTML파싱

## 🤔 질문
정규표현식은 누가 만들었고 어떤 원칙으로 설계됐을까? 그리고 왜 가끔 서버를 죽이며, 왜 HTML 파싱에는 절대 쓰지 말라고 할까?

## 📌 핵심 요약
- 정규식은 1943년 신경과학자들이 뇌를 수학으로 모델링한 데서 출발해, 1956년 클레이니가 단 3개의 연산자로 무한한 패턴을 표현하는 표기법으로 정립했다.
- Perl이 편의성을 위해 클레이니의 "기억하지 말자" 원칙을 어기면서 ReDoS라는 폭탄이 심어졌고, 30년 뒤 Cloudflare 27분 장애로 폭발했다.
- 정규식의 진짜 한계는 "유한한 상태"라는 본질에 있고, HTML 같은 무한 중첩 구조는 차원이 다른 문제라 파서로만 다룰 수 있다.

## 🔍 상세 설명

### Part 1. 기원 — 노숙자 천재와 뇌 박사

정규식의 출발점은 컴퓨터가 아니라 **뇌 연구실**이다.

1943년 시카고. **월터 피츠**라는 가출 청소년이 있었다. 12살 때 도서관에서 버트런드 러셀의 《수학 원리》를 읽고 저자에게 직접 오류를 지적하는 편지를 보낸 천재로, 정규 교육 없이 시카고 대학 강의를 청강하며 노숙 생활을 했다. 그가 신경과학자 **워렌 매컬러**를 만나 함께 던진 질문이 모든 것의 시작이다.

> "뉴런이 켜짐/꺼짐의 단순 단위라면, 이건 그냥 0과 1 아닌가? 논리 회로랑 똑같잖아?"

두 사람은 1943년 논문에서 **뉴런을 논리 단위로 모델링**했다. 이게 인공 신경망의 시조이자 **유한 오토마톤**의 출발점이다. 정규식의 직계 조상.

### Part 2. 클레이니의 3대 발견 (1956)

13년 후, 수학자 **스티븐 클레이니**가 매컬러-피츠 논문을 보고 "뇌가 만드는 모든 패턴을 종이에 적을 방법이 있을까?"라는 질문을 던진다. 그가 발견한 답은 충격적이었다 — **딱 3개의 기호만 있으면 충분하다.**

```
1. 연결      ab    "a 다음에 b"
2. 선택      a|b   "a 또는 b"  
3. 반복      a*    "a를 0번 이상"
```

이게 **regular expression**이라는 이름의 시작이다. 비유하자면 클레이니는 **레고의 기본 블록 3종**을 발견한 것이다 — 이 셋만 있으면 어떤 구조든 만들어낼 수 있다.

이때까지 정규식은 순수 수학 기호였다. 컴퓨터로 뭘 한다는 개념도 없었다.

### Part 3. 톰슨이 코드로 옮기다 (1968)

또 12년 후, 벨 연구소의 **켄 톰슨**(나중에 Unix와 C 언어를 만든 그 톰슨)이 텍스트 에디터를 만들다 클레이니의 정규식을 발견한다. 그는 종이 위 기호를 **실제 동작하는 알고리즘**으로 옮겼다 — 컴퓨터 역사상 정규식이 처음 코드가 된 순간이다.

톰슨의 알고리즘은 NFA(비결정성 유한 오토마톤) 방식으로, 입력 길이에 비례하는 시간만 걸렸다. **절대 폭발하지 않았다.**

이 톰슨의 정규식이 Unix 생태계로 퍼지며 익숙한 도구들이 등장한다.

```
1968  QED      톰슨이 정규식 첫 구현
1971  ed       Unix 표준 에디터
1973  grep     "global / regular expression / print"의 약자
1974  sed      스트림 편집기
1977  awk      텍스트 처리 언어
```

매일 쓰는 `grep`이 사실 정규식 명령어("global regular expression print")의 줄임말이다.

### Part 4. 정규식의 두 가지 원칙

여기까지의 역사로 정규식의 본질이 정의된다.

**원칙 1: 최소한의 도구로 최대한을 표현한다.**
3개 연산자만으로 표현 가능한 모든 것이 "정규 언어(regular language)"다. 이 한계 때문에 HTML 같은 중첩 구조는 정규식으로 표현할 수 없다 — 수학적 한계다.

**원칙 2: 기억하지 말고 흘려보낸다.**
유한 오토마톤은 현재 상태만 기억하고 한 글자씩 흘려보내며 처리한다. 과거 입력을 저장하지 않는다.

> 정규식은 **공항 입국 심사관**이다. 한 사람씩 차례로 보고 즉시 통과/거부 판단. 앞에 누가 있었는지 기억하지 않는다. 빠르고 효율적이지만, "오늘 들어온 사람 중 같은 이름이 있나?" 같은 건 못 한다.

이 "기억 안 함"이 속도의 비결이자 한계다.

### Part 5. Perl의 욕심과 사고의 씨앗

1987년, 언어학자 출신 프로그래머 **Larry Wall**이 Perl을 만들면서 정규식을 언어 문법에 1급 시민으로 박아넣었다. 90년대 웹 개발자들이 Perl로 몰린 핵심 이유였다.

근데 Larry는 클레이니의 깔끔한 3개 연산자로는 만족하지 못했다. **편의성을 위해 두 가지 위험한 기능**을 추가한다.

**역참조(Backreference)**
```perl
/(\w+)\s+\1/     # 같은 단어 두 번 반복 ("the the")
```

**전후방 탐색(Lookahead/Lookbehind)**
```perl
/^(?=.*[A-Z])(?=.*\d).{8,}$/   # 비밀번호 검증
```

이 기능들은 정말 편했지만, 클레이니의 "기억하지 말자" 원칙을 정면으로 위반한다. 1997년 PCRE(Perl Compatible Regular Expressions)로 표준화되며 PHP, Python, Ruby, Java, JavaScript 등 모든 언어로 퍼졌다.

문제는 이 기능을 구현하려면 톰슨의 NFA로는 불가능해서 **백트래킹(Backtracking)** 방식으로 엔진을 갈아엎어야 했다는 것이다.

> 톰슨 NFA는 **여러 명이 동시에 미로를 탐색**하는 것과 같다. 갈림길에서 인원을 나눠 모든 길을 동시에 확인 — 시간 일정.
> 백트래킹은 **혼자 미로를 탐색**하는 것과 같다. 한 길을 끝까지 가보고 막히면 돌아와서 다른 길 시도 — 운 나쁘면 모든 조합을 다 시도.

### Part 6. ReDoS와 Cloudflare 사고

백트래킹 엔진의 치명적 약점이 **Catastrophic Backtracking**이다.

```
정규식:  (a+)+b
입력:    aaaaaaaaaaaaaaaaaaaaaaaaaaaaa  (a만 30개, b 없음)

(a+)+로 a들을 그룹핑하는 방법: 약 2^(n-1) 가지
a 30개 → 약 5억 가지 조합 시도
a 40개 → 약 5천억 가지 (17분 소요)
a 50개 → 약 12일 소요
```

이 취약점을 **ReDoS(Regular expression Denial of Service)**라 부른다. ReDoS의 위험성은 1990년대부터 학계에 알려져 있었고, 2007년 Russ Cox가 〈Regular Expression Matching Can Be Simple And Fast〉라는 글에서 강하게 경고하며 화제가 됐다. Google은 이 위험을 인지하고 **2010년 RE2 엔진을 선제적으로 발표**했다 — 표현력을 일부 포기하고 모든 패턴이 선형 시간을 보장하는 엔진이다.

그리고 **2019년 7월 2일**, RE2 출시 9년 후 가장 유명한 ReDoS 사고가 터진다.

```
Cloudflare가 27분간 전 세계적으로 다운
원인: WAF 규칙에 추가된 정규식 한 줄
     (.* 가 여러 번 중첩된 패턴)
영향: Discord, Medium 등 수많은 서비스가 동시 마비
```

Cloudflare CTO의 사후 분석: "우리는 위험을 알고 있었지만, 시스템에 안전장치가 없었다." 이 사고가 ReDoS를 대중에게 알린 결정적 계기가 됐다.

```
시간선
1990s   ReDoS 개념 학계에서 정립
2007    Russ Cox의 경고 글 발표
2010    Google RE2 출시 (선제적 대응)
2016    Stack Overflow 34분 다운 (예고편)
2019    Cloudflare 27분 다운 (대형 사고)
```

### Part 7. 두 진영의 공존

PCRE와 RE2는 서로 대체하는 게 아니라 **각자의 영역에서 공존**한다.

| 진영 | 사용처 | 특징 |
|---|---|---|
| **PCRE 호환** | PHP, Python, Ruby, Java, JS, .NET | 표현력 ↑, ReDoS 위험 |
| **RE2 스타일** | Go, Rust, Python google-re2 | 안전 보장, 표현력 일부 제한 |

신규 언어인 Go와 Rust는 호환성 부담이 없어서 처음부터 RE2 스타일로 만들어졌다.

**PCRE가 안 사라지는 이유**:
- 역참조와 lookahead가 실무에 정말 자주 쓰임 (비밀번호 검증, 중복 단어 검출 등)
- 대부분의 실무 정규식은 안전한 패턴 (사고는 복잡한 패턴에서 발생)
- 25~30년치 코드와의 호환성 (PHP의 `preg_match`는 1999년부터)
- 타임아웃, 정적 분석 등 주변 방어책 발달

비유: **플라스틱 빨대**와 비슷하다. 위험을 알지만 너무 많이 쓰고 있어 한 번에 못 바꾸고, 종이 빨대(RE2)와 병행하며 위험한 곳부터 교체하는 중.

### Part 8. IDE와 도구의 안전망

요즘은 ReDoS를 미리 잡아주는 도구들이 발달해 있다.

```
1. 개발 중   IntelliJ + SonarLint
              → 위험 패턴 즉시 노란 줄 경고

2. 커밋 전    pre-commit hook + safe-regex
              → 위험 패턴 커밋 차단

3. PR 단계    SonarQube / CodeQL / eslint-plugin-redos
              → 자동 코드 리뷰

4. 런타임     .NET, Java의 정규식 타임아웃
              → 폭발해도 1초 만에 끊기
```

**Cloudflare 사고도 이 중 어느 단계에서라도 잡혔으면 막을 수 있었다.** 사후 분석에서 가장 후회한 부분이 "정적 분석 도구를 안 쓰고 있었다"였다.

다만 IntelliJ 기본 기능에는 ReDoS 감지가 없고, SonarLint 같은 별도 플러그인이 필요하다. 자동차의 안전벨트는 기본이지만 에어백·ABS는 옵션인 것과 비슷하다.

### Part 9. HTML 파싱 금지의 진짜 이유

정규식의 가장 유명한 격언인 **"HTML 파싱하지 마"**는 격언이 아니라 **수학적 사실**이다.

언어학자 노엄 촘스키가 1956년에 정의한 **촘스키 위계**:

```
Type 3: 정규 언어        ← 정규식이 다룰 수 있는 영역
Type 2: 문맥 자유 언어   ← HTML, XML, 프로그래밍 언어
Type 1: 문맥 의존 언어
Type 0: 무제한 언어      ← 자연어
```

정규식은 Type 3까지만 다룰 수 있고, HTML은 Type 2다. **계급이 다르다.**

> 자(ruler)로는 길이를 잴 수 있지만 부피는 못 잰다. 자에 눈금을 더 촘촘하게 한다고 부피를 잴 수 있게 되진 않는다. **차원이 다른 도구가 필요하다.**

핵심 이유는 **중첩**이다. HTML의 본질은 태그 짝 맞추기인데, 이건 사실상 괄호 매칭과 같다.

```
<div><p><span>안녕</span></p></div>
   = ( ( (    안녕    )   )    )
```

정규식이 "지금까지 괄호를 몇 개 열었는지" 세려면 무한한 상태가 필요한데, 유한 오토마톤은 이름 그대로 **유한한 상태**만 가능하다. 무한 중첩을 다룰 수 없다.

### Part 10. 그럼에도 HTML 파싱이 필요한 이유

HTML 파싱은 인터넷 인프라의 핵심이다.

| 영역 | 사례 |
|---|---|
| 일상 | 모든 브라우저(Chrome, Safari)의 렌더링 엔진 |
| 검색 | Google, Naver의 크롤러 |
| 상거래 | 가격 비교 사이트, 부동산 시세 비교 |
| 자동화 | Selenium, Playwright, RPA 도구 |
| 보안 | XSS 방어를 위한 DOMPurify |
| AI | ChatGPT/Claude 학습용 Common Crawl |

특히 **보안 분야에서 정규식으로 HTML을 처리하면 거의 100% 뚫린다.** 공격자가 `<img src=x onerror=alert(1)>`, `<svg><script>` 같은 별별 우회를 시도하기 때문이다.

### Part 11. 파서와 정규식은 협업자

오해를 정리하자면, "HTML 파싱"은 **HTML 구조 자체를 다루는 것**이지 HTML 안의 텍스트 내용을 다루는 게 아니다.

```
구조 분석 (정규식 ❌)
"이 페이지의 모든 <a> 태그의 href를 가져와"
"<div class='price'> 안의 가격 추출"
→ 파서 필요 (BeautifulSoup, cheerio, Jsoup)

텍스트 패턴 (정규식 ✅)
"본문 텍스트에서 이메일만 뽑기"
"가격 문자열에서 숫자만 추출"
→ 정규식 OK
```

**현장의 황금 패턴**:

```python
# 1. HTML 받기
html = requests.get(url).text

# 2. 구조 분석은 파서로
soup = BeautifulSoup(html, 'html.parser')
price_text = soup.find('span', class_='price').get_text()
# → "￦ 29,900원 (할인가)"

# 3. 텍스트 안의 패턴은 정규식으로
price = int(re.search(r'[\d,]+', price_text).group().replace(',', ''))
# → 29900
```

파서와 정규식은 경쟁자가 아니라 **각자의 영역에서 협업하는 도구**다.

> "운전하지 마"가 아니라 "면허 따고 운전해"인 것처럼, "HTML 파싱하지 마"가 아니라 **"HTML 파싱은 정규식 말고 진짜 파서로 해라"**가 정확한 메시지다.

## 💡 한 줄 정리
> 정규식의 역사는 "단순함의 발견 → 편의성의 욕심 → 사고 → 다시 단순함으로의 회귀"의 순환이며, 그 한계와 강점을 이해하고 파서와 협업시킬 때 비로소 안전하고 강력한 도구가 된다.
