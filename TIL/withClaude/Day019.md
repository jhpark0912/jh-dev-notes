# Day 019. 오픈소스는 어떻게 돈을 벌까? — 비즈니스 모델과 라이선스 전쟁

> 🗓️ 2026-03-31  
> 🏷️ Tags: `오픈소스` `비즈니스모델` `라이선스` `SSPL` `AWS` `포크` `오픈코어` `SaaS`

---

## 🤔 질문

Linux, Redis, Kafka, Docker가 무료인데 어떻게 회사가 유지될까?  
AWS가 오픈소스를 가져다 돈을 버는 것은 횡포인가, 정당한 사업인가?  
라이선스 변경은 왜 매번 실패하고, 오픈소스의 지속 가능성은 어떻게 확보하는가?

---

## 📌 핵심 요약

오픈소스는 **소프트웨어 자체가 아니라 그 위의 서비스(관리형 클라우드, 기술 지원, 고급 기능)로 수익**을 낸다.  
대기업은 **생태계 장악과 클라우드 유도** 전략으로 오픈소스에 투자하며, AWS의 "가져다 팔기" 논쟁은 2018년부터 반복되는 패턴이다.  
라이선스를 제한적으로 변경한 회사는 **커뮤니티 이탈과 포크로 오히려 손해**를 봤으며, **독립 재단 거버넌스**를 가진 프로젝트만이 이 문제에 면역이다.

---

## 🔍 Part 1. 오픈소스 비즈니스 모델 5가지

### 1-1. 오픈코어 (Open Core)

핵심 기능은 무료(오픈소스), 고급 기능은 유료(상용 라이선스). 게임의 부분 유료화와 같은 구조.

**GitLab:** Community Edition은 무료(Git 저장소, CI/CD 기본). Enterprise Edition은 유료(보안 스캐닝, 규정 준수, 우선 지원). Premium $29/user/월, Ultimate $99/user/월.

**기타:** Redis(핵심 무료, Enterprise 유료), Elasticsearch(검색 무료, ML/보안 유료), Grafana(대시보드 무료, Cloud 유료), MySQL(Community 무료, Enterprise 유료).

---

### 1-2. 클라우드 서비스 (SaaS/Managed Service)

"소프트웨어는 무료, 운영은 유료." 직접 설치하면 공짜지만, 설치·설정·모니터링·백업·업그레이드를 직접 해야 한다. 관리형 서비스를 쓰면 클릭 한 번에 전부 해준다.

**Confluent (Kafka):** Apache Kafka는 무료 오픈소스. Kafka 클러스터 운영은 악명 높게 복잡하여 전담 인력 2~3명 필요. Confluent Cloud는 클릭 몇 번으로 Kafka 생성, 모니터링·스케일링·백업 자동. Confluent 매출 약 1.2조 원.

**MongoDB Atlas:** MongoDB 무료 설치 가능. Atlas 관리형 서비스로 매출의 70% 이상. MongoDB Inc. 매출 약 2.7조 원.

---

### 1-3. 기술 지원 + 컨설팅

소프트웨어는 무료, 전문가 도움은 유료. 법전은 누구나 무료로 볼 수 있지만, 변호사에게 해석을 맡기면 유료인 것과 같다.

**Red Hat (Linux):** Linux 커널은 무료. RHEL은 기업용 안정성 보증(10년 장기 지원), 24/7 기술 지원, 규정 준수 인증을 유료 구독으로 제공. 매출 약 5.3조 원. 2019년 IBM이 45조 원에 인수. 무료 Linux로 45조 원짜리 회사를 만든 것.

---

### 1-4. 대기업의 전략적 투자

Day 010에서 다뤘던 Google의 Let's Encrypt 후원과 같은 논리.

**Google → Kubernetes:** K8s가 컨테이너 표준이 됨 → K8s를 쓸수록 클라우드 필요 → GCP에서 K8s를 가장 잘 지원 → K8s 사용자가 GCP 고객으로 전환. K8s 개발에 수백억 투자, GCP 매출 약 53조 원.

**Google → Android:** 스마트폰 OS 72% 점유 → Android 사용자가 Google 검색/Play Store 사용 → 광고 수익. 무료 제공으로 제조사 채택을 유도하여 Google 생태계 확장.

**Meta → React, PyTorch:** React가 프론트엔드 표준이 되면 Meta 기술 스택에 친숙한 개발자 풀이 확대되어 채용이 쉬워진다. PyTorch가 AI 연구 표준이 되면 최신 연구를 Meta AI 팀이 바로 활용 가능.

**대기업 전략 요약:** 생태계 장악(표준화), 인재 풀 확대, 클라우드 유도, 개발 비용 분산(커뮤니티 기여).

---

### 1-5. 듀얼 라이선스

같은 소프트웨어를 무료 라이선스(GPL 등, 소스코드 공개 의무)와 상용 라이선스(유료, 공개 의무 없음)로 동시 제공. "무료로 쓸 수 있지만 네 코드도 공개해야 해. 공개 싫으면 유료 라이선스를 사."

**MySQL:** Community(GPL, 무료, 포함 배포 시 소스 공개 의무), Enterprise(상용, 유료, 공개 의무 없음).

## 🔍 Part 2. 오픈소스 라이선스 전쟁 — AWS와의 충돌

### 2-1. 전쟁의 시작: MongoDB vs AWS (2018)

MongoDB가 오픈소스(AGPL)로 성공하자 AWS가 "Amazon DocumentDB"를 출시. MongoDB 호환 API를 그대로 제공하며 관리형 서비스로 수익을 올렸지만 MongoDB에 돌아오는 건 없었다.

MongoDB가 2018년 10월 라이선스를 SSPL로 변경. "클라우드 서비스로 제공하려면 전체 인프라 코드를 공개해!" — 사실상 AWS를 겨냥. 이것이 오픈소스 라이선스 전쟁의 시작점이 되었다.

---

### 2-2. Elasticsearch vs AWS (2021)

AWS가 "Amazon Elasticsearch Service"를 출시하며 Elastic의 상표권인 "Elasticsearch"를 서비스명에 그대로 사용. Elastic이 2021년 1월 SSPL + Elastic License 듀얼 라이선스로 변경하자, AWS가 마지막 오픈소스 버전을 포크하여 "OpenSearch"를 만들었다.

OpenSearch가 성공적으로 성장하면서 Elastic은 2024년 8월 AGPL v3를 추가하며 오픈소스로 부분 복귀했다.

---

### 2-3. HashiCorp(Terraform) vs OpenTofu (2023)

HashiCorp가 2023년 8월 Terraform의 라이선스를 MPL 2.0(오픈소스)에서 BSL(Business Source License)로 변경. "경쟁 서비스를 만드는 데 사용 금지."

커뮤니티 반발이 가장 격렬한 사례였다. 수만 명의 기여자가 함께 만든 프로젝트를 한 회사가 일방적으로 라이선스를 바꿨기 때문. Linux Foundation 지원 하에 OpenTofu 포크가 등장했고, IBM이 2025년 2월 HashiCorp를 약 8.5조 원에 인수했다.

---

### 2-4. Redis vs Valkey (2024)

Redis가 2024년 3월 BSD(매우 자유로운 라이선스)에서 SSPL/RSALv2로 변경. "더 이상 오픈소스가 아님." 30일 만에 Linux Foundation이 AWS, Google 등의 지원을 받아 "Valkey" 포크를 발표했고, 1년 만에 약 50개 회사가 기여하는 프로젝트로 성장. 대기업의 83%가 Valkey를 도입하거나 테스트 중이었다.

Redis CEO는 제한적 라이선스가 커뮤니티와의 관계를 손상시켰다고 공개적으로 인정하며, 2025년 5월 AGPL v3를 추가해 부분 복귀했다.

---

### 2-5. 반복되는 패턴

이 패턴은 매번 동일한 5단계를 거친다.

```
1단계: 오픈소스로 성공 → 사실상 표준이 됨
2단계: 클라우드 업체가 관리형 서비스로 수익
3단계: 원작 회사가 라이선스를 제한적으로 변경
4단계: 커뮤니티 반발 → 포크 등장
5단계: 포크가 성장 → 원작 회사가 다시 오픈소스로 복귀 시도
```

공통 요소는 단일 벤더 통제 + 클라우드 경쟁 + 벤처캐피털 압박이며, 업계 분석에 따르면 라이선스 제한이 매출을 개선했다는 증거는 없다.

## 🔍 Part 3. 라이선스 전쟁에 면역인 프로젝트

독립 재단이 관리하는 프로젝트는 특정 회사가 라이선스를 일방적으로 바꿀 수 없다.

```
면역인 프로젝트:
  PostgreSQL:  PostgreSQL Global Development Group 관리
  Kubernetes:  Google이 만들었지만 CNCF에 기증
  Linux:       Linus Torvalds + Linux Foundation
  Apache Kafka: Apache Software Foundation

라이선스를 바꾼 프로젝트 (전부 단일 회사 소유):
  MongoDB → MongoDB Inc.
  Elastic → Elastic N.V.
  HashiCorp → HashiCorp Inc.
  Redis → Redis Inc.
```

## 🔍 Part 4. "대기업 횡포"만은 아니다 — 다양한 관점

### 관점 1: 원작자 입장

"수년간 만든 소프트웨어를 AWS가 가져다 수조 원을 버는데 돌아오는 건 없다." 분명 불공정하게 느껴진다.

### 관점 2: 오픈소스 철학

"오픈소스는 누구든 어떤 목적으로든 자유롭게 사용 가능해야 한다. 대기업이든 개인이든 차별하면 오픈소스가 아니다." AWS가 쓰는 것도 이 자유의 일부.

### 관점 3: 양쪽 다 이득을 봤다

MongoDB, Elastic, Redis가 "표준"이 된 건 오픈소스였기 때문이다. 무료니까 모두가 썼고, 생태계가 형성되고, 표준이 됐다. 처음부터 유료였으면 이 규모의 성장은 불가능했다. MongoDB 시가총액 23조 원, Elastic 11조 원은 오픈소스로 시작했기에 가능했다.

### 관점 4: 라이선스 변경이 오히려 역효과

라이선스를 막으면 커뮤니티가 포크로 떠나고, 대기업이 포크를 지원하면서 오히려 더 강해진다. 원작 회사만 손해를 보는 결과.

### 관점 5: 구조적 문제

"무료로 제공하면서 돈을 벌어라" 자체가 어려운 과제. 대기업의 횡포라기보다 **오픈소스 비즈니스 모델의 구조적 한계**. Red Hat(기술 지원)이나 Confluent(관리형 서비스 차별화)처럼 "AWS보다 우리가 더 잘하는 것"을 증명한 회사만이 성공한다.

## 🔍 Part 5. 개발자가 알아야 할 라이선스 기본

```
MIT / Apache 2.0:    거의 제한 없음, 상용 포함 OK. 가장 편한 라이선스.
GPL:                사용 가능하지만, 포함 배포 시 소스코드 공개 의무. 상용 제품 주의.
AGPL:               네트워크 서비스로 제공해도 소스코드 공개 의무. 가장 주의.
SSPL / BSL:         클라우드 서비스로 제공 시 제한. 직접 사용은 보통 가능.
```

실무에서 새 라이브러리 도입 시 라이선스 확인은 필수. 단일 회사 소유 프로젝트는 라이선스 변경 리스크가 있으므로, 재단 관리 프로젝트(PostgreSQL, Kubernetes 등)가 더 안전한 선택이다.

## 💡 한 줄 정리

> 오픈소스 라이선스 충돌은 **MongoDB(2018) → Elastic(2021) → HashiCorp(2023) → Redis(2024)**로 반복되는 패턴이며, 라이선스 제한은 매번 커뮤니티 포크와 원작 회사의 손해로 귀결된다. "대기업 횡포"만이 아니라 **오픈소스 비즈니스 모델의 구조적 긴장**이며, **독립 재단 거버넌스**를 가진 프로젝트만이 이 문제에 면역이다. 근본 해결은 라이선스 변경이 아니라 **더 나은 비즈니스 모델 설계**에 있다.
