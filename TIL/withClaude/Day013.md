# Day 013. 넷플릭스는 서버가 터져도 왜 안 죽을까? — 장애 격리와 서킷 브레이커

## 🤔 질문

마이크로서비스에서 하나의 서비스가 죽으면 왜 전체가 같이 죽을까?  
서킷 브레이커는 어떻게 장애를 자동으로 감지하고 격리하며, 모놀리식에서도 쓸 수 있을까?  
Spring Boot Actuator는 뭐고, 실무에서 얼마나 쓰일까?

---

## 📌 핵심 요약

서킷 브레이커는 **장애가 발생한 서비스로의 요청을 자동 차단**하여 연쇄 장애를 방지하는 패턴이다.  
MSA 전용이 아니라 **네트워크를 타는 외부 호출이 있으면 어디든** 필요하며, Actuator + Prometheus + Grafana로 **자동 감지 → 자동 대응 → 자동 알림** 파이프라인을 구축한다.  
넷플릭스는 **Chaos Monkey로 장애를 일상화**하여 어떤 장애에도 살아남는 시스템을 만들었다.

---

## 🔍 Part 1. 장애 전파(Cascading Failure)

### 1-1. 하나가 죽으면 전체가 죽는 문제

```
쇼핑몰 서비스:
  주문 서비스 → 결제 서비스 → 재고 서비스
             → 알림 서비스 → 이메일 서비스

알림 서비스가 응답을 안 하면:
  1. 주문 서비스가 알림 응답 대기 (5초 타임아웃)
  2. 주문이 몰리면 → 스레드 전부 알림 대기 중
  3. 주문 서비스도 새 요청 못 받음 → 주문 장애!
  4. 결제 서비스도 주문 응답 대기 → 결제도 장애!
  5. 전체 시스템 다운!

→ 알림 하나 때문에 결제까지 죽음
```

비유하면, 아파트 한 집에서 수도관이 터졌는데 건물 전체 수도를 차단해야 하는 상황이다. 이상적으로는 그 집만 차단하고 나머지는 정상 사용해야 한다.

## 🔍 Part 2. 서킷 브레이커 — 전기 차단기에서 영감

### 2-1. 3가지 상태

```
CLOSED (정상): 모든 요청 통과, 실패율 모니터링
  → 실패율 임계값 초과 시 OPEN으로 전환

OPEN (차단): 요청을 아예 보내지 않음, 즉시 fallback 반환
  → 일정 시간 후 HALF-OPEN으로 전환

HALF-OPEN (반열림): 요청 몇 개만 시험적으로 전송
  → 성공 → CLOSED 복구
  → 실패 → 다시 OPEN
```

### 2-2. 실제 코드 (Resilience4j)

```java
@CircuitBreaker(
    name = "notificationService",
    fallbackMethod = "sendNotificationFallback"
)
public void sendNotification(String userId, String message) {
    notificationClient.send(userId, message);
}

public void sendNotificationFallback(String userId, String message, Exception e) {
    log.warn("알림 서비스 장애, 나중에 재시도");
    retryQueue.add(new NotificationRetry(userId, message));
}
```

```yaml
resilience4j:
  circuitbreaker:
    instances:
      notificationService:
        failure-rate-threshold: 50       # 실패율 50% 넘으면 OPEN
        wait-duration-in-open-state: 30s # 30초 뒤 HALF-OPEN
        sliding-window-size: 10          # 최근 10개 요청 기준
        minimum-number-of-calls: 5       # 최소 5번 호출 후 판단
```

### 2-3. 서킷 브레이커 유무 차이

```
서킷 브레이커 없을 때:
  알림 장애 → 스레드 200개 전부 대기 → 주문 서비스 먹통 → 전체 다운
  → 수 분 내 전체 장애

서킷 브레이커 있을 때:
  알림 장애 → 5번 실패 → 서킷 OPEN → 이후 즉시 fallback
  → 주문/결제 정상 동작, 알림만 나중에 재시도
  → 수 초 내 장애 격리
```

---

### 2-4. Fallback 전략들

```
기본값 반환:   추천 서비스 장애 → 인기 상품 Top 10 고정 반환
큐에 저장:     알림 장애 → 큐에 저장 후 복구 시 재발송
캐시 데이터:   가격 서비스 장애 → 마지막 캐시된 가격 반환
기능 비활성화: 리뷰 서비스 장애 → 리뷰 섹션 숨김

핵심 원칙: "100% 기능"보다 "핵심 기능이 살아있는 것"이 중요
  → 부가 기능(추천, 리뷰, 알림) 장애가 핵심 기능(주문, 결제)을 죽이면 안 됨
```

## 🔍 Part 3. 모놀리식에서도 쓸 수 있다

### 3-1. 필요 조건: MSA가 아니라 "네트워크 외부 호출"

```
서킷 브레이커가 보호하는 것: "내가 제어할 수 없는, 실패할 수 있는 외부 호출"

모놀리식 내부 메서드 호출:
  orderService.createOrder() → 같은 프로세스, 네트워크 안 탐
  → 서킷 브레이커 불필요

외부 서비스 호출 (모놀리식이라도 존재!):
  PG사 API (토스페이먼츠), 카카오 알림톡, AWS S3, 소셜 로그인...
  → 네트워크를 탐, 언제든 실패 가능
  → 서킷 브레이커 필요!
```

### 3-2. MSA에서 더 중요한 이유

```
모놀리식: 내부 호출 95% + 외부 호출 5% → 적용 범위 좁음
MSA:     모든 서비스 간 호출이 네트워크 → 적용 범위 넓음

→ "MSA 전용"이 아니라 "네트워크 호출 전용"
→ MSA에서는 모든 호출이 네트워크라서 더 중요할 뿐
```

## 🔍 Part 4. 서킷 브레이커 외 장애 격리 패턴

### 4-1. 벌크헤드 (Bulkhead) — 격벽 패턴

배의 격벽에서 영감. 서비스별로 스레드 풀을 분리하여 한 서비스 장애가 다른 서비스의 자원을 잡아먹지 못하게 한다.

```
벌크헤드 없이: 스레드 풀 200개 공유 → 알림 장애로 전부 점유 → 주문도 불가
벌크헤드 적용: 주문 전용 150개 + 알림 전용 50개 → 알림 장애 시 50개만 영향
```

### 4-2. 실무에서는 조합

```java
@CircuitBreaker(name = "notification")     // 서킷 브레이커
@Bulkhead(name = "notification")           // 격벽
@Retry(name = "notification")              // 재시도
@TimeLimiter(name = "notification")        // 타임아웃
public void sendNotification(...) { }

동작 순서:
  타임아웃(3초) → 재시도(최대 3번) → 서킷 브레이커(실패율 50% 차단)
  → 벌크헤드(전용 스레드 풀) → 전부 실패 시 fallback
```

## 🔍 Part 5. 넷플릭스와 Chaos Engineering

### 5-1. Chaos Monkey — 일부러 서버를 죽인다

넷플릭스의 철학: "장애는 반드시 발생한다. 막을 수 없다면 장애가 발생해도 살아남는 시스템을 만들자."

```
Chaos Monkey:     운영 환경에서 서버 인스턴스를 랜덤으로 종료
Chaos Gorilla:    가용 영역(AZ) 전체를 종료
Chaos Kong:       리전(Region) 전체를 종료!
Latency Monkey:   네트워크 지연을 인위적으로 추가
```

2011년 AWS 대규모 장애를 겪은 후 "장애에 대비하자"가 아닌 **"장애를 일상으로 만들자"** 로 전환. 이후 AWS 리전 장애가 발생해도 자동으로 다른 리전으로 전환되어 사용자는 거의 눈치 못 챈다.

### 5-2. Netflix OSS → 현재 세대로의 전환

Netflix가 오픈소스로 공개한 도구들이 업계 표준이 되었으나, 이후 유지보수 모드에 들어가며 Spring Cloud 생태계가 후속 도구를 만들었다.

```
Netflix 세대 (2015~2018)         현재 세대 (2020~)
Hystrix (서킷 브레이커)     →    Resilience4j
Zuul (API 게이트웨이)       →    Spring Cloud Gateway
Ribbon (로드 밸런서)        →    Spring Cloud Load Balancer
Eureka (서비스 디스커버리)   →    Eureka (유지) 또는 Consul
```

## 🔍 Part 6. 자동 감지와 모니터링 — Spring Boot Actuator

### 6-1. Actuator = 앱의 건강 검진 도구

Spring Boot 팀이 만든 공식 모듈로, 운영 중인 앱의 상태를 **코드 수정 없이 확인·관리**할 수 있는 엔드포인트 모음이다.

```
주요 엔드포인트:

/actuator/health     → 앱이 살아있는가? (DB, Redis, 디스크 자동 체크)
/actuator/metrics    → JVM 메모리, CPU, HTTP 요청 통계
/actuator/info       → 앱 버전, Git 커밋 정보
/actuator/loggers    → 실시간 로그 레벨 변경 (재시작 불필요!)
/actuator/threaddump → 스레드 상태 (병목 지점 파악)
/actuator/env        → 환경 설정 확인
```

비유하면, 앱 코드가 **자동차 엔진**이라면 Actuator는 **계기판**이다. 계기판 없이는 속도, 연료, 엔진 온도를 알 수 없고, 문제가 생겨도 차가 멈춰야 알게 된다.

### 6-2. 서킷 브레이커 + Actuator 연동

Resilience4j의 서킷 브레이커 상태가 Actuator 헬스 체크에 자동으로 노출된다.

```json
GET /actuator/health
{
  "components": {
    "circuitBreakers": {
      "notificationService": {
        "status": "CIRCUIT_OPEN",
        "details": {
          "failureRate": "65.0%",
          "state": "OPEN"
        }
      }
    }
  }
}
```

### 6-3. 모니터링 파이프라인

```
Spring Boot App
  → Actuator (/actuator/prometheus)
  → Prometheus (메트릭 저장, 주기적 수집)
  → Grafana (시각화 대시보드)
  → AlertManager (Slack/PagerDuty 알림)

서킷 OPEN 발생 시:
  → Actuator가 메트릭 노출
  → Prometheus가 수집
  → Grafana 대시보드에 반영
  → Slack 알림: "🚨 알림 서비스 서킷 OPEN! 실패율 65%"
  → 복구 시: "✅ 알림 서비스 정상화!"

→ 개발자가 로그를 뒤지는 게 아니라
   자동 감지 → 자동 대응 → 자동 알림 → 자동 복구
```

### 6-4. 보안 주의

```
외부 공개 가능:   /health (status만), /info
내부에서만:       /metrics, /prometheus, /loggers
절대 공개 금지:   /env (DB 비밀번호 노출), /heapdump (메모리 유출), /shutdown
```

### 6-5. Actuator의 실무 사용률

Spring Boot를 쓰는 곳에서는 사실상 필수다. Kubernetes 헬스 체크, Prometheus·Grafana 모니터링, AWS·Azure·GCP 클라우드 플랫폼 통합이 전부 Actuator를 전제로 설계되어 있어, 안 쓰면 오히려 이 기능들을 직접 구현해야 하는 상황이 된다.

```
규모별 사용 방식:

소규모: /health만 (로드 밸런서 헬스 체크용)
중규모: /health + /metrics + /loggers + Prometheus 연동
대규모: 전 엔드포인트 활용 + 커스텀 메트릭 + 알림 자동화
```

## 💡 한 줄 정리

> 서킷 브레이커는 **장애 서비스를 자동 차단하여 핵심 기능을 보호**하고, 벌크헤드·타임아웃·재시도와 조합하여 사용한다. Actuator가 메트릭을 노출하고 Prometheus·Grafana가 수집·시각화하여 **자동 감지 → 자동 대응 → 자동 알림** 파이프라인을 구성한다. 넷플릭스는 Chaos Monkey로 **장애를 일상화**하여 이 구조를 극한까지 검증했고, Netflix OSS에서 Resilience4j로의 세대 교체를 거쳐 현재 Spring Cloud 생태계의 표준이 되었다.
